# Fix duplicate ADR ID validation error pinning blame to single path

## Summary

The duplicate ID validation error pins blame to paths[0] (first alphabetically), but neither file is inherently the failing one — both contribute to the error.

## Problem

When two files share the same ADR ID, the duplicate ID validation error is pinned to `paths[0]` (the first alphabetically sorted path). The full list is in the error message, so the information is complete, but pinning to a single path is misleading—neither file is inherently the "failing" one.

## Evidence

- **File:** `internal/ahm/validation.go`, line 514
- **Context:** `report.addError("adr_duplicate_id", paths[0], fmt.Sprintf(...))`

## Fix Direction

Either omit the path field entirely for duplicate ID errors or use a dedicated error code that doesn't imply a single failing file, since both files contribute to the error. The simplest approach is to pass an empty path (empty string) since the message already lists all paths.

## Acceptance Notes

- [ ] Duplicate ADR ID validation error does not pin blame to a single path.
- [ ] Error message still lists all duplicate paths.
- [ ] Test explicitly verifies the reported path field (or absence) for duplicate ID errors.
