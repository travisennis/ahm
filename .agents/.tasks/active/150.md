# Drain accumulated warnings on emit to stop duplicate stderr output

## Summary

`app.emitWarnings` prints `a.warnings` without clearing it, and command paths register `defer a.emitWarnings()` at multiple nesting levels (helper, command handler, and writeIndexes). One underlying problem therefore prints repeatedly. Reproduced against commit ff15056: with one malformed task file present, `ahm task create "x"` prints four warning lines on stderr (two near-duplicate messages, each printed twice).

## Evidence

- `internal/ahm/cli.go:33-44` — `addWarning` appends; `emitWarnings` iterates without resetting `a.warnings`.
- Nested emit sites that stack on one command: `internal/ahm/task_find.go:50` (`resolveTask`), `internal/ahm/task_create.go:68`, `internal/ahm/indexes.go:13` (`writeIndexes`), `internal/ahm/task_status.go:36`; grep `defer a.emitWarnings` for the full list.
- Two call sites also add different messages for the same failure: the generic "some task files could not be parsed and were skipped" (`task_create.go:71`) and the detailed variant including the joined errors (`indexes.go:75`).

## Fix Direction

Make `emitWarnings` drain: print, then set `a.warnings = nil` (single-goroutine CLI, no locking needed). That alone removes the re-printing. Then decide whether to also dedupe identical messages within one batch (cheap seen map) so exact repeats collapse while the generic and detailed variants both still print — only if tests show remaining noise. Do not remove the nested defer sites; they guarantee warnings surface on early returns. Keep the "warning: " prefix and stderr destination stable (output formats are a compatibility surface).
