# Preserve or specify optional front matter order

## Summary

Review issue 40. renderTask always writes optional fields in fixed order, causing noisy diffs and combining with unknown-field loss.

## Decision

Approach: **Document canonical order** in the spec rather than preserving input order.

`ahm` is a tool that normalizes — deterministic output is the right property for a
managed-file system. The canonical order is documented in `docs/spec.md` and
`renderTask` continues to write in that fixed order. No code changes to the
render order logic are needed.

Canonical order (already implemented, needs documentation):
1. `id`
2. `title`
3. `status`
4. `priority`
5. `effort`
6. `labels`
7. `exec_plan`
8. `depends_on`
9. `created` (optional, omitted when empty)
10. `updated` (optional, omitted when empty)
11. `parent` (optional, omitted when empty)
12. `external_ref` (optional, omitted when empty)
13. Extra/unknown fields (sorted by key)

## Acceptance Notes

- [ ] `docs/spec.md` documents the canonical front matter order.
- [ ] Round-trip tests confirm that renderTask always produces canonical order regardless of input order.
- [ ] Round-trip tests confirm that Extra fields survive and appear in sorted-by-key order.
- [ ] No code change to `renderTask` order logic.
- [ ] `go test ./internal/ahm` covers the round-trip order behavior.
- [ ] `just ci` is green before handoff.
