# Investigate why pre-commit hooks appear to run twice

## Summary

During a recent commit, pre-commit hook output showed each hook executing twice:

```
go fmt check.............................................................Passed
go mod tidy check........................................................Passed
go test..................................................................Passed
golangci-lint............................................................Passed
go fmt check.............................................................Passed
go mod tidy check........................................................Passed
go test..................................................................Passed
golangci-lint............................................................Passed
Conventional Commit......................................................Passed
```

If hooks are genuinely running twice, that doubles the pre-commit latency and explains why the process feels slower than expected. This task is to diagnose and fix the double-run, or confirm it's a display artifact.

## Evidence

- Commit `3a7afb8` output showed 8 "Passed" lines for 4 configured hooks.
- `.pre-commit-config.yaml` defines 4 local hooks + 1 commit-msg hook for conventional commits.
- Possible causes: `prek` running both pre-commit and post-commit/pre-push stages, duplicate hook registrations, or the `just` recipes themselves invoking hooks recursively.

## Fix Direction

Investigation steps:

1. Run `prek run --verbose` or `prek run --all-files` to see hook invocation details.
2. Check `git config --local --list` for any duplicate hook paths.
3. Check if `.git/hooks/pre-commit` contains duplicate entries or if both `prek` and another hook manager are installed.
4. Check if the `just` recipes (`just test`, `just lint`) recursively trigger git operations that re-invoke hooks.
5. Once the cause is identified, fix it and verify with a test commit.

If it's a `prek` bug or configuration issue, the fix is small (remove duplicate registration). If it's the `just` recipes triggering recursive hooks, the fix may involve adding `--no-verify` to git commands inside just recipes or restructuring the hook chain.

Deliverable: a diagnosis (what was causing it) and either the fix or a clear plan for the fix if it's complex.

## Acceptance Notes

- [ ] The root cause of double hook execution is identified and documented
