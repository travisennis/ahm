# Redesign ahm context role around managed-work intake

## Problem

`ahm context` currently carries too much responsibility as the canonical source for agent workflow guidance. The project is moving toward a clearer split:

- `AGENTS.md` owns early agent operating-loop and workflow-routing instructions.
- `ahm agents suggestions` helps maintainers add ahm-aware routing guidance to project-owned `AGENTS.md` files.
- `ahm context` should provide live repository/session briefing and scoped workflow reference, not replace the project's top-level agent instructions.

This needs a deliberate contract update so agents know when to use `ahm context`, when to use scoped contexts, and when to return to `AGENTS.md` routing.

## Relevant Files

- `internal/ahm/context.go`
- `internal/ahm/context_test.go`
- `internal/templates/templates.go`
- `docs/references/workflow-spec.md`
- `docs/references/cli/commands.md`
- `docs/guides/workflow-upgrades.md`
- `ARCHITECTURE.md`
- `AGENTS.md`

## Fix Direction

1. Audit current default and scoped `ahm context` output for duplicated top-level routing guidance.
2. Define the intended roles explicitly:
   - Default `ahm context`: live repository/session briefing and useful next commands.
   - Scoped `ahm context task|plan|adr|research|docs`: detailed workflow reference for that managed surface.
   - `ahm agents suggestions`: advisory bridge text for project-owned `AGENTS.md`.
   - Project `AGENTS.md`: early operating loop, managed-work intake, and concrete workflow routing.
3. Update implementation and tests to match the new contract.
4. Update durable docs for any changed command behavior or workflow semantics.

## Acceptance Notes

- [ ] The default `ahm context` output no longer duplicates responsibilities better owned by project `AGENTS.md`.
- [ ] Scoped `ahm context` commands remain useful as workflow references for tasks, ExecPlans, ADRs, research, and docs.
- [ ] `ahm agents suggestions` remains aligned with the new split between managed-work intake and implementation routing.
- [ ] CLI docs and workflow specs describe the new role split clearly.
- [ ] Focused tests cover changed `context` and suggestion behavior.
