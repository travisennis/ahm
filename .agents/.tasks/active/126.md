# Add task search command for title and body matching

## Summary

Task resolution is by ID only. An agent that knows a task title or keyword but not the ID has to `task list --json` and grep client-side. A `task search <query>` command would save a round trip and keep discovery through `ahm`.

## Evidence

- `internal/ahm/task_find.go` — `resolveTask` and `resolveTaskFromTasks` match by exact ID, numeric match, or prefix. No title or body search.
- `internal/ahm/task_list.go` — `taskList()` filters by status and labels. No text search.
- No `task search` command exists in `task_commands.go`.

## Fix Direction

1. Add a `task search <query>` subcommand to `task_commands.go`.

2. Implement search as case-insensitive substring matching against task titles. Optionally also match against the task body (the body can be large, so title-only is the safe default; body search could be opt-in with a `--full` flag).

3. Text output matches `task list` format: `ID [Status] Priority Effort Title`.

4. JSON output returns the same `[]Task` structs as `task list --json`.

5. Support the existing `--status` and `--label` filters so search can be scoped (e.g., `task search "timeout" --status Open`).

6. Empty results follow the same empty-state behavior as `task list` (see task 122).

## Acceptance Notes

- [ ] `task search <query>` matches tasks by case-insensitive substring on title
- [ ] `task search <query> --json` returns parsed task structs
- [ ] `task search <query> --status Open --label area:cli` composes filters with search
- [ ] Empty results print "No tasks found." in text mode and `[]` in JSON mode
- [ ] `task search` with no query produces a usage error
