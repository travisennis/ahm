# Add ahm audit delegation command

## Summary

Add `ahm audit`: delegate the codebase-improvement audit (the
finding-improvements procedure) to an external coding agent, require findings
as a schema-constrained JSON result, and mechanically create one `Open` task
per finding through ahm's normal task creation path.

## Problem

The procedure lives in
`internal/templates/workflow/finding-improvements-SKILL.md` (~469 lines, the
largest of the three skills), installed to
`.agents/skills/finding-improvements/SKILL.md`. Per ADR 014 (revised
2026-07-07) it becomes a delegation command. Two failure modes drive the
design: static audits propose work that already exists in the backlog (fixed
by live state in the prompt), and agent-created tasks drift from ahm's task
conventions (fixed by ahm owning creation from structured findings).

## Design

Prompt: procedure text adapted from the SKILL.md body — preserve its hard
rules in substance: senior-advisor read-only posture, never modify source,
never reproduce secret values, every finding fully self-contained — composed
with live state:

- Existing active task titles with ids and labels (compact lines) — the
  dedupe list; cap with a pointer to `ahm task list` if large.
- Label vocabulary so findings propose consistent `type:*`/`area:*`/`risk:*`
  labels.
- Validation findings, so known workflow issues are not re-reported.

The interactive present-findings-for-acceptance step in the skill is
deliberately dropped: per ADR 014 there is no interactive gate. `Open` status
is the acceptance gate; grooming (`ahm task groom`, 156c) later accepts or
recommends cancelling each proposal.

Delegation: same orchestration reuse as 156c (`--agent`,
`default_work_agent`, repo-root execution, session capture, `--dry-run`
prints the prompt with no delegation or writes). If 156c lands first, follow
its structure exactly.

Result contract: schema-constrained JSON findings (same per-agent schema
transport as 156c). Draft finding shape (exact schema is
implementation-owned):

```json
{"findings": [{"title": "…", "problem": "…", "relevant_files": ["…"],
  "fix_direction": "…", "acceptance_notes": ["…"],
  "labels": ["type:…", "area:…"], "priority": "P2", "effort": "S"}]}
```

Task creation: for each schema-valid finding, create a task through the
internal creation path (`internal/ahm/task_create.go`) with status `Open`,
the provenance label `source:audit` added to the agent-proposed labels, and a
body rendered from the finding fields using the standard task skeleton
(Problem / Relevant Files / Fix Direction / Acceptance Notes). ID allocation,
front matter, placement, and index regeneration stay ahm-owned. On invalid
output: no writes, preserved raw output, nonzero exit.

## Acceptance Notes

- [ ] `ahm audit` composes the procedure with the live dedupe list, label
      vocabulary, and validation findings, delegates to the configured
      agent, and creates one `Open` task per schema-valid finding with the
      `source:audit` label.
- [ ] The procedure's hard rules (read-only on source, self-contained
      findings, no secret values) survive the adaptation into the prompt.
- [ ] Invalid or schema-violating agent output produces no writes, preserved
      output for inspection, and a nonzero exit.
- [ ] `--dry-run` prints the composed prompt and performs no delegation or
      writes.
- [ ] The `ahm prime` routing table gains "Audit for improvements →
      `ahm audit`".

      arg-builder/parser changes follow `docs/guides/testing.md`, with
      live-run evidence recorded here per the `area:agent` +
