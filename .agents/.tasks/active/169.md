# Surface workflow validation after record mutations

## Summary

Mutating workflow commands currently update records and regenerate indexes without surfacing workflow validation findings that the mutation just created or left behind. This let completed tasks reference an active ExecPlan and left an active ExecPlan with outcomes until a later `ahm status` pass exposed the drift. Add a reusable workflow-only post-mutation validation summary for record-mutating commands so normal operations report these problems immediately.

Written against commit `5cbd271`.

## Evidence

- `internal/ahm/task_status.go:113` - `task complete` only runs acceptance-note validation before completion.
- `internal/ahm/task_status.go:168` - after writing the task and dependents, completion calls `a.writeIndexes()` and returns success output without running broader workflow validation.
- `internal/ahm/indexes.go:12` - `writeIndexes` regenerates indexes and snapshots records, but does not validate the resulting workflow state.
- `internal/ahm/status.go:11` - `ahm status` is the command that runs `validateWorkflowScoped` and reports the findings that normal mutations currently miss.

## Fix Direction

Add a small shared helper that runs the workflow validation scope after successful non-dry-run workflow mutations and emits concise warnings through the existing `a.addWarning` / `emitWarnings` channel. Start with commands that already mutate records and call `writeIndexes`: `task create`, `task complete` / `start` / `accept` / `reopen` / `cancel`, `task dep add/remove`, `task comment`, `adr` lifecycle commands, and explicit `ahm index`. Keep link validation out of this post-mutation warning pass so known Markdown-link false positives do not drown out core workflow drift. Do not change generated index formats or task file formats. Stop and report back if the only viable design would change success exit codes for warning-only validation findings.

## Acceptance Notes

- [ ] Completing a task that still references an active ExecPlan emits a stderr warning containing `task_completed_exec_plan_active` or equivalent actionable wording during `ahm task complete`, without requiring a separate `ahm status` run.
- [ ] The post-mutation pass uses workflow-scope validation only; it does not emit `markdown_link_missing` findings from the links scope.
- [ ] JSON and plain output behavior is documented if it changes; otherwise warnings stay on stderr and structured stdout remains compatible.
- [ ] Focused tests cover at least `task complete` and `ahm index` creating or leaving workflow warnings, following the patterns in `internal/ahm/task_commands_test.go` and `internal/ahm/validation_test.go`.
- [ ] Verification: `go test ./internal/ahm` passes, then `just ci` passes before handoff.
