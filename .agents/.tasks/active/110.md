# Document or harden partial-write behavior of mid-batch index writes

## Summary

`writeIndexes()` writes 8 generated index files sequentially in a loop, returning early on the first write error. If a mid-batch write fails (e.g., disk full, permissions), earlier files are updated, the failed file is stale, and later files are also stale. This leaves a temporarily inconsistent index state. The state self-heals on the next successful `ahm index` run. The behavior is not documented, which could confuse users after a disk-full error during `ahm task create` or `ahm index`.

## Evidence

- `internal/ahm/indexes.go:13-31` — `writeIndexes()` loops over sorted paths calling `writeFileAtomic`, returning `err` immediately on any failure.
- `internal/ahm/indexes.go:20-30` — the loop body has no rollback, no pre-check, and no warning emission for partial state.
- The 8 index paths are defined in `indexWritesFor()` at `indexes.go:72-85` (tasks root index, 3 bucket indexes, research index, 2 exec-plan indexes, ADR index).

## Fix Direction

Two options, choose the simpler one:

**Option A (document)**: Add a code comment above the loop explaining that index writes are best-effort and not rolled back, and that `ahm index` self-heals. Update `docs/references/workflow-spec.md` to note this behavior.

**Option B (harden)**: Pre-validate that all 8 target directories are writable before writing any file. If any pre-check fails, abort with a clear error before mutating any index. This prevents partial state without needing rollback logic.

Option A is simpler and matches the project's pragmatic style. If you choose Option B, the pre-check is cheap (a stat on each directory) and gives a clean error before any mutation.

Files in scope: `internal/ahm/indexes.go`, possibly `docs/references/workflow-spec.md`.
Out of scope: adding cross-file atomic commit or transaction semantics (overengineered for this use case).
