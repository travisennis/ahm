# Fix slice aliasing that corrupts reported dependency cycles

## Summary

`taskDependencyCycles` records cycles with `append(path[start:], id)`, which aliases the shared DFS path backing array. Later appends during the same traversal overwrite elements of already-recorded cycles, so `ahm task dep cycles`, `task dep add` rejection messages, and `task_dependency_cycle` validation findings can name tasks that are not in the cycle. Reproduced against commit ff15056: a graph whose only cycle is b->c->d->e->g->b reports `b -> c -> d -> e -> g -> h`.

## Evidence

- `internal/ahm/validation.go:796-806` — `cycles = append(cycles, append(path[start:], id))` inside the DFS; when `cap(path) > len(path)` the recorded slice shares backing with `path`, and later `dfs(dep, append(path, id))` calls (line 817) write into that backing.

Repro structure: tasks a->b->c->d->e->g, g depends on [b, h], h->i, i has no deps; all active. The recorded cycle's closing element is overwritten during h's traversal.

## Fix Direction

Copy the cycle before recording: build a fresh slice (e.g. `cycle := append(append([]string{}, path[start:]...), id)`) and append that to `cycles`. One-line fix in `internal/ahm/validation.go` plus a regression test using the graph above. Existing cycle tests (`internal/ahm/task_deps_test.go` and validation tests) pin deterministic order — keep them passing. Nothing else in the DFS needs to change.
