# Apply structured task revisions from ahm task groom

## Problem

The current grooming result schema supports only `accept` and `comment` actions plus dependency and label changes. Once the structured-revision design is accepted, the command needs a validated, atomic path for applying delegated body and permitted metadata revisions so a single grooming run can make an objectively fixable task ready for work.

## Relevant Files

- `internal/ahm/task_groom.go` — prompt, schema, result parsing, validation, summaries, and applier.
- `internal/ahm/task_groom_test.go` — command, parser, validation, and mutation coverage.
- `internal/ahm/tasks.go` — task parser and renderer.
- `internal/ahm/task_acceptance.go` — acceptance readiness checks that may be reused or extended.
- `internal/ahm/output.go` — text and JSON output conventions.
- `docs/references/cli/task-commands.md` and `docs/cli.md` — public command contract.

## Fix Direction

Implement the structured revision representation chosen by the ADR task. Extend the delegated schema and embedded prompt, validate every revision before any task is written, preserve task identity and protected content, and apply revisions through ahm's atomic workflow-record machinery. Support revise-only outcomes and atomic revise-and-accept when the resulting record is ready. Keep cancellation and any fields reserved by the ADR outside autonomous authority.

The batch path must remain all-or-nothing: malformed results, out-of-scope tasks, invalid dependencies, invalid metadata, or invalid revised bodies must leave every target unchanged. Text and JSON summaries should distinguish comments, revisions, and acceptance without exposing unstable internal details.

## Acceptance Notes

- [ ] The groom result schema and prompt represent the ADR-approved revision surface.
- [ ] Schema parsing and semantic validation reject malformed, out-of-scope, or protected-field changes before writes.
- [ ] Revisions preserve task identity, required front matter, comments, and other ADR-protected content.
- [ ] Revise-only and revise-and-accept outcomes are applied atomically through ahm-owned writes.
- [ ] Multi-task grooming remains all-or-nothing when any verdict is invalid.
- [ ] Text, JSON, and dry-run output document proposed/applied revision behavior.
- [ ] Focused tests cover successful revision, revision plus acceptance, preservation, validation failures, and batch rollback.
- [ ] CLI and architecture documentation reflect the implemented contract.
- [ ] Full CI passes.
