# Add session context command

## Summary

Add a top-level `ahm context` command that gives humans and coding agents a compact, dynamic briefing for the current repository workflow state. The command should serve the same purpose as Beadwork's `bw prime`, but use ahm terminology and avoid the `prime` command name.

## Problem

Agents currently discover workflow state by reading installed docs and running several separate commands such as `ahm status`, `ahm task next`, `ahm task ready`, `ahm task blocked`, and `git status`. That works, but it burns session context and makes it easy to miss important state such as dirty worktrees, in-progress tasks, blocked work, or workflow validation failures.

A single context command should orient an agent at session start without replacing the existing task, validation, or index commands.

## Proposed Direction

Add `ahm context` as a read-only top-level command. It should gather and print the minimum useful orientation for a fresh human or agent session:

- Repository root and workflow/template status.
- Whether validation currently passes, or a concise warning when it does not.
- Git branch and dirty worktree state when available, without running git operations that mutate state.
- Current in-progress tasks.
- Next ready task and a short ready queue summary.
- Blocked and open/untriaged task counts.
- Pointers to the exact commands to inspect details, such as `ahm task show <id>`, `ahm task ready`, and `ahm doctor`.

The default text output should be concise and stable enough for agents to consume. JSON output should expose structured sections instead of forcing consumers to parse prose. Plain output should follow the existing project conventions for stable line-oriented or compact output.

Do not implicitly start tasks, accept tasks, edit workflow files, run `ahm index`, commit, fetch, push, or invoke external agents.

## Design Questions

- Decide whether `ahm context --agent` is needed, or whether default output should already be agent-friendly.
- Decide whether dirty worktree detection should call `git status --porcelain` directly or use an internal helper with graceful fallback outside git repositories.
- Decide whether workflow validation should run fully by default or use a cheaper status-level check with a pointer to `ahm doctor` for details.
- Decide whether this feature introduces a durable architectural decision that needs an ADR before implementation.

## Acceptance Notes

- [ ] `ahm context` prints a useful session briefing in a repository with an installed `.agents` workflow.
- [ ] The command is read-only and does not rewrite indexes or workflow files.
- [ ] Dirty git worktree state is surfaced when detectable, with graceful fallback when git is unavailable.
- [ ] In-progress, ready, blocked, and open task state is summarized accurately.
- [ ] `--json` output is covered by tests and documented.
- [ ] User-visible command behavior is documented in `docs/cli.md` or the relevant CLI reference page.
- [ ] Focused CLI tests cover text output, JSON output, no-task repositories, and validation-warning scenarios.
