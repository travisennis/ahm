# Document Go package path fallback for restricted agent environments

## Summary

Agents in restricted environments (sandboxed shells, some CLI tools) cannot use `go test ./...` because the `./...` pattern is sometimes rejected as a path traversal. The repo root also isn't a Go package, so `go test .` fails. CONTRIBUTING.md documents the correct commands but assumes `./...` always works. Adding the full module-path fallback (`go test github.com/travisennis/ahm/internal/...`) gives agents a reliable second path that works in any environment.

## Evidence

- `CONTRIBUTING.md:14-25` — command catalog shows `go test ./...` but no fallback.
- `go.mod:1` — module is `github.com/travisennis/ahm`.
- The repo root is not a Go package (documented in CONTRIBUTING.md "Project-Specific Guidance"), so `go test .` fails.
- Agent tools vary in how they handle `./...` — some sandboxed environments reject it. The full module path always resolves correctly.

## Fix Direction

Add explicit package-path fallbacks to the command catalog in CONTRIBUTING.md:

```markdown
just test           # go test ./...
                    # or: go test github.com/travisennis/ahm/internal/...
just vet            # go vet ./...
                    # or: go vet github.com/travisennis/ahm/internal/...
just test-race      # go test -race -cover ./...
                    # or: go test -race -cover github.com/travisennis/ahm/internal/...
```

Add a one-liner in "Project-Specific Guidance":

```markdown
**Go package paths.** When `go test ./...` is unavailable (restricted shells,
sandboxed agents), use the full module path from `go.mod`:
`go test github.com/travisennis/ahm/internal/...`
```

Files in scope: `CONTRIBUTING.md` only. Out of scope: changing any Go source or build config.
