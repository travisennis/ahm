# Replace ref-backed records with committed .ahm workflow state

## Problem

The ref-backed records design removes workflow churn from the project branch,
but it makes ahm responsible for synchronization, divergence, worktree and
clone behavior, recovery, remote compatibility, and multiple representations
of the same record state. Release-readiness review reproduced data-loss and
workflow-breaking paths in those responsibilities.

The `.agents/` to `.ahm/` namespace change remains valuable because it separates
project-owned agent content from ahm-owned workflow state. Source records should
move to `.ahm/` while remaining ordinary committed project files so normal Git
provides durability, cloning, worktree behavior, merging, and recovery.

See `migrate-issues.md` for the reproduced failures and original review.

## Direction

- Keep `ahm records migrate` as the explicit migration trigger.
- Move task, research, ExecPlan, and config paths from `.agents/` to `.ahm/`.
- Preserve normal Git tracking for source records; do not seed or synchronize a
  custom records ref.
- Treat records as branch-scoped state that follows normal checkout, merge,
  conflict, and recovery behavior; do not promise a repository-global live
  backlog across simultaneous branches.
- Keep workflow indexes under `.ahm/` ignored and regenerable while preserving
  committed project-documentation indexes such as `docs/adr/index.md`.
- Return `ahm prime` to local repository preparation and briefing behavior.
- Migrate existing legacy committed repositories directly; ref-backed mode is
  unreleased and has no compatibility contract or adopter migration path to
  preserve.
- Supersede ADR 013 and document the new compatibility contract before coding.

## Acceptance Notes

- [ ] A new accepted ADR supersedes ADR 013 and defines committed `.ahm/` source records.
- [ ] An ExecPlan coordinates the cross-cutting transition.
- [ ] `ahm records migrate` moves legacy records to committed `.ahm/` paths without custom-ref operations.
- [ ] `prime` and normal workflow mutations do not fetch, push, or mutate custom records refs.
- [ ] Successful routine commands do not dirty committed config with machine-local sync state.
- [ ] Source records and supported attachments remain protected by normal Git tracking.
- [ ] Concurrent branch edits use normal Git integration; same-record and sequential-ID collisions surface as visible conflicts rather than hidden synchronization behavior.
- [ ] Workflow indexes are deterministic, ignored, and regenerated without branch-history churn; `docs/adr/index.md` remains committed.
- [ ] Fresh clones and linked worktrees use normal Git behavior without hidden record loss.
- [ ] CLI, workflow, upgrade, architecture, and rollback documentation reflects the committed `.ahm/` model.
