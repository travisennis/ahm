# Add opt-in project documentation health checks

## Summary

Add generic, opt-in documentation health checks for repositories without
assuming this project's documentation layout.

## Problem

The mined idea originally named this repository's docs (`README.md`,
`docs/cli.md`, `docs/spec.md`, `docs/upgrades.md`, and ADR indexes), but `ahm`
is meant to work in any project. A project-documentation health check cannot
hard-code this repo's filenames beyond broadly common conventions, and it
should not fail projects just because they organize documentation differently.

`ahm` already has a managed `.agents/DOCS.md` workflow guide and scoped
Markdown link validation for managed workflow files. The remaining opportunity
is an explicit, opt-in audit surface for durable project docs.

## Relevant Files

- `internal/ahm/cli.go` for command or flag wiring.
- `internal/ahm/validation.go` for validation orchestration.
- `internal/ahm/validation_test.go` for coverage.
- `internal/templates/workflow/DOCS.md` for documentation-audit guidance.
- `docs/cli.md` for command and finding documentation.
- `docs/spec.md` for the validation boundary.

## Fix Direction

Implement this only as an opt-in check, likely after or alongside scoped
validation modes. Discover common project documentation surfaces rather than
requiring a fixed layout:

- `README*`
- `docs/`
- `CONTRIBUTING*`
- `CHANGELOG*`
- `ARCHITECTURE*`
- `DESIGN*`
- `docs/adr/`

Start with structural checks that are low-risk and deterministic, such as
broken relative Markdown links in discovered docs. This task owns the opt-in
discovery, finding, and output framework plus the relative-link finding. Do not
implement design-doc index validation here; that is task 052, which builds on
this framework. ADR-index checks may be deferred to a follow-up as well. Avoid
subjective content quality scoring, model calls, or attempts to edit source
code.

Default `status` and `doctor` behavior should remain focused on managed
workflow files unless the user explicitly requests project-doc validation.

## Acceptance Notes

- [ ] Project documentation checks are opt-in and do not change default
      `ahm status` or `ahm doctor` behavior.
- [ ] The implementation discovers common documentation surfaces without
      requiring this repository's `docs/cli.md`, `docs/spec.md`, or
      `docs/upgrades.md` files to exist elsewhere.
- [ ] Broken relative links in discovered project docs produce clear findings.
- [ ] Design-doc index validation is left to task 052; this task only provides
      the framework it builds on. Any ADR-index checks included here run only
      when the target repository has clearly adopted that convention.
- [ ] Checks are deterministic, read-only, and do not call models or edit
      source files.
- [ ] Text, `--json`, and `--plain` output include the new findings
      consistently.
- [ ] `docs/cli.md` documents how to run the opt-in project-doc checks and any
      new finding codes.
- [ ] `docs/spec.md` documents that project documentation is not scanned by
      default.
- [ ] `go test ./internal/ahm` covers repositories with no docs, common docs,
      and broken project-doc links.
- [ ] `just ci` is green before handoff.
