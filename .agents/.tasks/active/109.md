# Add timeout to external agent execution in task work

## Summary

`ahm task work` delegates work to external coding-agent CLIs via `exec.Command` in `runTaskWorkCommand`. There is no timeout, deadline, or cancellation mechanism — if the agent CLI hangs (network timeout, API outage, client bug), the `ahm` process hangs forever with no recovery path except Ctrl+C. The review session, completion handoff, and commit handoff steps inherit the same unbounded execution.

## Evidence

- `internal/ahm/task_work.go:146` — `runTaskWorkCommand` calls `cmd.Run()` via `exec.Command(executable, args...)` with no `context.Context`, no timeout, no deadline.
- `internal/ahm/task_session.go:32` — `taskWorkWithSession` calls `taskWorkRunCommand` for the work session.
- `internal/ahm/task_session.go:65` — `runReview` calls `taskWorkRunCommand` for the review pass.
- `internal/ahm/task_session.go:94` — `runCompletion` calls `taskWorkRunCommand` for the completion handoff.
- `internal/ahm/task_session.go:103` — `runCommit` calls `taskWorkRunCommand` for the commit handoff.

## Fix Direction

Add a generous but finite timeout (e.g., 30 minutes) using `exec.CommandContext` with `context.WithTimeout`. Wire it through `runTaskWorkCommand` so all agent invocations benefit. The timeout should be configurable — start with a hardcoded constant, and consider a future `--timeout` flag on `ahm task work`. 

The `taskWorkRunCommand` signature changes to accept `context.Context`. Callers (`taskWorkWithSession`, `runReview`, `runCompletion`, `runCommit`) pass the context through. Test stubs for `taskWorkRunCommand` will need the new signature.

Files in scope: `internal/ahm/task_work.go`, `internal/ahm/task_session.go`. 
Out of scope: adding a `--timeout` CLI flag (future work).
If the agent CLI does not support `CommandContext` semantics (unlikely on any modern Go platform), STOP and report back.
