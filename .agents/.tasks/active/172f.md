# Retire ref-backed records commands and sync metadata

## Problem

Once source records remain committed, the ref-backed `records status`, `pull`,
`push`, `sync`, and sync-focused `doctor` behavior has no durable role. Keeping
it would preserve the highest-risk maintenance surface and leave conflicting
storage semantics in the CLI and metadata model.

## Relevant Files

- `internal/ahm/records.go`
- `internal/ahm/records_commands.go`
- records command and plumbing tests
- CLI and workflow metadata references

## Fix Direction

Retain `ahm records migrate` for the committed namespace migration and remove or
replace the ref-sync-only command surface according to the replacement ADR.
Delete the unreleased `store_mode: ref`, `records_ref`, `records_remote`, and
`records_last_sync` model and its private-ref plumbing outright. There is no
adopter state, custom ref, or released CLI compatibility to preserve.

## Acceptance Notes

- [ ] General CLI help no longer presents unsupported ref synchronization as usable behavior.
- [ ] Ref-only plumbing, metadata, remote restrictions, and tests are removed rather than deprecated.
- [ ] Unknown non-ref metadata fields remain preserved where promised.
- [ ] The remaining `records migrate` command matches the replacement ADR.
