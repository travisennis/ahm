# Default new tasks to Open and add task accept

## Summary

Change task creation so new tasks default to `Open`, and add an explicit
`ahm task accept <id>` command that moves accepted work into `Pending`.

## Problem

The task status model distinguishes captured-but-untriaged work from ready
backlog work:

- `Open` means newly captured work that still needs triage before it is ready
  for the queue.
- `Pending` means ready backlog work that can be picked up when priority and
  dependencies allow.

Today `ahm task create` defaults new tasks to `Pending`. That is convenient, but
it can put untriaged or underspecified work directly into `task ready` and
`task next`.

## Fix Direction

- Change the `task create` default status from `Pending` to `Open`.
- Keep `--status <value>` so callers can still create `Pending`, `Blocked`, or
  other supported statuses explicitly.
- Add `ahm task accept <id>` as the intentional transition from `Open` or other
  active non-terminal states into `Pending`.
- Keep `ahm task reopen <id>` for reopening completed or cancelled work into
  `Pending`; do not rely on `reopen` as the main triage command.
- Regenerate indexes after `task accept`, matching the existing status
  transition commands.
- Update CLI docs and workflow docs so the Open -> Pending triage step is clear.

## Acceptance Notes

- [ ] `ahm task create <title>` creates tasks with `status: Open` by default.
- [ ] `ahm task create <title> --status Pending` still creates a Pending task.
- [ ] `ahm task accept <id>` sets the task status to `Pending`, keeps active

- [ ] `ahm --dry-run task accept <id>` previews the same kind of status
      transition output as `start`, `complete`, `cancel`, and `reopen`.
- [ ] CLI docs and workflow docs describe the new default and accept command.
