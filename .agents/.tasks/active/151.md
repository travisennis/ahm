# Serialize task status transitions to close the dependent auto-unblock race

## Summary

`ahm task complete` performs an unlocked read-compute-write over the whole task set: it reads all tasks, computes which Blocked dependents become unblockable, and rewrites them. Two concurrent completions of tasks X and Y that are both dependencies of Blocked task D can each observe the other as incomplete, so neither unblocks D. D then stays Blocked silently and permanently — no validator flags a Blocked task whose dependencies are all Completed. ADR 010 scoped locking to `task create` and stated other mutations "should get their own narrow locks only when a concrete race is identified"; this task records that identified race. Analytic finding (MED confidence) against commit ff15056 — not observed in the wild, but the same parallel-agent usage that exposed the task-create race applies.

## Evidence

- `internal/ahm/task_status.go:35-153` — `taskStatusWithArgs` reads tasks, computes `taskUnblockDependents` (line 156), and writes the moved task, unblocked dependents, and indexes without a lock.
- `internal/ahm/task_work.go:127-140` — `markTaskInProgress` is a smaller unlocked read-modify-write on the same files.
- `internal/ahm/lock.go:17` — existing `acquireWorkflowLock` primitive.
- `docs/adr/010-task-create-id-allocation-lock.md` — narrow-locks-per-identified-race policy; dry-run must not take locks.

## Fix Direction

Two complementary parts; implement (1), consider (2):

1. Acquire a workflow lock around the non-dry-run body of `taskStatusWithArgs`, following the acquire/defer-release pattern in `internal/ahm/task_create.go:52-61`. Decide and document the lock name (a shared "task-mutate" vs. reusing "task-create" — reusing serializes against creates too, which is likely fine). Keep `--dry-run` lock-free.
2. Cheap detection net: add a validation warning (e.g. `task_blocked_deps_complete`) when a Blocked active task has all dependencies Completed, so any historical stuck state self-surfaces via `ahm status`. Validation codes are a compatibility surface — additive codes are allowed; do not rename existing ones.

Coordinate with the ref-backed records epic (task 138 / ADR 013): if task 144 has reshaped the workflow write paths by the time this is picked up, STOP and report back rather than forcing the lock into a changed design.
