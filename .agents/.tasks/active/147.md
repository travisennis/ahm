# Serialize ahm adr create ID allocation with the workflow lock

## Summary

`ahm adr create` computes the next ADR ID with an unlocked read-compute-write sequence, so concurrent invocations allocate the same ID. Reproduced against commit ff15056: six concurrent `ahm adr create` runs in a fresh repo produced four files numbered 001 and two numbered 002, leaving permanent `adr_duplicate_id` validation errors. This is the same race ADR 010 (docs/adr/010-task-create-id-allocation-lock.md) identified and fixed for `ahm task create`; the fix was never applied to ADR creation.

## Evidence

- `internal/ahm/adr_commands.go:198-245` — `adrCreateParsed` calls `nextADRID` and writes the new file without acquiring any lock.
- `internal/ahm/adrs.go:230-248` — `nextADRID` reads existing ADRs/files and returns max+1; two concurrent processes both observe the same max.
- `internal/ahm/adr_commands.go:237` — the file is written without a prior existence check, unlike `task_create.go:106` which stats the target and reports a retryable error.
- `internal/ahm/lock.go:17` — `acquireWorkflowLock(root, name)` already provides the mkdir-based repository-local lock used by `task create` (`internal/ahm/task_create.go:52-61`).

Repro shape: in a scratch repo, run several `ahm adr create "<distinct titles>"` concurrently; `ahm status` then reports `adr_duplicate_id` errors.

## Fix Direction

Mirror the `task create` pattern from `internal/ahm/task_create.go`: when not in dry-run, acquire `acquireWorkflowLock(a.opts.root, "adr-create")` before collecting ADRs and allocating the ID, and release it after indexes are written (defer with the same release-error handling as task create). Also add a post-allocation existence check before `writeFileAtomic`, returning an explicit "retry adr create" error on collision. Keep `--dry-run` lock-free (ADR 010 decision driver). Do not add locks to other adr commands in this task. Note: ADR 013 (ref-backed records) explicitly keeps ADRs in normal project history, so this fix does not conflict with the in-flight records epic (task 138).

Docs: mention the lock in the ADR command reference only at the same depth the task-create docs document theirs; match existing documentation style.
