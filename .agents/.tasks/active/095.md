# Document best practices for ahm task accept

## Problem

`ahm task accept` exists, but the workflow docs need stronger guidance on when and how to use it. Without clear instructions, agents may leave created tasks as `Open`, prematurely mark vague tasks as ready, or bypass acceptance by editing front matter directly.

## Relevant Files

- `.agents/TASKS.md`
- `docs/cli.md`
- `internal/templates/workflow/`
- `internal/ahm/task_commands.go`
- Tests for `ahm task accept`

## Fix Direction

Document the intended lifecycle around `Open` and `Pending` tasks:

- Create new captured issues as `Open` unless they are already fully scoped.
- Use `ahm task accept <id>` when triage confirms the task is actionable.
- Define what must be true before acceptance: clear problem, likely files or command surface, labels, priority, effort, dependencies, and acceptance notes.
- Explain when not to accept: missing decisions, unresolved dependencies, vague scope, or required ExecPlan/ADR not created yet.

Prefer workflow instructions that agents can follow mechanically.

## Acceptance Notes

- [ ] `.agents/TASKS.md` explains when to use `ahm task accept`.
- [ ] `docs/cli.md` includes user-facing examples for `ahm task accept`.
- [ ] Embedded workflow templates carry the same guidance for new installs/upgrades.
- [ ] Tests or fixture updates verify the command remains documented where expected.
