# Generate ADR index and validate ADR records

## Blocker

Blocked on task 075 for ADR collection/parsing and on task 074 for the
validation-scope and legacy-severity decisions recorded in ADR 009 plus the
feature ExecPlan.

## Summary

Generate `docs/adr/index.md` from ADR records via `ahm index` and add ADR
validation findings to `ahm status` / `ahm doctor`.

## Background

Design per ADR 009 (task 074) and
`.agents/.research/topics/madr-adr-management-design.md`. This changes
generated index semantics, so it is covered by the feature ExecPlan from task
074.

## Implementation Notes

- Add the ADR index to `indexWritesFor` and `generatedIndexTargets` in
  `internal/ahm/indexes.go`: a deterministic table (ADR, Title, Status, Date)
  sorted by ID, with the standard "generated by `ahm index`, do not edit"
  banner and `escapeCell` escaping. `README.md` and `index.md` are excluded
  from collection.
- Dry-run reports the ADR index only when stale, matching tasks 045 and 064.
- Validation findings: ADR parse failures, invalid status values (fixed set
  plus `superseded by ADR-NNN` pattern), ID/filename mismatches, duplicate
  IDs, supersede references that do not resolve, and a stale generated ADR
  index. Legacy-format files surface as a distinct finding pointing at
  `ahm adr migrate`.
- Per ADR 009, fold the checks into the existing `workflow` scope or add a
  dedicated check scope; update `validCheckScopes` and flag help if a new
  scope is added. Decide legacy-file severity so self-hosted `ahm status`
  stays usable until task 081 migrates this repo's ADRs.
- Update `docs/spec.md` (workflow state, file ownership boundary, validation
  scopes/findings) and `docs/cli.md` (`index`, `status`, `doctor`, validation
  findings sections) in the same change.
