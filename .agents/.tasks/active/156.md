# Replace managed skills with command-based procedures and ahm prime

## Summary

Track the ADR, CLI, template, migration, and documentation work to stop
installing managed agent skills into consumer repositories and instead expose
their procedures, plus a session-start briefing, directly from the `ahm`
binary.

## Context

Decided in a design discussion on 2026-07-01, modeled on beadwork's
`bw prime` / `bw onboard` approach (github.com/jallum/beadwork):

- The three managed skills (`.agents/skills/preflight`,
  `.agents/skills/grooming-backlog`, `.agents/skills/finding-improvements`)
  stop being installed. Their procedures move into the binary as live,
  state-aware scoped contexts: `ahm context groom`, `ahm context improve`,
  `ahm context preflight`. No per-repo customization of procedure content.
- A new read-only `ahm prime` command becomes the canonical agent
  session-start entry point (state-rich, instruction-light). This aligns with
  ADR 013, which reserves `ahm prime` for records sync; task 143 later layers
  the sync step on top of the command built here.
- Unscoped `ahm context` becomes a usage error pointing at `ahm prime` and
  the valid scopes. Scoped `ahm context <scope>` remains the instruction and
  procedure channel.
- `ahm agents suggestions` is removed in favor of `ahm onboard`, which prints
  a minimal paste-ready AGENTS.md snippet. A low-severity `ahm doctor` check
  flags an AGENTS.md that does not reference `ahm prime`.
- Content architecture: the pasted snippet carries only the bootstrap
  invariant (always run `ahm prime`) and safety invariants (never hand-edit
  generated indexes); the prime output carries live state plus the
  managed-work routing table; scoped contexts carry full instructions.
  Operating-loop guidance is project-owned and never emitted by ahm.
- JSON parity is kept for all new outputs (structured data in Go, text
  rendered from the same struct).
