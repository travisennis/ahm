# Fix test assertion patterns to surface all failures

## Summary

907 calls to `t.Fatal`/`t.Fatalf` vs 36 calls to `t.Error`/`t.Errorf` across test files. The assertion helpers in `test_helpers_test.go` use `t.Fatalf`, so the first failed assertion aborts the test and hides subsequent failures. Over CI this compounds — each run reveals only the first problem.

The fix: change assertion helpers from abort-on-fail to report-and-continue, then audit the 907 callsites to decide which justify a hard abort vs. a soft failure.

## Files in scope

- `internal/ahm/test_helpers_test.go` — assertion helpers (`assertContainsAll`, `assertNotContains`, `assertFileContainsAll`, `mustRead`, `writeFile`, `templateBytes`, `writeTaskFile`, `writeTaskFileWithPriority`, `runCLIFromDir`)
- `internal/ahm/*_test.go` — all 18 test files

## Files out of scope

- Non-test source files
- Test data files (`internal/ahm/testdata/`)

## Steps

### Step 1: Fix assertion helpers in `test_helpers_test.go`

Change these helpers from `t.Fatalf` to `t.Errorf` so a failure reports the problem but doesn't abort:

- `assertContainsAll` (line 43): `t.Fatalf` → `t.Errorf`
- `assertNotContains` (line 52): `t.Fatalf` → `t.Errorf`
- `assertFileContainsAll` (line 59): calls `assertContainsAll`, no change needed
- `mustRead` (line 66): keep `t.Fatal` — can't continue without file content
- `writeFile` (line 74, 77): keep `t.Fatal` — test setup failures should abort
- `templateBytes` (line 85): keep `t.Fatal` — missing embedded template is fatal
- `writeTaskFile` (line 92): delegates to `writeTaskFileWithPriority`, no change
- `writeTaskFileWithPriority` (line 98, 113): keep `t.Fatal` — setup failures abort
- `runCLIFromDir` (line 25-28, 33): keep `t.Fatal` for setup, keep `t.Errorf` at line 34 (correct as-is)

### Step 2: Search-and-replace the mechanical callsites

In every `*_test.go` file, find calls that use `t.Fatalf` or `t.Fatal` **after the setup phase** (i.e., inside test assertions, not setup helpers). The mechanical rule:

- `t.Fatalf` → `t.Errorf` when the test can meaningfully continue
- `t.Fatal` → `t.Error` when the test can meaningfully continue
- Keep `t.Fatalf`/`t.Fatal` when the test cannot continue (nil pointer, setup failure)

**Do not change** `t.Fatal` calls inside these helpers: `mustRead`, `writeFile`, `templateBytes`, `writeTaskFile`, `writeTaskFileWithPriority`, `runCLI`, `runCLIFromDir`.

For every other `t.Fatal`/`t.Fatalf` in test files, ask: "if this assertion fails, can the rest of the test provide useful signal?" If yes, change to `t.Error`/`t.Errorf`. If no (nil pointer dereference, missing critical resource), keep as-is.

### Step 3: Run the suite and fix any cascading failures

```bash
go test -count=1 ./internal/ahm/ 2>&1 | tail -40
```

Some tests that currently work because they pass their only assertion before hitting a `t.Fatal` in a helper may now run further and expose latent issues. Fix each one by adding a `return` after `t.Errorf` when continuing really is pointless, or fix the underlying logic if the test was wrong.

### Step 4: Verify no regressions

```bash
go test -race -count=1 ./internal/ahm/
go vet ./internal/ahm/
just fmt-check
```
