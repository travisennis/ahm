# add tests for workflow lock mechanism (lock.go)

## Summary

`internal/ahm/lock.go` implements a directory-based mutex with retry loop, timeout, and stale lock detection. It serializes task creation, status transitions, and dependency mutations across processes. There are zero tests for this mechanism. A concurrency bug here could cause data loss (concurrent writes to the same task file) or indefinite hangs (deadlock on lock acquisition).

## Evidence

- `internal/ahm/lock.go` — 59 lines of non-trivial concurrency code with no corresponding `lock_test.go`.
- The lock is used by `taskCreateParsed` (line 44), `taskStatusWithArgs` (implicit through index writes), and `taskDepUpdate` — all critical task state mutations.

## Fix Direction

Add `internal/ahm/lock_test.go` with concurrent tests. Key scenarios:

1. **Basic acquire/release**: acquire a lock, verify it blocks another acquire attempt on the same name, release, verify the second acquire succeeds.
2. **Concurrent serialization**: launch N goroutines that each acquire the lock, increment a shared counter, and release. Assert the counter equals N after all complete.
3. **Timeout**: hold a lock and verify a concurrent acquire attempt times out after `workflowLockTimeout` (10 seconds). Override `workflowLockTimeout` for the test to a shorter duration.
4. **Stale lock cleanup**: create a lock directory, manipulate its modtime to be older than `workflowLockStaleAfter` (30 minutes), then verify a new acquire attempt cleans it up and succeeds.

Use `go test -race` on the test file. Follow the existing test patterns in `write_test.go` and `validation_test.go` (temp directories, `TestMain`-style setup if needed).

Do NOT touch the lock implementation itself unless a test reveals a bug. Do NOT change the exported lock API.

- [ ] All four scenarios above are covered (acquire/release, serialization, timeout, stale cleanup).
