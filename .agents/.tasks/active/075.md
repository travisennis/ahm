# Add MADR ADR model parsing and rendering

## Status

Ready after task 074 created ADR 009 and the feature ExecPlan.

## Summary

Add the ADR domain model to `internal/ahm`: collect, parse, render, and
resolve MADR records in `docs/adr/`, mirroring the Task model in
`internal/ahm/tasks.go`. This is the foundation every `ahm adr` command and
the ADR index builds on.

## Background

Design per ADR 009 (task 074) and
`.agents/.research/topics/madr-adr-management-design.md`. MADR front matter
must fit ahm's existing front matter grammar (`parseFrontMatterLine` rejects
YAML block lists and block scalars), so list-like fields are comma-separated
scalars.

## Implementation Notes

- New `internal/ahm/adrs.go` plus `adrs_test.go` with an `ADR` struct: ID,
  Slug, Title, Status, Date, DecisionMakers, Consulted, Informed, Extra
  (unknown front matter preserved), Path, Body.
- `adrFilePaths` / `collectADRs` over `docs/adr/*.md`, skipping `README.md`
  and `index.md`; deterministic ordering by ID.
- Filename convention `NNN-kebab-slug.md`; ID is the numeric prefix. Detect
  ID/filename mismatches.
- `parseADR` reuses `parseFrontMatter`; title comes from the first H1.
- `renderADR` writes a canonical front matter field order and preserves
  unknown fields, mirroring `renderTask`. Round-trip parse->render must be
  stable.
- `resolveADR` accepts `9`, `009`, or `009-slug` references without substring
  false positives (mirror `resolveTask`, task 001); `nextADRID` allocates the
  next number.
- Status validation accepts the fixed set (`proposed`, `accepted`,
  `rejected`, `deprecated`) plus the `superseded by ADR-NNN` pattern.
- Legacy-format files (no front matter, `# ADR NNN: Title` heading) must not
  abort collection; classify them distinctly so commands stay usable and
  `ahm adr migrate` can target them (mirrors task 033 resilience).
- Use the atomic write helpers in `internal/ahm/write.go` for any mutation.

## Relevant Files
