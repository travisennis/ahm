# Repair bucket mismatch when task status already matches

## Problem

`taskStatus` (`task_commands.go:671-674`) early-returns "`<id>` already
`<status>`" when the front-matter status matches the requested one, without
checking the file's bucket. A task whose front matter says `Completed` but
which sits in `.agents/.tasks/active/` — exactly the pitfall AGENTS.md warns
about — cannot be repaired with `ahm task complete <id>`: validation flags
`task_bucket_mismatch`, but the repair command refuses to act. The only
workaround is reopen-then-complete or a hand move plus `ahm index`.

The same gap exists for Cancelled-in-active and for `markTaskInProgress`
(`task_commands.go:313`), which always writes to `active/` without removing a
mis-bucketed original.

## Fix Direction

When the status already matches but the bucket does not, perform the move
(write to the correct bucket, remove the old path, regenerate indexes) instead
of returning early. Keep the true no-op (status and bucket both match) as an
early return.
