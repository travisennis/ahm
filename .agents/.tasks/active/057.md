# Add opt-in task work commit handoff

## Blocker

Two blockers:

1. **No ExecPlan.** This task is `Effort: L`. Per AGENTS.md rules, L-effort tasks require an ExecPlan before implementation. An ExecPlan must be created first.

2. **Unresolved design decisions** on:
   - Whether `ahm` ever runs `git commit` itself or only prompts the delegated agent.
   - The explicit consent mechanism (flag or subcommand).
   - Whether commit handoff requires a clean validation pass first.
   - Conventional Commit enforcement.
   - Whether push/PR creation remains out of scope.

See the Design Questions section below.

## Summary

Explore and implement an opt-in commit handoff for completed task work, matching
the useful parts of `scripts/task-workflow.sh` without violating `ahm`'s
no-implicit-git boundary.

## Background

The shell workflow resumes the original `cake` session and asks it to commit
the completed work for the task. ADR 006 kept commits out of the MVP because
`ahm` must not perform implicit git operations. A future feature may still be
useful if it is explicit, auditable, and clear about whether `ahm` or the agent
is responsible for git actions.

## Design Questions

- Should `ahm` ever run `git commit` itself, or should it only prompt the
  delegated agent?
- What explicit flag or subcommand is strong enough consent for commit-related
  behavior?
- Should commit handoff require a clean validation pass first?
- Should `ahm` enforce Conventional Commit formatting or leave that to the
  agent and commit hooks?
- Should push or PR creation remain out of scope even for opt-in commit
  handoff?
