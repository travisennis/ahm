# Fix release version injection broken by const templates.Version

## Problem

`.goreleaser.yaml` injects the release version with
`-X github.com/travisennis/ahm/internal/templates.Version={{ .Version }}`, but
`Version` is declared as a `const` (`internal/templates/templates.go:9`). The
linker's `-X` flag only sets `var` symbols, so the injection silently does
nothing. Verified empirically: building with
`go build -ldflags "-X github.com/travisennis/ahm/internal/templates.Version=9.9.9" ./cmd/ahm`
still prints `0.2.0`.

No tags have been pushed yet, so this has not shipped — but the first tagged
release will report `0.2.0` from `ahm --version`, in `ahm status`
`template_version`, and in every consumer's `.agents/ahm.json`.

Task 023 made `Version` a `const` on the stated assumption that there was "no
separate release pipeline that tags differently". That assumption was wrong:
the goreleaser config has injected (or attempted to inject) the tag version all
along.

## Design consideration

`templates.Version` currently serves double duty: CLI binary version and
workflow template version recorded in consumer `ahm.json`. If the fix is simply
`const` -> `var`, every binary release will change what gets stamped into
consumer metadata even when templates did not change.

## Fix Direction

Choose one:

1. Revert to `var` and keep goreleaser injection (simple, but conflates binary
   and template versions).
2. Separate the two: add a binary version variable (e.g. an
   `internal/version` package, `var`, injected by goreleaser) used for
   `--version`/`version`, and keep `templates.Version` as a `const`
   template-schema version bumped only when embedded templates change.
   `ahm.json` `version` keeps tracking the template version.

Option 2 is preferred. Record the decision (ADR or `docs/upgrades.md`) since it
changes durable version semantics.
