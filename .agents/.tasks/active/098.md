# Document CRLF normalization in ahm adr migrate help text

## Summary

`ahm adr migrate` converts CRLF to LF during migration. This is documented in docs/cli.md but not in the command's help text. Users may notice line-ending changes in version control diffs.

## Problem

`migrateLegacyADR` reads via `readWorkflowFile` which normalizes CRLF to LF, then renders via `renderADR` which uses LF. A legacy file with CRLF line endings will be converted to LF during migration. This is documented behavior (`docs/cli.md` says "CRLF line endings are normalized to LF"), but migration is not a pure metadata operation—it also changes line endings, which users may notice in version control diffs.

## Evidence

- **File:** `internal/ahm/adr_migrate.go`, line 43 (`readWorkflowFile`)
- **File:** `internal/ahm/adr_migrate.go`, line 153 (`renderADR`)

## Fix Direction

Document this CRLF→LF normalization explicitly in the `ahm adr migrate` help text and/or the command's long description. The behavior itself is acceptable and intentional—users just need to know about it.

## Acceptance Notes

- [ ] `ahm adr migrate --help` mentions that CRLF line endings are normalized to LF during migration.
- [ ] No behavioral change to the migration logic itself.
