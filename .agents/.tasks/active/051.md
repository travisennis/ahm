# Add scoped validation modes

## Summary

Add opt-in validation scopes so users can run targeted subsets of validation
without changing the default `status` and `doctor` behavior.

## Problem

`status` and `doctor` currently run the full managed workflow validation set.
That is appropriate as the default, but future documentation-oriented checks
should not make routine workflow validation noisy or slow. Users should be able
to ask for only workflow checks, only project documentation checks, or only link
checks when they are auditing a specific surface.

## Relevant Files

- `internal/ahm/cli.go` for command wiring and flag parsing.
- `internal/ahm/validation.go` for validation orchestration.
- `internal/ahm/validation_test.go` for validator and CLI coverage.
- `docs/cli.md` for user-facing flag documentation.
- `docs/spec.md` for durable validation-scope behavior.

## Decision

Approach: **Use a `--check` flag with string values.**

- Single `--check` flag, repeatable or comma-separated values.
- Values: `workflow`, `links`, `project-docs` (future).
- Default (no `--check` flag) = run all checks (current behavior).
- `--check workflow` = only managed workflow checks.
- `--check links` = only Markdown link checks.
- `--check project-docs` = only project documentation checks (once 053 is implemented).
- `--check workflow,links` or `--check workflow --check links` = combined scopes.

One flag to learn. Scales to N scopes. Composes cleanly for combined scopes.
Flags are added as a `[]string` persistent flag on `status` and `doctor` commands.

An ADR is not needed — this is an implementation detail of the existing
validation contract, not a new architectural decision.

## Fix Direction

Use a `--check` flag with string values `workflow`, `links`, and future
`project-docs`. Design a small scope mechanism inside `validateWorkflow`
that filters which validation groups run. Scope flags are explicit filters;
the default (no `--check`) runs all checks.

Keep the default behavior unchanged. If project documentation checks are not
implemented yet, `project-docs` is accepted by the flag but produces a clear
message or no-ops when the check group has no registered validators.

Because this changes user-visible CLI behavior and validation semantics, update
`docs/cli.md` and `docs/spec.md` in the same change.
