# Require a reason when cancelling a task

## Blocker

Two blockers:

1. **No ExecPlan.** This task changes task state transitions (cancel writes a reason, changes front matter or body, moves file). Per AGENTS.md rules, changes to task state transitions require an ExecPlan before implementation.

2. **Unresolved design decisions** on:
   - Whether the reason is a required `--reason` flag or a configurable strictness setting.
   - Storage: body section, front matter field, or both.
   - Whether `--force` bypasses the requirement.
   - Whether `task cancel` warns on `TODO`-filled acceptance notes.

An ADR documenting the design is also needed (see ADR 005 for precedent).

See the Design Questions section below.

## Summary

`ahm task cancel <id>` currently records nothing about why a task was
abandoned. `.agents/TASKS.md` asks the operator to "note in the task body why
it is being cancelled", but nothing enforces it, so cancellations silently
lose their rationale.

Motivating example: task 026 was cancelled with its Acceptance Notes still at
`- [ ] TODO` and no reason recorded anywhere. Months later the owner could not
recall why it was cancelled, and the task had to be reopened to re-litigate a
decision that was probably already made once.

## Design Questions

- Should the reason be a required `--reason <text>` flag on `task cancel`, or
  a warning-by-default that `strict_acceptance`-style metadata config can
  escalate to an error (mirroring ADR 005 / task 048's acceptance-notes
  pattern)? A hard requirement is simpler and matches the existing
  "do not cancel without a reason" workflow guidance.
- Where is the reason stored? Options: a `## Cancellation Reason` section
  appended to the body (visible in the task file), a `cancelled_reason` front
  matter field (machine-readable, shows up in `--json`), or both. Body section
  preferred for human readers; decide whether front matter is also worth it.
- Should `--force` bypass the requirement for automation edge cases?
- Should `task cancel` also warn when Acceptance Notes still contain the
  seeded `- [ ] TODO`, as a hint the task was never triaged before being
  abandoned?

## Fix Direction

- Add a reason input to `task cancel` in `taskCommand`/`taskStatus`
  (`internal/ahm/task_commands.go`); the cancel path is the
  `status == "Cancelled"` branch of `taskStatus`.
- Persist the reason per the storage decision; `renderTask` round-trips
  unknown front matter via `Extra`, so a new field needs explicit ordering if
  front matter is chosen.
- Update `.agents/TASKS.md` template guidance (embedded copy in
  `internal/templates/workflow/TASKS.md`) to describe the enforced behavior.
- This changes a task state transition, so per AGENTS.md it needs an ExecPlan
  before implementation, and as a user-visible behavior change it likely
  needs an ADR (see `docs/adr/005-task-acceptance-completion-checks.md` for
  the closest precedent).

## Acceptance Notes

- [ ] Cancelling without a reason is impossible (or blocked per the decided
      strictness model), with a clear error message.
- [ ] The reason is persisted in the cancelled task file and survives
      `renderTask` round-trips.
- [ ] `--dry-run` shows the reason without writing.
- [ ] Tests cover missing reason, provided reason, dry-run, and any
      force/strict paths.
- [ ] `docs/cli.md`, `docs/spec.md`, and the TASKS.md template are updated;
      ExecPlan and ADR exist per AGENTS.md rules.
