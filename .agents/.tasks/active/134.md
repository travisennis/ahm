# reduce reflection and any dispatch in output.go text rendering

## Summary

`internal/ahm/output.go` uses `any` dispatch with 9 explicit type cases plus a `reflect` fallback in `writeTextValue` (lines 58-105). Every new data shape passed to `a.emit()` must either be a supported type or fall through to the reflection/JSON path. The JSON and plain output paths are trivial (`json.Marshal`), but text output requires manual formatting logic that is scattered across helper functions (`writeMapText`, `writeSliceText`, `writeMapSliceText`, `writeStringMapText`, etc.). Adding a new command with a novel output shape risks broken text output that may not be caught by tests.

## Evidence

- `internal/ahm/output.go:58-105` — `writeTextValue` dispatches on `any` with cases for `string`, `bool`, `int`, `int64`, `float64`, `map[string]any`, `map[string]string`, `map[string][]string`, `map[string]int`, `[]any`, `[]map[string]any`, then falls through to `reflect.Slice`, `reflect.Struct`, `reflect.Pointer`, and a final `%v` fallback.
- `internal/ahm/output.go:164-227` — multiple formatting helper functions with subtle differences (e.g., `writeMapText` vs `writeMapTextInline`).
- New command output types (e.g., `taskComment`, `taskWork` preview maps) construct `map[string]any` inline, which the reflection chain must handle correctly.

## Fix Direction

Introduce a `textRenderer` interface:
```go
type textRenderer interface {
    RenderText(w io.Writer) error
}
```

For text mode, `emitText` checks if the value implements `textRenderer` and calls `RenderText`. Each command data struct that produces text output gets a `RenderText` method. The `map[string]any` map and slice helpers can stay, but the `any` dispatch in `writeTextValue` should be reduced to only the types that actually pass through it.

Scoped to `internal/ahm/output.go` and the struct types that currently produce text output via `a.emit()`. Do NOT change the JSON or plain output paths. Do NOT change the `a.emit()` function signature.

Preserve existing text output format exactly — this is a compatibility surface. The refactor must produce byte-identical text output for all existing commands. Run `just ci` and compare text output before/after.

## Acceptance Notes

- [ ] A `textRenderer` interface exists and is used by `emitText`.
- [ ] All existing command output types that produce text implement `textRenderer` or pass through a retained map/slice helper.
- [ ] `just ci` passes. Text output from `ahm status`, `ahm task list`, `ahm context`, `ahm doctor`, `ahm task show`, `ahm task next`, `ahm task labels`, `ahm task dep cycles`, and `ahm task dep tree` is byte-identical to before.
- [ ] No behavior changes to JSON or plain output.
