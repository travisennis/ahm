# Fail loudly on corrupt workflow metadata instead of resetting it

## Problem

`install.go:26` reads metadata with `meta, _ := readMetadata(root)`. A corrupt
`.agents/ahm.json` (bad merge, manual edit) is indistinguishable from a missing
one: `ahm init`/`ahm upgrade` proceeds with zero-valued metadata and rewrites
the file, permanently losing `strict_acceptance`, `default_work_agent`, and all
file-tracking hashes — which also turns every locally-modified managed file
into a conflict on the next upgrade.

The same ignored-error pattern silently changes behavior elsewhere:

- `task_commands.go:684` — strict acceptance enforcement is silently disabled
  when metadata is unreadable (`ahm task complete` proceeds).
- `task_commands.go:235` (`selectTaskWorkAgent`) — `default_work_agent` is
  silently ignored and the hardcoded default used.

## Fix Direction

In `readMetadata` callers, distinguish `fs.ErrNotExist` (fine: fresh install /
unmanaged repo) from read or JSON parse errors (abort with an actionable
message naming `.agents/ahm.json`). Audit every `readMetadata` call site:
`install`, `validateManagedFiles`, `validateGeneratedIndexes`,
`validateMarkdownLinks`, `status`, `doctor`, `selectTaskWorkAgent`, and the
strict-acceptance check in `taskStatus`. Validation paths may report a finding
instead of aborting (a `metadata_missing` error already exists; consider a
distinct `metadata_corrupt` code).
