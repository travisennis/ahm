# Document ahm task create in TASKS.md workflow

## Summary

`.agents/TASKS.md` and its embedded template
`internal/templates/workflow/TASKS.md` describe creating tasks only through
the manual workflow: writing a Markdown file in `.agents/.tasks/active/` and
then running `ahm index`. The `ahm task create` command already exists, is
documented in `docs/cli.md`, and is the preferred CLI-driven method, but
neither TASKS.md document mentions it.

## Problem

The "Creating Tasks" section of `.agents/TASKS.md` opens with:

> Create new tasks in `.agents/.tasks/active/` with the next available
> three-digit id...

The "Updating Tasks" section lists `ahm task start`, `ahm task complete`,
and `ahm task cancel` as the managed commands but omits `ahm task create`.
Readers who follow the documented workflow exclusively will create tasks by
hand, which is more error-prone, misses the ID-allocation and index-regeneration
logic that `ahm task create` handles automatically, and produces less
consistent task bodies.

## Relevant Files

- `.agents/TASKS.md` — the local workflow document consumed by agents.
- `internal/templates/workflow/TASKS.md` — the embedded template from which
  `.agents/TASKS.md` is created/upgraded.
- `internal/ahm/cli.go` — the `taskCreateParsed` handler and the task-create
  command wiring.
- `docs/cli.md` — the user-facing CLI reference (already documents the
  existing `ahm task create` behavior).

## Fix Direction

Update both `.agents/TASKS.md` and `internal/templates/workflow/TASKS.md` so
the "Creating Tasks" section:

1. Introduces `ahm task create <title> [flags]` as the preferred creation
   path.
2. Notes the flags available (`--priority`, `--effort`, `--labels`, `--status`,
   `--description`) or points to `docs/cli.md` for the full reference.
3. Keeps the manual creation instructions as the fallback or detail reference
   (for agents that cannot run `ahm` or want full control over the body).
4. States that `ahm task create` regenerates indexes automatically, so no
   separate `ahm index` is needed after creation.

Also check whether the "Do not run `ahm index` after..." paragraph near the top
of TASKS.md needs updating. Today it lists only `start`, `complete`, and
`cancel`; `create` also regenerates indexes.

Scope: docs-only changes to TASKS.md and the embedded template. No Go code,
no CLI behavior changes, no test changes. However, because the embedded
template is a managed file source, `go test ./internal/templates` should be
run after the template edit to verify it still embeds correctly.

If the task workflow documentation is considered a durable workflow contract,
create or update an ADR before implementation. That is unlikely for this
change, but note the question in the acceptance notes.

## Acceptance Notes
