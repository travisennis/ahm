# Capture and reuse task work agent sessions

## Decision

Implement session capture as an in-process orchestration primitive, not as
repository-persisted session metadata.

- `cake` is the first supported full-workflow agent because it exposes
  `session_id` through `cake --output-format json` and can resume with
  `cake --resume`.
- `ahm` should keep the captured session ID in memory during a single
  `task work` orchestration run and feed it to later review, completion, and
  commit handoff steps.
- Do not write a session metadata file for this task. Persisted sessions are a
  separate future feature only if `ahm` adds cross-command or cross-process
  resume behavior.
- Shape the implementation around a provider/session capability boundary so
  `codex`, `claude`, and `cursor` can be fully supported later without
  redesigning the workflow.
- Agents that do not yet expose the needed structured session capture/resume
  contract should fail clearly for full-workflow orchestration, while retaining
  the existing basic handoff behavior where supported.

## Summary

Capture a task-work agent session ID when the selected agent supports it and
reuse that session during the same `ahm task work` orchestration run.

## Background

`scripts/task-workflow.sh` depends on `cake --output-format json` returning a
`session_id`, then resumes that session with `cake --resume`. The current
`ahm task work` command performs only the initial handoff and intentionally
does not parse provider output. This task adds the minimal session plumbing
needed for later review, completion, and commit handoff tasks while avoiding a
new persisted session state format.

## Provider Direction

- Add a small internal capability model for agents that can:
  - start work and return structured output containing a session ID;
  - resume an existing session with a follow-up prompt.
- Implement those capabilities for `cake` first.
- Leave room for `codex`, `claude`, and `cursor` implementations by keeping
  session parsing and resume argv construction provider-specific.
- Do not store prompts, provider responses, credentials, or session IDs in task
  front matter or repository-local state as part of this task.
