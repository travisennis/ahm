# Configure task work agents and models by role

## Problem

Repository metadata currently supports one default task-work agent through `default_work_agent`, plus `taskWork.promptFile`. That is too coarse for the default `ahm task work` flow because the command can run multiple agent-backed phases with different jobs: implementation, independent review, feedback resume, and commit handoff.

A project should be able to express role-specific defaults in the ahm config file. For example, implementation might use `codex` with one model, while review uses `claude` or `codex` with a different model. The config shape should also leave room for future agent-backed roles without forcing another flat top-level setting.

## Relevant Files

- `internal/ahm/install.go` - metadata/config schema, JSON parsing/rendering, unknown-field preservation, and `.agents/ahm.json` / `.ahm/config.json` behavior.
- `internal/ahm/task_work.go` - selection precedence for implementation, review, feedback resume, and commit handoff phases.
- `internal/ahm/task_agents.go` and `internal/ahm/task_session.go` - role-specific agent/model selection and provider-specific argument builders.
- `internal/ahm/task_commands.go` - CLI overrides, help text, examples, and dry-run preview.
- `internal/ahm/install_test.go`, `internal/ahm/task_commands_test.go`, and golden agent fixtures - config round-trip and orchestration coverage.
- `docs/references/workflow-spec.md`, `docs/references/cli/task-commands.md`, `docs/cli.md`, `docs/guides/workflow-upgrades.md`, and ADR 006 - durable config and task-work behavior documentation.

## Fix Direction

Design and implement a config shape under the existing `taskWork` block for role-specific defaults. One likely shape is:

```json
{
  "taskWork": {
    "promptFile": ".agents/prompt.md",
    "implementation": { "agent": "codex", "model": "gpt-5-codex" },
    "review": { "agent": "claude", "model": "sonnet" }
  }
}
```

Keep backwards compatibility with existing `default_work_agent` and `taskWork.promptFile`. Define precedence clearly, with explicit CLI flags winning over role config, role config winning over legacy `default_work_agent`, and the built-in default used last. Decide whether feedback resume and commit handoff always use the implementation agent/session or whether they can become separately configurable roles; document the decision.

This task should build on or coordinate with the `--model` flag task so command-line model overrides and configured role models share one resolver instead of diverging.

Update ADR 006 or create a focused follow-up ADR before code changes if the config shape becomes a durable architectural decision.

## Acceptance Notes

- [ ] `.agents/ahm.json` and `.ahm/config.json` can configure task-work implementation and review agent/model defaults under `taskWork` without breaking existing metadata.
- [ ] Existing `default_work_agent` behavior remains supported and has documented precedence relative to role-specific config.
- [ ] CLI overrides such as `--agent` and `--model` win for the appropriate phase without mutating repository config.
- [ ] `ahm --dry-run task work <id>` reports the resolved agent and model for each configured phase that would run.
- [ ] Review, feedback resume, and commit handoff role behavior is explicit, tested, and documented.
- [ ] Config parse/render tests preserve unknown fields and cover legacy metadata, `.ahm/config.json`, and role-specific taskWork config.
- [ ] CLI docs, workflow spec, upgrade guide, and ADR notes are updated; focused task-work/config tests and required live agent smoke evidence are recorded if orchestration changes.
