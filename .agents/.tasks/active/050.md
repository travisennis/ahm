# Add task work agent handoff command

## Summary

Add an `ahm task work <id>` command that resolves a task and hands it to a supported coding-agent CLI. The MVP should support `cake`, `codex`, and Cursor, default to `cake`, and allow the default to be overridden by command-line flag or repository configuration.

This is intentionally an exploration feature, but it should still establish a clear boundary between deterministic task workflow behavior owned by `ahm` and delegated agent execution owned by the selected external tool.

## Background

The current local helper script at `scripts/task-workflow.sh` orchestrates a `cake` workflow:

- start a coding-agent session for a task-specific prompt;
- run an independent deslop review;
- resume the original session with the review feedback;
- ask the agent to complete the task and commit the result.

That workflow is useful, but it is currently shell-script-only and `cake`-specific. Folding the task handoff into `ahm` would let `ahm` resolve task IDs, validate task state, and provide a stable interface while still deferring implementation work to an external agent.

## MVP Scope

- Add `ahm task work <id>`.
- Add an agent selection flag, tentatively `--agent <cake|codex|cursor>`.
- Default to `cake` when no flag or config override is provided.
- Add a repository configuration mechanism for the default work agent.
- Resolve the task with existing task resolution behavior before invoking any agent.
- Refuse to work completed or cancelled tasks.
- Check task dependencies before invoking the agent.
- Start the task or otherwise define whether `task work` should mark the task `In Progress` before handoff.
- Generate a consistent task-work prompt that includes the resolved task ID and enough instruction for the agent to read the workflow docs and task file.
- Invoke the selected external CLI without requiring `ahm` to know provider credentials.
- Return a clear error when the selected agent executable is missing.
- Document the command and configuration behavior in `docs/cli.md`.
- Add focused tests for command wiring, option/config precedence, task-state validation, missing executable behavior, and prompt/argv construction.

## Design Questions

- Should config live in a new repository-local file such as `.agents/config.json`, or should `.agents/ahm.json` grow beyond metadata and managed-file hashes?
- Should `task work` call `ahm task start <id>` before handoff by default, require an explicit `--start`, or only validate and invoke the agent?
- Should review orchestration, such as the current deslop pass, be out of scope for the MVP or available behind an explicit flag?
- Should automatic task completion and git commit remain out of scope for the MVP? The current project spec lists model/coding-agent calls and implicit git operations as v1 non-goals, so this feature likely needs an ADR before implementation.
- What is the exact Cursor CLI executable and non-interactive invocation shape to support?
