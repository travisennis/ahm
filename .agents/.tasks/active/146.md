# Add subtask creation support to task create

## Summary

`ahm` documents parent tracker tasks with lettered child task IDs such as `047a`, `047b`, and `047c`, and the parser/resolver already understand suffixed IDs. However, `ahm task create` can only allocate the next top-level numeric ID. Agents must hand-create or manually patch child tasks, which increases the chance of stale indexes and inconsistent `parent` metadata.

## Relevant Files

- `internal/ahm/task_create.go` — task creation argument parsing, ID allocation, locking, file writing, and index regeneration.
- `internal/ahm/task_commands.go` — `ahm task create` flags and help text.
- `internal/ahm/tasks.go` — task parsing/rendering, `parent` field, `splitTaskID`, and task sorting.
- `internal/ahm/task_find.go` — task ID resolution for suffixed IDs.
- `internal/ahm/tasks_test.go` and `internal/ahm/task_commands_test.go` — likely focused test locations.
- `docs/references/cli/task-commands.md`, `docs/references/cli/task-file-format.md`, and `docs/references/workflow-spec.md` — CLI and workflow documentation.

## Fix Direction

Add first-class subtask creation support to `ahm task create`, likely through a flag such as `--parent <id>` or `--subtask-of <id>`. When the flag is present, `ahm` should resolve the parent task, allocate the next available lettered child ID under that parent (`137a`, then `137b`, etc.), write `parent: <parent-id>` in front matter, and otherwise preserve existing `task create` behavior.

The implementation should keep regular top-level ID allocation unchanged. It should reject missing or ambiguous parent IDs, avoid collisions with existing child files across active/completed/cancelled buckets, work in `--dry-run`, and regenerate indexes exactly like normal task creation. Consider whether the parent must be `Tracking` or whether non-tracking parents should produce a warning; make that policy explicit in docs and tests.

## Acceptance Notes

- [ ] `ahm task create "Child" --parent 137` creates the next available child ID for parent `137` and writes `parent: 137`.
- [ ] Child allocation scans active, completed, and cancelled task buckets and skips existing suffixes to avoid collisions.
- [ ] Existing top-level `ahm task create` behavior and ID allocation remain unchanged.
- [ ] Missing, malformed, or ambiguous parent values produce clear usage/runtime errors.
- [ ] `--dry-run` previews the child ID and target path without writing files or indexes.
- [ ] Help text and CLI/workflow docs describe subtask creation and the parent task expectations.
- [ ] Focused tests cover creation, dry-run, collision avoidance, parent resolution, sorting/resolution of suffixed IDs, and docs-visible output where applicable.
