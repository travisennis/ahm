# Make the task work agent timeout configurable and documented

## Summary

`ahm task work` kills the delegated agent subprocess after a hardcoded 30 minutes (`taskWorkTimeout`), and each phase (work, review, feedback resume, commit handoff) gets its own 30-minute cap. The limit is not documented in the CLI reference, so long-running L/XL delegated tasks die mid-session with no way to opt into a larger budget. Written against commit ff15056.

## Evidence

- `internal/ahm/task_work.go:15-18` — `taskWorkTimeout = 30 * time.Minute` const; applied per invocation in `runTaskWorkCommand` (lines 142-151).
- `docs/references/cli/task-commands.md` — the `task work` section documents flags and agent contracts but never mentions the timeout.
- Task 109 (completed) added the timeout to prevent hung agent CLIs; configurability was out of its scope.

## Fix Direction

Add a `--timeout <duration>` flag to `ahm task work` (Go duration syntax, default 30m, must be greater than zero), threaded into the context used by `runTaskWorkCommand`. Keep the default and per-phase application unchanged. Update `docs/references/cli/task-commands.md` to document both the default and the flag, and extend the command help examples per the repo's help-text conventions (see `internal/ahm/task_commands.go:185-208`). Out of scope: per-agent defaults and `.agents/ahm.json` configuration — flag only; if review of ADR 006 suggests metadata configuration is clearly better, STOP and propose before building both. This does not change agent argument builders or parsers, so golden fixtures should not need recapture; if orchestration behavior changes beyond the deadline value, consult docs/guides/testing.md.

## Acceptance Notes
