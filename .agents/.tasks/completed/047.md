# Add ExecPlan lifecycle coherence validation

## Summary

Extend `ahm doctor` with a plan-side validator that flags ExecPlans whose
bucket placement, completion narrative, and progress checklist disagree.
Today the only plan-related checks ([validateTaskExecPlans](../../../internal/ahm/cli.go))
run through tasks that reference a plan; standalone plans go unverified.

## Problem

ExecPlans have no front matter — status is implicit in whether the file sits
in `.agents/exec-plans/active/` or `.agents/exec-plans/completed/`, plus the
state of four narrative sections required by `.agents/PLANS.md`
(`Progress`, `Surprises & Discoveries`, `Decision Log`,
`Outcomes & Retrospective`). Plans can drift out of coherence (active plan
with a filled-in retrospective, completed plan with no retrospective,
completed plan with open `- [ ]` progress items, plans missing mandatory
sections) without any signal from `ahm`.

Adding an `info` severity also unlocks lower-noise findings that are useful
but should not gate `ahm doctor`'s `OK` status.

## Relevant Files

- `internal/ahm/cli.go` for `validateWorkflow`, `validateTaskExecPlans`,
  `execPlanHasRetrospective`, `resolveExecPlanReference`, and the
  `validationReport` / `validationFinding` types.
- `internal/ahm/cli_test.go` for validator coverage.
- `internal/templates/workflow/PLANS.md` as the authoritative list of
  mandatory ExecPlan sections.
- `docs/cli.md` for the `ahm doctor` finding reference.
- `docs/spec.md` if the new finding codes are part of durable workflow
  semantics.

## Fix Direction

Add `validateExecPlans(root, tasks, &report)` invoked from
`validateWorkflow`. Walk both `.agents/exec-plans/active/` and
`.agents/exec-plans/completed/`, skipping `index.md`. For each plan emit
warnings:

- `exec_plan_active_with_outcomes` — plan in `active/` whose
  `## Outcomes & Retrospective` body is non-empty (reuse the line-walking
  approach from `execPlanHasRetrospective`).
- `exec_plan_completed_without_outcomes` — plan in `completed/` with an
  empty or missing `## Outcomes & Retrospective` section.
- `exec_plan_completed_with_open_progress` — plan in `completed/` whose
  `## Progress` section still contains `- [ ]` items.
- `exec_plan_missing_section` — plan missing any of the four mandatory
  sections.

Add a new `info` severity to `validationReport` (alongside existing
`Errors` and `Warnings`) and emit one info finding:

- `exec_plan_orphan` — plan that no task references via `exec_plan:`. Use
  the existing `resolveExecPlanReference` data already collected during
  task parsing; build the reverse index once.

Findings flow through the existing `a.emit(report)` JSON output. `info`
findings must not change `report.OK`. Update CLI text output (if any) to
print the new tier consistently.

Heading matching should be case-insensitive and accept `##` or `###`
levels to stay tolerant of plan style variations.

Because this changes user-visible CLI behavior and durable workflow
semantics, update `docs/cli.md` and `docs/spec.md` in the same change.
