# Fix Cake stream-json parser to read type-tagged events

## Problem

Commit bf6c1ae switched `ahm task work` to `cake --output-format stream-json`
orchestration, but `parseCakeSessionID` and `parseCakeReviewFeedback` look for
an `"event"` discriminator key in each JSONL line. Cake actually serializes
`StreamRecord` with serde `#[serde(tag = "type", rename_all = "snake_case")]`
(cake repo: `src/types/session.rs`), so real lines look like:

```json
{"type":"task_start","session_id":"<uuid>","task_id":"<uuid>","timestamp":"..."}
{"type":"task_complete","subtype":"success","is_error":false,"result":"...","duration_ms":1000,...,"session_id":"<uuid>",...}
```

Every line unmarshals cleanly but `Event` is always empty, so no `task_start`
event is ever matched. `parseCakeSessionID` returns an empty session ID, and
`ahm task work <id> --commit` fails with:

```
error: cannot run commit handoff: no session ID returned by cake
```

`parseCakeReviewFeedback` has the same bug and silently returns empty review
feedback, so `--review` always skips the feedback-resume step.

The unit tests did not catch this because the fixtures were written with the
same invented `{"event": ...}` schema the parser expects.

## Relevant Files

- `internal/ahm/task_commands.go` — `cakeStreamEvent`, `parseCakeSessionID`,
  `parseCakeReviewFeedback`
- `internal/ahm/task_commands_test.go` — cake stream-json fixtures
- Cake schema reference: `~/Projects/cake/src/types/session.rs`
  (`StreamRecord`, `TaskStartData`, `TaskCompleteData`, `TaskOutcome`) and
  `~/Projects/cake/src/clients/agent/agent_tests.rs`

## Fix Direction

- Change the `cakeStreamEvent` discriminator field to unmarshal from
  `"type"` instead of `"event"` and update both parser functions.
- The top-level `result` field is correct as-is: cake flattens `TaskOutcome`
  into the `task_complete` record via `#[serde(flatten)]`.
- Rewrite the test fixtures to mirror cake's real serialization (type tag,
  `subtype`, `is_error`, `task_id`, `timestamp` fields) so the tests validate
  against the actual contract instead of round-tripping the parser's own
  assumptions.
- No docs change expected: `docs/cli.md` and ADR 006 describe the events by
  name (`task_start.session_id`, `task_complete.result`), which remains
  accurate.

## Acceptance Notes

- [x] `parseCakeSessionID` extracts the session ID from real cake stream-json
      output shaped `{"type":"task_start","session_id":...}`. The
      `cakeStreamEvent` discriminator now unmarshals from `"type"`, and
      `TestCakeSessionIDParsing` includes a full-fidelity record sequence
      (task_start with timestamp, reasoning, function_call,
      function_call_output, task_complete with usage) mirroring cake's
      `StreamRecord` serde serialization.
- [x] `parseCakeReviewFeedback` extracts `result` from real
      `{"type":"task_complete",...}` lines. `TestParseCakeReviewFeedback`
      covers the full record shape, an omitted `result` (success outcome with
      no result text), and the last-task_complete-wins rule.
- [x] Test fixtures match cake's serde serialization: all fixtures use the
      `"type"` tag, task_complete lines carry flattened `subtype`/`is_error`
      outcome fields, and message lines carry `role`. A regression case
      asserts the legacy invented `{"event": ...}` key is ignored.
- [x] `go test ./internal/ahm` and `just ci` pass (CI run 2026-06-12,
      including lint, vet, vuln, and goreleaser snapshot).
