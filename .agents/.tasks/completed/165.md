# Use non-dot record directories under .ahm

## Problem

ADR 013 moved ref-backed workflow records into tool-owned `.ahm/`, but it kept legacy hidden directory names for some record types, such as `.ahm/.tasks/` and `.ahm/.research/`. Inside an already-hidden `.ahm/` root, the extra dot prefixes make the layout harder to scan without adding useful hiding behavior.

Ref-backed repositories should use clear non-dot directory names under `.ahm/`: `.ahm/tasks/` for tasks, `.ahm/research/` for research, and any other ahm-owned internal subdirectories should be reviewed for the same convention. Legacy committed-record repositories should keep the existing `.agents/.tasks/` and `.agents/.research/` paths unless a separate migration explicitly changes legacy storage.

## Relevant Files

- `internal/ahm/workflow_paths.go` - central storage-mode-aware path resolution.
- `internal/ahm/records.go` and `internal/ahm/records_commands.go` - ref-backed snapshot, materialization, sync, migrate, and doctor behavior.
- `internal/ahm/install.go`, `internal/ahm/indexes.go`, `internal/ahm/status.go`, `internal/ahm/validation.go` - directory creation, index rendering, and validation paths.
- `internal/ahm/task_*.go`, `internal/ahm/context.go`, `internal/ahm/agents.go` - task operations and user-facing guidance that mention record paths.
- `internal/templates/workflow/` - embedded workflow references and skills that mention fallback paths.
- `docs/references/workflow-spec.md`, `docs/guides/workflow-upgrades.md`, `docs/references/cli/`, `ARCHITECTURE.md`, and ADR 013 - durable storage layout documentation.
- Tests and golden fixtures covering records migration, ref-mode task paths, generated-index exclusion, context output, and doctor/status diagnostics.

## Fix Direction

Change ref-mode path resolution so new and migrated repositories use non-dot record directories under `.ahm/`. Include a compatibility plan for repositories already migrated with `.ahm/.tasks/` or `.ahm/.research/`, such as detecting the old layout, moving it during `ahm records migrate` resume or upgrade/doctor-assisted repair, and keeping diagnostics explicit.

Audit all user-facing output for the new paths. Generated indexes should remain generated and should move with their source record family. The records snapshot should continue excluding generated indexes and should keep writing only to `refs/ahm/records`, never to project branches, `HEAD`, or the project index.

## Acceptance Notes

- [x] Ref-backed task records live under `.ahm/tasks/` instead of `.ahm/.tasks/`; ref-backed research records live under `.ahm/research/` instead of `.ahm/.research/`.
- [x] Other ahm-owned subdirectories under `.ahm/` are reviewed. The lock directory at `.ahm/.lock/` is kept dot-prefixed because it is an internal implementation detail (cross-process synchronization); changing it would break lock coordination between old and new binary versions during upgrades, and a dot prefix follows the convention of hidden implementation detail directories. The managed `.ahm/.gitignore` stays as-is since it follows standard gitignore naming convention.
- [x] Legacy `.agents/.tasks/` and `.agents/.research/` behavior is unchanged for committed-record repositories.
- [x] Migration, resume, status, doctor, records snapshot/materialization, and generated-index behavior handle the new layout and diagnose or repair old ref-mode dot-directory layouts.
- [x] CLI docs, workflow spec, upgrade guide, embedded templates, and relevant tests are updated for the new `.ahm/tasks/` and `.ahm/research/` paths.
- [x] Verification follows `CONTRIBUTING.md`: all `./internal/...` tests pass (both packages), `go build ./...` succeeds, and template tests pass.
