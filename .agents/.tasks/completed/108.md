# Enumerate valid statuses in ahm task list --help

## Summary

The `ahm task list --help` output shows:

```
--status strings   Filter tasks by status (comma-separated or repeatable)
```

It does not tell the user what the valid status values are. The same gap may also exist on the `ahm task create --status` flag, though the immediate ticket covers `task list`.

Valid statuses are (from `internal/ahm/indexes.go:statusOrder()`):

- `Open`
- `Pending`
- `In Progress` (accepted as `in-progress`, `inprogress`, `in_progress`)
- `Blocked`
- `Tracking`
- `Completed`
- `Cancelled`

## Relevant Files

- `internal/ahm/task_commands.go` — `taskListCommand()` registers the `--status` flag with its current description string. The description is the only thing that needs to change.
- `internal/ahm/indexes.go` — `statusOrder()` is the canonical source of valid statuses.
- `internal/ahm/task_enum.go` — `normalizeTaskStatus()` and related validation logic; the error message already lists valid statuses when an invalid value is passed.
- `docs/references/cli/task-commands.md` — the existing CLI reference already enumerates valid statuses in a list under "Task Commands" and mentions case-insensitive per-entry matching with `in-progress` as an alias for `In Progress` in the `task list` section.

## Fix Direction

Update the `--status` flag description in `taskListCommand()` inside `internal/ahm/task_commands.go` to include the valid statuses. For example:

```go
cmd.Flags().StringSliceVar(&statuses, "status", nil, "Filter tasks by status; valid: Open, Pending, In Progress, Blocked, Tracking, Completed, Cancelled")
```

This is a low-risk, S-sized docs-only change to a flag description string. No behavioral surface changes. After the change, verify:

1. `ahm task list --help` shows the valid statuses.
2. `ahm task list --status invalid` still exits with a clear error listing valid statuses (already works, verify it still does).
3. All existing tests pass.
