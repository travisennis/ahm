# removeStaleWorkflowLock uses os.Remove on directory; replace with RemoveAll

## Summary

`removeStaleWorkflowLock` in `internal/ahm/lock.go` uses `os.Remove(lockPath)` to clean up a stale lock directory. On Unix, `os.Remove` on a directory fails with `ENOTEMPTY` if the directory contains any files. Since the error is silently discarded (`_ = os.Remove(lockPath)`), a non-empty stale lock directory is never cleaned up, causing `acquireWorkflowLock` to retry until the 10-second timeout, returning an opaque timeout error to the user.

This is unlikely with current code (locks are empty directories), but any future change that writes a file inside the lock directory (e.g., a PID file for debugging) would cause permanent acquisition failure.

## Evidence

- `internal/ahm/lock.go:26` — `os.Mkdir(lockPath, 0o755)` creates an empty directory as the lock.
- `internal/ahm/lock.go:50-55` — `removeStaleWorkflowLock` calls `os.Stat(lockPath)` then `_ = os.Remove(lockPath)`. `os.Remove` on a non-empty directory returns `ENOTEMPTY` on Unix, which is silently discarded.
- `internal/ahm/lock.go:38` — after `removeStaleWorkflowLock` returns, `os.Mkdir` is retried, but if the lock still exists (because removal silently failed), the loop continues until timeout.

## Fix Direction

Replace `os.Remove(lockPath)` with `os.RemoveAll(lockPath)` at `internal/ahm/lock.go:54`. `os.RemoveAll` removes the directory and any contents, matching typical lock cleanup semantics.

This is a one-line change. Add a test case to `lock_test.go` (created by the related lock test task) that creates a non-empty stale lock directory and verifies cleanup succeeds.
