# Add ref-backed records migration workflow

## Blocker

Blocked until tasks 139 and 140 provide the private-ref plumbing and records
metadata model needed for migration.

## Summary

Add the opt-in migration flow that moves ahm-managed state into `.ahm/`, seeds refs/ahm/records, writes safe internal gitignore and metadata changes, previews branch/index effects, and prints the user-run git rm --cached command instead of silently untracking project-owned records.

## Relevant Records

- ADR: `docs/adr/013-use-ref-backed-workflow-record-storage.md`
- ExecPlan: `.agents/exec-plans/active/138-ref-backed-workflow-records.md`

## Work Direction

Implement opt-in migration separately from routine `ahm upgrade`. Migration
should move task, research, ExecPlan, generated-index, sync-state, and config
files into `.ahm/`, install or update `.ahm/.gitignore`, seed the records ref,
set metadata, and preview the required branch/index changes. It must not
silently run `git rm --cached` for project-owned records, and it must preserve
committed project-owned `.agents/` content such as `.agents/prompt.md`.

## Acceptance Notes

- [x] `ahm records migrate --dry-run` previews all filesystem, metadata, ref,
      and user-run git-index changes.
      `TestRecordsMigrateDryRunPreviewsWithoutWriting` proves the preview lists
      every move, the gitignore/config/legacy-config actions, the ref seed, and
      the git cleanup command while writing nothing (no `.ahm/`, no ref, legacy
      files intact).
- [x] Real migration seeds `refs/ahm/records`, writes committed `.ahm/config.json`,
      and installs safe internal `.ahm/.gitignore` entries.
      `TestRecordsMigrateMovesRecordsSeedsRefAndPrintsGitCleanup` verifies the
      seeded ref holds source records but not generated indexes, config carries
      `store_mode: "ref"` with explicit ref/remote while preserving prior
      fields, and `.ahm/.gitignore` ignores `/.tasks/`, `/.research/`, and
      `/exec-plans/` while config stays committed. Project-owned
      `.agents/prompt.md` is untouched.
- [x] Migration prints the required `git rm -r --cached .agents/.tasks
      .agents/.research .agents/exec-plans .agents/ahm.json` command instead of
      running it. The same test asserts nothing is staged, `HEAD` is unmoved,
      and the legacy paths remain tracked until the user runs the printed
      command.
- [x] Rollback and partially migrated states are documented and diagnosed.
      Rollback steps are documented in `docs/references/cli/commands.md`
      (`records migrate`) and `docs/guides/workflow-upgrades.md`. Re-running
      `ahm records migrate` resumes interrupted migrations (identical targets
      treated as moved; differing targets fail as conflicts), and
      `ahm records doctor` gained a `migration` check for leftover legacy
      record paths, leftover legacy config, and still-tracked git index paths
      (`TestRecordsMigrateResumesPartialStateAndRejectsConflicts`,
      `TestRecordsDoctorDiagnosesPartialMigration`).

Verification: `go test ./internal/ahm/ -run 'TestRecordsMigrate|TestRecordsDoctor'`,
full `just ci` (tests with race+cover, vet, fmt-check, tidy-check,
golangci-lint, govulncheck, goreleaser check), `just docs-md-lint`, and a
manual end-to-end run in a scratch repository covering dry-run, migration,
user-run `git rm -r --cached` plus commit, `records doctor` (ok), `records
sync` push to a bare remote, and `records status` (equal).

Note: workflow commands read `.ahm/` paths only after task 144; until then
migration remains a developer-facing opt-in.
