# Make root detection fail outside managed repositories

## Summary

Review issue 6. detectRoot silently falls back to the current working directory when no .git or workflow metadata exists, risking accidental scaffolding into arbitrary directories. Acceptance: commands require an existing .git or .agents/ahm.json unless --root is explicit, error messages explain remediation, and install/init behavior remains intentional and documented.

## Acceptance Notes

- [x] `detectManagedRoot()` (replacing `detectRoot()`) walks up from cwd checking for `.git` (directory) or `.agents/ahm.json` (file).
- [x] When neither marker is found, an error is returned explaining remediation: `--root` or `ahm init`.
- [x] All strict commands (`status`, `doctor`, `index`, `task list`, `task next`, `task create`, `task show`, task status transitions, task dep commands) fail outside managed repos with exit code 1 and the remediation message.
- [x] `init` and `upgrade` use `detectRootOrCWD()` (lenient), allowing them to run in any directory.
- [x] `--root` flag bypasses root detection entirely.
- [x] `init` succeeds in a bare directory; after `init`, `status` works because `.agents/ahm.json` now exists.
- [x] `version` and `--help` work anywhere (no root detection needed).
- [x] All existing tests continue to pass; 6 new tests cover the new behavior.
- [x] `docs/cli.md` updated to document the new strict/lenient behavior, error remediation, and `--root` exemption.
- [x] `.git` file (git worktree) is a pre-existing limitation — `detectManagedRoot` checks `stat.IsDir()` for `.git`, so worktrees where `.git` is a regular file are not recognized. Noted for future work.
