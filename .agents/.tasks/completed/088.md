# Add golden agent transcript fixtures and capture recipe

## Problem

The `ahm task work` agent parsers encode assumptions about external CLI output
schemas, and the unit tests validate those assumptions against hand-written
fixtures derived from the same assumptions. This circularity shipped a broken
integration in bf6c1ae: the cake stream-json parser looked for an invented
`"event"` discriminator key while cake emits `"type"`, and every test passed
because the fixtures used the invented key too (fixed in task 087).

Codex orchestration rests on the same kind of guessed assumptions
(`thread.started` events, `thread_id` field), and tasks 082 (Claude Code) and
084 (cursor) will add more parsers of the same shape. Nothing in the repo ties
any fixture to real tool output.

## Relevant Files

- `internal/ahm/task_commands.go` — `parseCakeSessionID`,
  `parseCakeReviewFeedback`, `parseCodexSessionID`, `parseCodexReviewFeedback`
- `internal/ahm/task_commands_test.go` — inline fixtures these goldens augment
- `justfile` — home for the capture recipe

## Implementation Notes

- Add `internal/ahm/testdata/agents/` with transcripts captured from the real
  binaries: cake work run, cake review (deslop) run, codex exec run, codex
  resume run. Extend per agent as orchestration support lands.
- Stamp each golden with provenance: agent version (`cake --version`,
  `codex --version`), capture date, and the exact command used. Use a sidecar
  metadata file (e.g. `cake-work.meta`) since JSONL cannot carry comments.
- Add a `just capture-agent-fixtures` recipe that re-runs each installed agent
  with a trivial, cheap prompt (smallest model, low max tokens where the CLI
  supports it) and rewrites the goldens. Skip agents not on PATH. Document
  that this step costs real LLM calls and is run manually, not in CI.
- Add parser unit tests that read the committed goldens and assert a session
  ID and review feedback are extracted. These run hermetically in `just ci`
  (goldens are committed); they must fail, not skip, if a golden is missing.
- Consider scrubbing machine-specific content (absolute paths) at capture
  time; session IDs and usage numbers are fine to commit.

## Acceptance Notes

- [x] Goldens for cake (work + review) and codex (exec + resume) are committed
      under `internal/ahm/testdata/agents/` with version-stamped provenance.
      Captured live from cake 0.1.0 and codex-cli 0.138.0 on 2026-06-12 via
      `just capture-agent-fixtures`; sidecar `.meta` files record agent
      version, capture date, and exact command.
- [x] Parser unit tests consume the goldens and pass without network access
      (`internal/ahm/task_commands_golden_test.go`; a missing golden fails,
      not skips). The codex resume golden was captured by resuming the exec
      golden's session, so the round-trip test pins that a captured thread ID
      is actually resumable.
- [x] `just capture-agent-fixtures` refreshes goldens from installed agents
      and skips missing ones (verified by running with agents stripped from
      PATH: exits 0, leaves goldens untouched). Usage and cost documented in
      `internal/ahm/testdata/agents/README.md` and the justfile comment.
      The capture runs in a scratch repo with the ahm workflow installed so
      the cake review capture exercises the real deslop skill; transcripts
      are scrubbed of machine-specific paths and the local username.
- [x] `just ci` passes hermetically (fmt, tidy, vet, race tests, lint, vuln,
      build, release-check all green).

Live-run evidence: cake work transcript shows `task_start` with
`session_id`; codex exec and resume transcripts both show `thread.started`
with the same `thread_id`. The runReview prompt was hoisted to the
`taskWorkReviewPrompt` const and `TestCaptureScriptUsesReviewPrompt` keeps
the capture script's copy in sync.
