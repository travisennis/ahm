# Support returning an ADR to proposed via ahm adr

## Problem

`ahm adr` exposes forward lifecycle transitions only: `accept`, `reject`, `deprecate`, and `supersede`. There is no command that returns an ADR to `proposed`, so undoing a premature status change requires hand-editing the front matter and running `ahm index` — exactly the manual fallback the ADR command surface exists to remove.

Real occurrence (2026-07-11): ADR 016 was accepted during task 160a, the maintainer wanted it back at `proposed` for review, and the only path was a manual front-matter edit plus `ahm index`.

## Relevant Files

- `internal/ahm/adr_commands.go` — command wiring; the status-verb spec table (`accept`/`reject`/`deprecate`) and the generic `adrSetStatus(id, status)` helper already exist, so the change is mostly exposing another transition.
- `internal/ahm/adr_commands_test.go` — lifecycle transition, refusal, dry-run, index-regeneration, and record-preservation coverage.
- `docs/references/cli/commands.md` — the ADR lifecycle command sections.
- `internal/templates/workflow/adr-README.md` — embedded `ahm context adr` instruction table and status section.
- `docs/cli.md` — index entry for CLI commands.
- `docs/adr/009-madr-adr-management.md` — accepted architectural contract for the ADR command family and status model.

## Fix Direction

Add `ahm adr propose <id>` as a narrowly scoped correction for a MADR-profile ADR that was mistakenly created or marked `accepted` before review was complete. It sets the status back to `proposed`, updates the date, and regenerates the ADR index without rewriting user-owned body prose.

Resolved decisions:
- No separate ADR is needed. This is an anticipated extension of ADR 009 (MADR ADR Management). Update ADR 009's Decision Outcome and Confirmation to record the command and the strict lifecycle policy before implementation.
- ADR status changes preserve decision history. `propose` permits only `accepted -> proposed`, plus an idempotent `proposed -> proposed` invocation. It must refuse `rejected`, `deprecated`, and `superseded by ADR-NNN` records because those statuses record durable lifecycle outcomes, not premature acceptance mistakes.
- Apply the same lifecycle policy consistently to the existing commands rather than guarding only `propose`:
  - `accept`: `proposed -> accepted`; `accepted -> accepted` is idempotent.
  - `reject`: `proposed -> rejected`; `rejected -> rejected` is idempotent.
  - `deprecate`: `accepted -> deprecated`; `deprecated -> deprecated` is idempotent.
  - `supersede`: only an `accepted` ADR may become `superseded by ADR-NNN`; rerunning the same supersession remains idempotent. The replacement ADR must be `accepted` because a proposed or terminal decision cannot replace an accepted decision.
  - `rejected`, `deprecated`, and `superseded by ADR-NNN` are otherwise terminal. Changing the decision later requires a new ADR and the existing supersession or reference mechanisms; ordinary status commands must not erase the historical outcome.

Alternative considered: a generic `ahm adr status <id> <value>`; the verb form matches the existing surface and keeps `superseded by ADR-NNN` reachable only through `ahm adr supersede`, which preserves the reciprocal-reference invariant.

Behavior notes:
- Respect `--dry-run` and regenerate `docs/adr/index.md` like the other lifecycle commands.
- The `exactArgs` error message for `adr propose` must include `propose` alongside the existing verbs.
- Validate the source status before any write. A refused transition must leave both ADR records and generated indexes unchanged and return a clear runtime error naming the current status and allowed transition.
- Preserve ADR body bytes, line endings, and unknown front-matter fields for ordinary status transitions.
