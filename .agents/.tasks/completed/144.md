# Integrate ref-backed records with workflow write paths

## Summary

Integrate record snapshot behavior with task, research, and ExecPlan write paths while keeping generated indexes local-only and preserving legacy committed-record behavior until migration.

## Relevant Records

- ADR: `docs/adr/013-use-ref-backed-workflow-record-storage.md`
- ExecPlan: `.agents/exec-plans/active/138-ref-backed-workflow-records.md`

## Work Direction

Wire ref-backed record behavior into task, research, and ExecPlan writes without
changing legacy behavior for repositories that have not opted in. Ensure
generated indexes remain local-only. In ref-backed mode, source records live
under `.ahm/`; project-owned `.agents/` content remains committed and outside
the record snapshot set. Review `ahm task work` commit handoff so gitignored
task state is not swept into normal project commits after migration.

## Acceptance Notes

- [x] Legacy committed-record repositories behave as before.
      `internal/ahm/workflow_paths.go` resolves record paths to `.agents/`
      whenever metadata is missing or `store_mode` is absent/`committed`, and
      the full pre-existing test suite passes unchanged.
- [x] Ref-backed repositories snapshot `.ahm/` source records after supported
      task, research, and ExecPlan mutations. All mutating commands funnel
      through `writeIndexes`, which now calls `snapshotRecordsAfterWrite`
      (ref mode only, dry-run aware, warns instead of failing after a
      successful record write). `snapshotRecordsRef` reuses the prior commit
      when the record tree is unchanged so re-runs stay idempotent.
      Proven by `TestTaskCreateInRefModeWritesAhmRecordsAndSnapshotsRef`,
      `TestTaskCompleteInRefModeMovesRecordAndSnapshotsRef`,
      `TestIndexInRefModeSnapshotsHandEditsAndStaysIdempotent`, and
      `TestDryRunTaskCreateInRefModeWritesNothing`.
- [x] Generated indexes are regenerated locally but excluded from ref
      snapshots. Index writes follow the mode-aware paths under `.ahm/`, the
      snapshot selector still excludes `index.md`, and the ref-mode tests
      assert the records ref never contains index files.
- [x] `ahm task work` commit handoff remains coherent when task files are
      gitignored and synced through records refs. The commit prompt is
      storage-mode aware: migrated repositories are told to commit only
      project source changes and not to force-add gitignored `.ahm/` files
      (`TestBuildTaskWorkCommitPromptIsModeAware`).

Verification: `go test ./...`, full `just ci` (tests with race+cover, vet,
fmt-check, tidy-check, golangci-lint, govulncheck, goreleaser check),
`just docs-md-lint`, and a manual end-to-end run in a scratch repository:
legacy init/create/complete, `records migrate` plus the printed
`git rm -r --cached` cleanup, ref-mode `task create`/`task complete` writing
`.ahm/` records with automatic ref snapshots and untouched `HEAD`/index,
`records sync` to a bare remote, and a fresh clone recovering the backlog via
`records pull` + `ahm index` + `ahm task list`.

Additional integration in this task: mode-aware paths for validation
(including legacy `.agents/exec-plans/...` `exec_plan:` references resolving
to migrated `.ahm/exec-plans/...` plans), `ahm init`/`ahm upgrade` workflow
directory creation, the workflow lock root, and stale `.tmp` cleanup.

Preflight (L/XL, three passes): updated `docs/references/workflow-spec.md`
and `docs/references/glossary.md` so the documented lock location is
storage-mode aware, renamed the `cleanupStaleTempsIn` parameter that no
longer only names `.agents`, and unified the research index path on the
`workflowPaths` helper. The commit-prompt contract wording in ADR 008 was
left unchanged because that ADR is superseded by ADR 012 and the live
behavior is documented in `docs/references/cli/task-commands.md` under
accepted ADR 013. `just ci` and `just docs-md-lint` re-run clean after the
preflight fixes.
