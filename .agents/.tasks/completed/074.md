# Write ADR and ExecPlan for MADR ADR management

## Summary

Decide and record the durable design for MADR-only ADR management (`ahm adr`)
before implementation starts. Write ADR 009 under `docs/adr/` and create the
feature ExecPlan under `.agents/exec-plans/active/`. This task gates tasks
075-081.

## Background

Research:

- `.agents/.research/topics/madr-adr-management-design.md` (design synthesis)
- `.agents/.research/sources/adrs-cli-and-madr-format.md` (adrs CLI, MADR 4.x)

`ahm` currently installs `docs/adr/README.md` documenting a custom
Nygard-style ADR format. The feature adopts MADR 4.x as the only supported
format and adds an `ahm adr` command family mirroring `ahm task`.

## Grooming Notes

Ready for work. The product/design basis is captured in the referenced
research topic; this task's work is to ratify those choices in ADR 009 and the
feature ExecPlan before implementation tasks start.

## Decisions To Ratify

- ADR directory stays `docs/adr/` (not MADR's default `docs/decisions/`).
- Numbering stays three-digit `NNN` with kebab-case slug filenames
  (`009-short-title.md`); numbers are stable and never reused.
- Front matter is a constrained MADR profile compatible with ahm's front
  matter grammar: scalar values only; `decision-makers`, `consulted`, and
  `informed` are comma-separated scalars; no YAML block lists or block
  scalars; unknown fields preserved on rewrite.
- Status set: `proposed`, `accepted`, `rejected`, `deprecated`, and the
  pattern `superseded by ADR-NNN`. Rule on how partial supersession
  (legacy "Superseded in part") is represented now that MADR has no such
  status; recommendation in the topic note is body-level notes.
- Ownership boundary: ADR bodies are user-owned; `ahm` owns the generated
  `docs/adr/index.md` and the front matter mutations performed by `ahm adr`
  commands (plus the defined supersession note).
- Supersession is explicit and bidirectional via
  `ahm adr supersede <old> --by <new>`.
- Migration approach for legacy `**Status:** / **Date:**` ADRs is
  metadata-only conversion; bodies untouched (task 081).
- Whether ADR validation joins the `workflow` check scope or gets a dedicated
  scope, and what severity legacy-format files produce before migration.
- Whether ADR 009 itself is written in the constrained MADR profile as the
  first example record, or in the current format and migrated by task 081.
