# Add opt-in task work commit handoff

## Resolved Prerequisites

Prerequisites resolved before implementation:

1. **ExecPlan created.** This task is `Effort: L`, so implementation proceeded
   under `.agents/exec-plans/completed/057-task-work-commit-handoff.md`.

2. **ADR boundary updated.** ADR 008 permits explicit delegated commit handoff
   while preserving the rule that `ahm` does not run git operations itself. ADR
   006 is marked superseded in part by ADR 008.

Dependencies 055 and 058 are complete. They are no longer blockers.

## Decision

Implement commit handoff as a later prompt to the original working agent
session. `ahm` must not run `git commit` itself.

- Keep this task separate from session capture, review, and completion.
- The handoff is explicit and opt-in.
- The commit prompt should tell the delegated agent to ensure the task is
  completed before committing, matching `scripts/task-workflow.sh`.
- `--commit` does not require the user to also pass `--complete`. The normal
  task work instructions already tell the agent to work the task through
  completion; the commit handoff should include the script's instruction to
  make sure the task is marked completed before committing.
- Push and PR creation remain out of scope.
- Commit-message format is project-defined. The delegated prompt should ask the
  agent to commit the completed work, not prescribe Conventional Commit or any
  other message convention unless the target project's own instructions do.
- The full workflow assumes supported agents provide session capture and
  resume. Do not spend implementation effort adding special guard behavior for
  sessionless agents in this task.
- `scripts/task-workflow.sh` is the behavioral source of truth for the desired
  sequence: work the task, run preflight review, resume the work session to fix
  review feedback, then resume the work session to commit completed work.

## Summary

Explore and implement an opt-in commit handoff for completed task work, matching
the useful parts of `scripts/task-workflow.sh` without violating `ahm`'s
no-implicit-git boundary.

## Background

The shell workflow resumes the original `cake` session and asks it to commit
the completed work for the task. ADR 006 kept commits out of the MVP because
`ahm` must not perform implicit git operations. This feature is useful only if
it remains explicit, auditable, and clear that the delegated agent, not `ahm`,
is responsible for git actions.

## Provider Direction

- Use the provider/session boundary from task 056.
- Implement the commit follow-up prompt for `cake` first.
- Keep command structure compatible with later `codex`, `claude`, and `cursor`
  full-workflow support.
- Do not prescribe commit-message convention in `ahm`; the target project's
  instructions and hooks own commit policy.
