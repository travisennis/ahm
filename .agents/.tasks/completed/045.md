# Make index dry run report only stale generated indexes

Created: 2026-05-29

## Problem

`ahm --dry-run index` currently prints every generated index target even when the
files on disk already match the content that `ahm index` would write. In
`/Users/travisennis/Projects/cake`, this caused `just ci` to fail at
`task-index-check` immediately after successful `ahm index` and
`ahm --force index` runs.

Observed output included:

```text
.agents/.research/index.md
.agents/.tasks/active/index.md
.agents/.tasks/cancelled/index.md
.agents/.tasks/completed/index.md
.agents/.tasks/index.md
.agents/exec-plans/active/index.md
.agents/exec-plans/completed/index.md
```

The current implementation of `writeIndexes` treats dry run as a preview of all
write targets and does not compare rendered index content with existing files.
That conflicts with the workflow wording that dry run previews indexes that
would be rewritten and with downstream CI recipes that treat any dry-run output
as stale generated indexes.

The behavior expectation in `cake` comes from two places:

- `.agents/TASKS.md` says `ahm --dry-run index` previews generated indexes that
  would be rewritten.
- `justfile` recipe `task-index-check` captures `ahm --dry-run index` output
  and fails CI when the output is non-empty, printing "Task indexes are stale."

## Relevant Files

## Fix Direction

Update `ahm --dry-run index` so it reports only generated index files that are
missing or whose rendered content differs from the file on disk. A clean
repository immediately after `ahm index` should produce no dry-run index output.

Keep non-dry-run `ahm index` behavior simple and deterministic. If the command
continues to rewrite all generated indexes during non-dry-run execution, make
the dry-run behavior and docs explicit so CI can depend on the clean/no-output
contract.
