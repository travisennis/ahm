# Support comma-separated statuses in task list --status

Created: 2026-06-12

## Summary

Allow `ahm task list --status` to filter by multiple statuses at once via a
comma-separated list or repeated flags.

## Problem

`ahm task list --status <value>` accepts exactly one status. Filtering for
several statuses at once (for example `open,pending` to see all untriaged plus
ready backlog work) requires multiple invocations and manual merging. The flag
is registered as a plain `StringVar` and the filter does a single exact match.

## Relevant Files

- `internal/ahm/task_commands.go`
  - `taskListCommand` (~line 164): registers `--status` with
    `cmd.Flags().StringVar(...)` for `mode == "all"`.
  - `taskList` (~line 699): normalizes the single value via
    `normalizeTaskStatus` and calls `filterTasksByStatus`.
  - `filterTasksByStatus` (~line 778): exact single-status match.
  - `normalizeTaskStatus` / `enumKey` / `enumError` (~lines 840-858): existing
    normalization (case-, space-, hyphen-, underscore-insensitive) and usage
    error formatting; reuse these per entry.
- `internal/ahm/cli.go` (~line 149): `ahm status --check` already uses
  `StringSliceVar` with "(comma-separated or repeatable)" help text; follow
  that convention.
- `internal/ahm/task_commands_test.go`: `TestTaskListFiltersStatus` (~line 457)
  covers the single-status path.
- `docs/cli.md`: `task list` section documents `--status` with single-value
  examples.

## Fix Direction

- Change the `--status` flag in `taskListCommand` to
  `cmd.Flags().StringSliceVar(...)` so it accepts a comma-separated list and
  repeated flags, matching the `--check` convention. Update the help text to
  "Filter tasks by status (comma-separated or repeatable)".
- Pass the slice through to `taskList`. Normalize each entry with
  `normalizeTaskStatus`; return the existing usage error
  (`unsupported task status ...`) on the first invalid entry, including empty
  entries produced by stray commas.
- Deduplicate normalized statuses, then filter with set membership. Either
  generalize `filterTasksByStatus` to take a set/slice or replace it; keep
  output ordering identical to today (priority rank, then task ID, from
  `filterTasks`).
- Single-status behavior must remain backward compatible:
  `--status pending` and `--status "in progress"` keep working unchanged.
- Update the `task list` section of `docs/cli.md` in the same change, adding a
  comma-separated example such as `ahm task list --status open,pending`.
- No ADR needed: this extends an existing flag following the established
  `--check` comma-separated convention and creates no new durable contract.

## Acceptance Notes

- [x] `ahm task list --status open,pending` lists tasks in either status,
      sorted by priority rank then ID.
- [x] Repeated flags work: `--status open --status pending` matches the
      comma-separated form.
- [x] Normalization still applies per entry (`--status "in progress,OPEN"`).
- [x] An invalid entry in the list fails with the existing usage error and
      non-zero exit code.
- [x] Single-status invocations behave exactly as before; `--json` output
      remains parsed task structs.
- [x] Tests cover multi-status, duplicate, invalid-entry, and repeated-flag
      cases (extend `TestTaskListFiltersStatus` or add a focused test).
- [x] `docs/cli.md` `task list` section updated with the new flag semantics
      and an example.
