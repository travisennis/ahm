# Fix ahm init output reporting legacy .agents index paths in .ahm layout

## Summary

After task 175, a fresh ahm init correctly creates the committed .ahm/ layout on disk, but the command output indexes: section still prints legacy paths (.agents/.research/index.md, .agents/.tasks/active/index.md, .agents/exec-plans/...) while the files are actually written under .ahm/. Observed 2026-07-12 smoke-testing commit 376d063: created/metadata sections print .ahm/ paths, indexes section prints .agents/ paths.

## Relevant Files

- internal/ahm/install.go or the index-writing report path (writeIndexes result labels)
- internal/ahm/indexes*.go (reported rel paths likely hardcoded to legacy layout)

## Fix Direction

Report index paths from workflowPathsFor-derived paths rather than hardcoded legacy constants so init/upgrade/index output matches the active layout. Cover with an assertion in the init output test.

## Acceptance Notes

- [x] Fresh init output lists index paths under .ahm/
  - Fixed `install()` in `internal/ahm/install.go` to compute index paths from the
    already-determined `recordsDir` variable via `workflowPaths{root, recordsDir}`
    rather than calling `a.indexWriteTargets()` which re-derives from on-disk
    metadata state. On fresh init the `.ahm/config.json` hasn't been written yet
    so `workflowPathsFor` returned `.agents/` paths.
- [x] Legacy repositories still report .agents/ index paths
  - The `install()` function preserves existing metadata layout: when
    `hasExistingMeta` is true, `recordsDir` comes from `workflowPathsFor(root)`
    which reads the on-disk metadata file, so legacy `.agents/ahm.json` repos
    continue to report `.agents/` index paths.
- [x] Test asserts output paths match the active layout
  - `TestInstallWritesExpectedScaffoldOutput` updated to assert all index paths
    in output are under `.ahm/` and no `.agents/` paths appear.
