This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

## Purpose / Big Picture

After this change, every managed write that `ahm` performs (metadata, generated indexes, task files, installed/upgraded templates) is crash-safe: a crash or power loss mid-write will never leave a truncated or corrupt file. The user simply re-runs the command and the write completes cleanly.

Concurrent `ahm` invocations on the same repository are also safe from interleaved corruption because each write is atomic (either fully visible or not visible at all).

## Surprises & Discoveries

- `os.Rename` on Unix is atomic when source and destination are on the same filesystem. Since we write the `.tmp` file in the same directory as the target, this guarantee holds.
- On macOS, directory `fsync` is required after rename to guarantee durability through a crash/power-loss. Without it, a crash before the directory metadata is flushed can leave the rename unapplied. The helper calls `fsync` on the parent directory.
- The `install` function has two separate `os.WriteFile` calls (for new files and updated files) plus the `writeMetadata` call. All three need routing through the atomic helper.

## Decision Log

- Decision: Use sibling `.tmp` files in the same directory rather than a separate temp directory.
  Rationale: Rename is only atomic across the same filesystem. Sibling files guarantee same-filesystem.
  Date/Author: 2026-05-29 / Travis

- Decision: Defer advisory locking to a future change.
  Rationale: Atomic writes eliminate corruption risk. Concurrent-invocation corruption is rare for a CLI tool. Cross-platform flock adds complexity without a demonstrated need.
  Date/Author: 2026-05-29 / Travis

- Decision: Clean up stale `.tmp` files opportunistically in init/upgrade/index, not on every command.
  Rationale: Most `ahm` commands are read-only (status, doctor, task list, task show). Running a directory scan on every command is unnecessary overhead.
  Date/Author: 2026-05-29 / Travis

## Plan of Work

### Step 1: Create the atomic write helper

Create `internal/ahm/write.go` with:

- `writeFileAtomic(path string, data []byte, perm os.FileMode) error` — the core helper.
- `cleanupStaleTemps(root string) error` — scans `.agents/` for `*.tmp` files and removes orphans.
- A package-level `sync.Mutex` is NOT used; atomicity comes from the filesystem rename.

The helper algorithm:

```
1. Create parent directories (MkdirAll).
2. Remove any existing <path>.tmp to clean up from a prior crash.
3. Write <path>.tmp with WriteFile.
4. Sync the temp file (syscall.Fsync on the fd, or file.Sync).
5. Rename <path>.tmp -> path.
6. Sync the parent directory.
```

For steps 4 and 6, use `*os.File` methods: open the file, call `.Sync()`, close.

### Step 2: Route all managed writes through the helper

Replace every occurrence of:

```go
os.WriteFile(path, data, perm)
```

in the write sites listed above with:

```go
writeFileAtomic(path, data, perm)
```

For `writeMetadata`, also change `os.MkdirAll` + `os.WriteFile` to just `writeFileAtomic` (which already creates parent dirs).

### Step 3: Add stale-temp cleanup to init/upgrade/index

In the `install` function (both `init` and `upgrade` paths) and in `writeIndexes`, call `cleanupStaleTemps(a.opts.root)` near the top of the function.

### Step 4: Add tests

Create `internal/ahm/write_test.go` with tests for:

- `writeFileAtomic` writes content correctly (read back and verify).
- `writeFileAtomic` creates parent directories.
- Failure simulation: if a write fails before rename, the original file is unchanged.
- Stale `.tmp` file is removed before a new write.
- Rejection of path traversal in path (no `/../` escaping the root).
- `cleanupStaleTemps` removes orphaned `.tmp` files.

### Step 5: Update docs and config

- Add `*.tmp` to `.gitignore` (line for `**/*.tmp` or `.agents/**/*.tmp`).
- Update `docs/spec.md` with a section describing the atomic write guarantee.
- Update the task ADR (ADR 001) reference if needed.

## Idempotence and Recovery

- All steps can be run repeatedly. The `writeFileAtomic` helper is idempotent: writing the same content twice produces the same result.
- If a crash occurs during implementation, re-running `ahm index` or `ahm init` will clean up any stale `.tmp` files.
- The `cleanupStaleTemps` function is also idempotent and safe to call multiple times.
