# ExecPlan: Split the ahm CLI implementation into focused modules

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

See `.agents/PLANS.md` for the ExecPlan conventions used in this repository.

## Purpose / Big Picture

`ahm` currently keeps most command wiring, workflow installation, metadata handling, validation, task parsing, task command behavior, index rendering, and output formatting in one large file, `internal/ahm/cli.go`. After this change, a contributor can navigate directly to the area they need without scanning a 2,400-line file. The observable behavior must stay the same: existing CLI commands such as `ahm status`, `ahm doctor`, `ahm index`, `ahm task list`, and `ahm task show` continue to produce the same results and the existing test suite still passes.

This is an internal refactor. It does not introduce new flags, new commands, new persisted data formats, or new user-visible workflow semantics.

## Context and Orientation

`ahm` is a Go CLI implemented with Cobra. Cobra is the command-line parser used by the project; command objects define a command name, flags, argument validation, and the function to run. The CLI entrypoint is `cmd/ahm/main.go`, which delegates into package `internal/ahm`.

Before this refactor, `internal/ahm/cli.go` contained nearly all CLI behavior. Its top-level declarations included the `options` and `app` structs, the exported `Main` function, root command construction, workflow install and metadata helpers, status and doctor commands, validation helpers, AGENTS.md suggestion helpers, output formatting, task parsing and rendering helpers, task command handlers, dependency helpers, and generated index rendering.

After this refactor, `internal/ahm/cli.go` contains command wiring and shared app/usage definitions. Focused implementation files now hold the moved behavior: `root.go` for root detection, `install.go` for install and metadata handling, `status.go` for status and doctor commands, `validation.go` for workflow validation, `agents.go` for AGENTS.md suggestions, `output.go` for shared output formatting, `tasks.go` for task model/parsing/rendering, `task_commands.go` for task command behavior, `task_deps.go` for task dependency commands, `task_migrate.go` for task front matter migration, and `indexes.go` for generated indexes.

`internal/ahm/write.go` contains atomic write helpers added by an earlier task. Keep it unchanged unless imports or nearby call sites require formatting.

`internal/ahm/cli_test.go` contains broad CLI tests. Task 030 is reserved for splitting this test file, so task 012 should avoid unnecessary test-file movement.

`docs/cli.md` is the user-facing CLI reference. Because this refactor did not change CLI behavior, the only docs update needed there was the implementation-location reference near the top of the file.

## Plan of Work

Start by marking the task in progress and regenerating indexes through `ahm` commands, not by hand. Then split `internal/ahm/cli.go` into files that match the current declaration clusters. Each move should preserve the exact package name, function signatures, receiver types, constants, variables, and comments unless a comment only made sense in the old location.

Create or populate these files:

`internal/ahm/cli.go` should keep `options`, `app`, `Main`, `errValidationFailed`, `usageError`, `noArgs`, `(*app).run`, `(*app).command`, `(*app).agentsCommand`, `(*app).taskCommand`, `(*app).simpleCommand`, and `(*app).lenientCommand`. This file remains the command wiring map.

`internal/ahm/root.go` should contain `(*app).detectRoot`, `(*app).detectRootOrCWD`, and `detectManagedRoot`.

`internal/ahm/install.go` should contain the `metadata` struct, `(*app).install`, `(*app).ensureWorkflowDirs`, `readMetadata`, `writeMetadata`, `hashBytes`, and `readWorkflowFile`.

`internal/ahm/status.go` should contain `(*app).status` and `(*app).doctor`.

`internal/ahm/validation.go` should contain `validationReport`, `validationFinding`, all `validate*` helpers, Markdown link validation helpers, task dependency cycle validation, and `(*validationReport).addError` / `addWarning`.

`internal/ahm/agents.go` should contain `agentsSuggestionsReport`, `agentSuggestionWithHit`, `(*app).agentsSuggestions`, `(*app).collectAgentSuggestions`, and `relPath`.

`internal/ahm/output.go` should contain `(*app).emit`.

`internal/ahm/tasks.go` should contain `taskFileInfo`, `Task`, task file collection, parsing, front matter helpers, list formatting helpers, task ID sorting helpers, task counts, and task rendering.

`internal/ahm/task_commands.go` should contain task command construction and behavior other than dependency and migration helpers: task creation, listing, enum validation, status changes, task resolution, and dependency completion helpers.

`internal/ahm/task_deps.go` should contain task dependency subcommand construction and behavior: `(*app).taskDepCommand`, `(*app).taskDepUpdateCommand`, `exactArgs` if it is only used there, `(*app).taskDepUpdate`, `(*app).taskDepTree`, and `(*app).taskDepCycles`.

`internal/ahm/task_migrate.go` should contain task migration behavior and front matter normalization helpers used by `ahm task migrate`.

`internal/ahm/indexes.go` should contain `(*app).writeIndexes`, stale index checks, index target and write collection, generated index target lists, sorted key helpers, root/bucket/research/ExecPlan index rendering, Markdown document collection, cell escaping, and status/priority/effort order helpers.

After each large cluster move, run a narrow compile check such as `go test ./internal/ahm -run TestMain`. If the exact test name does not exist, run `go test ./internal/ahm`. Use Go compiler errors to adjust imports rather than manually guessing all imports up front.
