# Detect duplicate task IDs across buckets in validation

## Problem

ADR validation reports `adr_duplicate_id`, but no equivalent exists for tasks. Task status transitions write the new bucket file before removing the old one (task_status.go, task_work.go, task_groom.go), so a crash between the two leaves the same ID in two buckets. `collectTasksForPaths` sorts with unstable sort.Slice keyed only on ID, so `resolveTask` picks an arbitrary copy, and generated indexes list the task twice. Validation at best emits a bucket-mismatch warning, and only when the statuses disagree.

## Relevant Files

- `internal/ahm/validation.go` (validateTaskFiles; compare adr_duplicate_id at validateADRs)
- `internal/ahm/task_status.go` (write-then-remove move window)
- `internal/ahm/tasks.go` (collectTasksForPaths sort)

## Fix Direction

Add a `task_duplicate_id` validation error listing the conflicting file paths when the same task ID parses from more than one bucket. Consider making task resolution refuse mutation commands on a duplicated ID so recovery is explicit.

## Decision

Mutation commands must refuse to resolve or change a duplicated task ID and
must name the conflicting paths plus the manual recovery action. Arbitrarily
selecting one copy risks mutating or deleting the wrong record; read-only list
and validation commands may continue so the repository remains diagnosable.
