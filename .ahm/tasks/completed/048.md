# Add acceptance-notes completeness check on task complete

## Summary

When a task is marked completed, parse its Acceptance section and warn (or
block, in strict mode) if the section is missing, still contains the seeded
`- [ ] TODO` placeholder, or has any unchecked items. Surface the same
condition as a validator finding for already-completed tasks so drift shows
up in `ahm doctor`.

## Problem

`ahm task create` seeds new tasks with `## Acceptance Notes\n\n- [ ] TODO`
(see `taskCreateParsed` in `internal/ahm/cli.go`). `AGENTS.md` instructs
agents to "fill in Acceptance Notes when practical" before
`ahm task complete <id>`, but nothing in `ahm` actually checks this. Tasks
get marked Completed with the placeholder still in place, and that drift
is invisible until a human reads the file.

## Relevant Files

- `internal/ahm/cli.go` for `taskStatus`, `taskCreateParsed`,
  `validateTaskFiles`, and the `validationReport` types.
- `internal/ahm/cli_test.go` for CLI and validator coverage.
- `internal/templates/workflow/TASKS.md` for the documented acceptance

- `docs/cli.md` for the `ahm task complete` behavior and the new flag.
- `docs/spec.md` for the persisted-config schema change.

## Fix Direction

Add a parser helper (e.g. `parseAcceptanceNotes(body []byte)`) that:

- Locates the first `##` or `###` heading whose trimmed, case-insensitive
  text matches one of `Acceptance Notes`, `Acceptance Criteria`, or
  `Acceptance`.
- Walks subsequent lines until the next heading at the same or higher
  level.
- Recognizes `- [ ]` and `* [ ]` (with optional leading indent) as
  unchecked items; treats the literal `- [ ] TODO` (case-insensitive) as
  a placeholder.
- Returns enough information to distinguish three outcomes: section
  missing, placeholder present, or unchecked items present.

Wire that helper into two integration points:

1. `taskStatus` when transitioning to `Completed`: print a stderr warning
   for each finding. In strict mode, return a non-zero error instead.
   Honor the existing `--force` flag as the override that downgrades
   strict-mode errors back to warnings. Preserve existing dry-run
   behavior.
2. `validateTaskFiles`: for tasks whose status is `Completed`, emit one
   of:
   - `task_acceptance_missing`
   - `task_acceptance_placeholder`
   - `task_acceptance_unchecked`

Add persisted strict-mode configuration to `.agents/ahm.json` (a new
`strict_acceptance` boolean alongside the existing metadata). Load it
during `detectRoot`/`readMetadata` so `taskStatus` can consult it
without a CLI flag. Document the field and an example in `docs/spec.md`.

Heading matching is intentionally flexible: any of the three accepted
phrasings, any case, `##` or `###`. Do not also match section bodies or
unrelated headings.

Because this changes user-visible CLI behavior and durable workflow
semantics, update `docs/cli.md` and `docs/spec.md` in the same change.
