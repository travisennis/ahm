# Add example invocations to usage error messages

## Summary

Usage errors like `task show requires an id` and `adr supersede requires --by` are clear about what's missing but don't show the correct form. The `writing-clis-for-agents` skill says: "On missing required flags: exit immediately with a clear message and a correct example invocation, not a hang."

## Evidence

Current behavior:

```
$ ahm task show
task show requires an id

$ ahm task create
task create requires a title

$ ahm adr supersede 001
adr supersede requires --by
```

Desired behavior (skill pattern):

```
task show requires an id
  ahm task show <id>

adr supersede requires --by
  ahm adr supersede <old-id> --by <new-id>
```

Commands with usage errors that need example invocations:
- `task show`, `task create`, `task work`, `task start`, `task complete`, `task cancel`, `task accept`, `task reopen`, `task comment`
- `task dep add`, `task dep remove`, `task dep tree`
- `adr create`, `adr show`, `adr accept`, `adr reject`, `adr deprecate`, `adr supersede`
- `context` (unknown scope)

## Fix Direction

Append a correct example invocation to each `usageError` message. The example should show the command with placeholder arguments (`<id>`, `<title>`, `<old-id> --by <new-id>`, etc.). Keep it on one indented line after the error message, matching the skill's pattern.

- [x] Every `usageError` for a missing required argument includes a correct example invocation
- [x] Examples use placeholder syntax (`<id>`, `<title>`, etc.) consistent with the command's `Use` field
