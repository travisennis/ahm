# Normalize supersede status check to handle non-canonical casing

## Summary

The supersede guard compares against canonical lowercase form. If someone hand-edits status to non-canonical case (e.g. 'Superseded by ADR-002'), the prefix check fails and the guard is bypassed.

## Problem

The guard preventing re-superseding an already-superseded ADR compares the canonical lowercase status form. If someone hand-edits the status to a non-canonical case (e.g. `Superseded by ADR-002`), the prefix check `strings.HasPrefix(oldADR.Status, "superseded by ADR-")` fails, the record is re-superseded with a second supersession status, and only the final write corrects the casing. The end state is consistent, but the pre-write guard is bypassed.

## Evidence

- **File:** `internal/ahm/adr_commands.go`, line 282
- **Context:**
  ```go
  if strings.HasPrefix(oldADR.Status, "superseded by ADR-") && oldADR.Status != nextStatus {
      return fmt.Errorf("ADR %s is already %s", oldADR.ID, oldADR.Status)
  }
  ```

## Fix Direction

Normalize `oldADR.Status` with `strings.ToLower` before the prefix check.

**Decision:** Use `strings.ToLower` rather than the `adrSupersededStatusPattern`
regex, because the regex (`^superseded by ADR-[0-9]{3}$`) is itself
case-sensitive (no `(?i)` flag). Normalizing with `ToLower` is simpler and
doesn't require touching the regex definition.
