# Fix duplicate ADR ID validation error pinning blame to single path

## Summary

The duplicate ID validation error pins blame to paths[0] (first alphabetically), but neither file is inherently the failing one — both contribute to the error.

## Problem

When two files share the same ADR ID, the duplicate ID validation error is pinned to `paths[0]` (the first alphabetically sorted path). The full list is in the error message, so the information is complete, but pinning to a single path is misleading—neither file is inherently the "failing" one.

## Evidence

- **File:** `internal/ahm/validation.go`, line 514
- **Context:** `report.addError("adr_duplicate_id", paths[0], fmt.Sprintf(...))`

## Fix Direction

Pass an empty path (empty string) to `report.addError` instead of `paths[0]`,
since the error message already lists all duplicate paths and neither file is
inherently the failing one.

**Decision:** Empty path approach chosen. A dedicated error code (alternative
approach) would add unnecessary complexity since the error code
`adr_duplicate_id` already describes the issue. An empty path avoids
misleading blame while keeping the error message fully informative.

## Acceptance Notes

- [x] Duplicate ADR ID validation error does not pin blame to a single path.
- [x] Error message still lists all duplicate paths.
- [x] Test explicitly verifies the reported path field (or absence) for duplicate ID errors.
