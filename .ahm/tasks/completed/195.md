# Make stale workflow lock reclamation ownership-safe

## Problem

`removeStaleWorkflowLock` stats a stale lock path and then calls `RemoveAll`. Between those operations another process can replace the stale directory with a live lock, which is then deleted. Lock release also treats an already-missing directory as success, hiding stolen ownership.

## Relevant Files

- `internal/ahm/lock.go` — acquisition, stale reclamation, and release.
- `internal/ahm/lock_test.go` — lock ownership and contention tests.
- `docs/adr/001-atomic-writes-and-concurrency.md` — concurrency guarantees.

## Fix Direction

Use an ownership-safe stale-lock reclamation protocol, such as atomically renaming the observed lock aside before deletion and validating ownership on release. Ensure only the successful claimer can remove the stale instance.

## Acceptance Notes

- [x] Stale reclamation atomically quarantines and identity-checks the observed directory, leaving a replacement lock intact.
- [x] Release identity-checks the acquired directory and returns ownership loss through command-level lock wrappers when it is missing or replaced.
- [x] `TestRemoveStaleWorkflowLock_DoesNotRemoveReplacement` deterministically replaces the lock between observation and reclamation.
- [x] Existing acquisition, contention, timeout, serialization, and stale-cleanup tests pass under the race detector; `just ci` passes.
