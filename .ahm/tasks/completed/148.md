# Reject newlines in task and ADR create fields rendered into front matter

## Summary

Free-text CLI inputs are rendered verbatim into workflow front matter, so values containing newlines inject arbitrary front matter fields or corrupt the file. Reproduced against commit ff15056: `ahm task create "Innocent task" --labels "$(printf 'type:task\nstatus: Completed')"` creates a task whose parsed status is `Completed` — bypassing the status enum validation and bucket rules — and a multi-line title writes a corrupt task file with duplicate front matter keys. This violates the architectural invariant that file formats are validated at the boundary (ARCHITECTURE.md).

## Evidence

- `internal/ahm/tasks.go:323-357` — `renderTask` writes `title`, `labels`, and other fields with raw `%s` into front matter lines.
- `internal/ahm/task_enum.go:8-19` — `validateTaskCreateEnums` validates status/priority/effort but never inspects `title` or `labels`.
- `internal/ahm/adr_commands.go:198-233` plus `internal/ahm/adrs.go:186-211` — `adr create --decision-makers` flows unvalidated into `renderADR` front matter the same way (`decision-makers: %s`). An ADR title with a newline breaks the rendered H1 (title is not front matter for ADRs).
- Downstream, `parseFrontMatter` (`internal/ahm/tasks.go:141-158`) silently accepts the injected lines, so the corruption round-trips.

## Fix Direction

Add boundary validation: reject values containing `\n` or `\r` with a usage error for `task create` title and labels, and `adr create` title and `--decision-makers`. Prefer rejection over escaping to keep the file format simple and deterministic. Optionally add a defensive guard in `renderTask`/`renderADR` (replace newlines with spaces) so future callers cannot corrupt files even if a new command forgets validation — if added, test it separately. Verify `task comment --author` only reaches the Markdown body (it appears to), and skip it if so. Out of scope: quoting/escaping schemes for the front matter format, changes to `parseFrontMatter`, and values already on disk (`task migrate` owns those).

Escape hatch: if other create/update flags render into front matter beyond the ones listed, extend the same validation and note them on completion.

## Acceptance Notes

- [x] `ahm task create` rejects titles and labels containing newlines with a usage error (exit code 2) and writes nothing.
- [x] `ahm adr create` rejects titles and --decision-makers values containing newlines the same way.
- [x] Tests added in `internal/ahm/task_commands_test.go` and `internal/ahm/adr_commands_test.go` following the existing usageError test patterns.
- [x] The labels-injection repro above no longer produces a Completed task.
- [x] `just ci` passes.

## Escape Hatch Note

Other create/update flags that render into front matter were checked:
- `task comment --author` only reaches the Markdown body (via `formatComment`), not front matter.
- `work task --agent` is enum-validated.
- No other task/adr create flags render user input into front matter besides title, labels, and decision-makers.
- Defensive guards added in `renderTask` (title, labels) and `renderADR` (title, decision-makers) to replace newlines with spaces as a safety net for future callers.
