# Auto-unblock tasks after dependency completion

## Summary

When ahm task complete finishes a dependency, automatically transition dependent Blocked tasks to Pending once all of their depends_on tasks are Completed, then regenerate indexes so newly unblocked work appears in the ready queue.

## Problem

Completing task 075 left task 076 in explicit `Blocked` status even though its
only dependency, `075`, is now `Completed`. The current ready/blocked filtering
already treats `Pending` tasks with incomplete dependencies as blocked, but it
does not automatically change explicit `Blocked` tasks back to `Pending` when
their dependency set becomes satisfied.

## Relevant Files

- `internal/ahm/task_commands.go`
- `internal/ahm/task_commands_test.go`
- `docs/cli.md`
- `.agents/TASKS.md`

## Fix Direction

- Add dependency-aware unblock logic to the `task complete` path.
- After the completed task is written and the old active file is removed, scan
  active tasks for `status: Blocked` tasks whose `depends_on` list includes the
  completed task.
- Move only tasks whose full dependency list is now completed to `Pending`,
  stamp `updated`, rewrite them in place under `.agents/.tasks/active/`, and
  include them in the same index regeneration.
- Do not unblock tasks that are blocked for reasons other than dependencies if
  any dependency remains incomplete or the task has no dependency relationship
  to the completed task.
- Preserve `--dry-run` semantics by reporting the completed task move plus any
  tasks that would be unblocked without writing files.
- Document the behavior for `task complete`.
