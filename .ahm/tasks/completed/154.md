# Spike: design ExecPlan and research lifecycle commands

## Blocker

Blocked on task 138 because the design for ExecPlan and research lifecycle
commands must account for the accepted ref-backed records storage and write-path
model.

## Summary

Design spike, not a build task. Tasks and ADRs both have full CLI lifecycles (`task create/list/show/complete/...`; `adr create/list/show/accept/supersede/migrate` — ADRs 003, 009), while ExecPlans and research notes have templates, directories, generated indexes, and validation (ADR 004; the `exec_plan` task field; `ahm context plan|research`) but zero CLI verbs: agents are instructed to hand-create files that ahm then validates and indexes. The most error-prone steps (correct bucket, filename, mandatory sections) are unassisted. Deliverable: a written design for the command surface, not an implementation. Written against commit ff15056.

## Evidence

- `internal/ahm/adr_commands.go` and `internal/ahm/task_commands.go` — full lifecycle verbs exist for ADRs and tasks.
- `internal/ahm/cli.go` command wiring — no `plan` or `research` command exists.
- `internal/ahm/validation.go:247-383` — ExecPlans carry mandatory-section and lifecycle validation that hand-created files routinely trip (`exec_plan_missing_section`, bucket/outcomes rules).
- `internal/ahm/indexes.go:84-110` — research and exec-plan indexes are generated from hand-created files.
- `internal/ahm/install.go:209-226` — `ensureWorkflowDirs` installs the `.agents/exec-plans/` and `.agents/.research/` conventions.

## Fix Direction

Investigate and produce a design covering: which verbs pay their way (`plan create/list/complete`? `research add/list/archive`?), how `plan complete` interacts with ExecPlan lifecycle validation (Outcomes & Retrospective gating), how templates are sourced (embedded vs. installed), and how the commands interact with ref-backed record storage (ADR 013 / task 138) — this task is blocked on 138 because the storage shape determines the write paths. Enumerate open questions and a proposed follow-up task breakdown. Do NOT implement commands in this task. Output: a proposed ADR (via `ahm adr create`) or an ExecPlan, plus the follow-up task list.
