# Reject overflowing numeric task IDs

## Problem

`splitTaskID` discards `strconv.Atoi` errors. A numeric prefix larger than `int` can be treated as a valid ID with a clamped value, corrupting ordering, prefix resolution, or next-ID allocation.

## Relevant Files

- `internal/ahm/tasks.go` — task ID splitting and ordering.
- `internal/ahm/task_create.go`, `task_find.go` — allocation and resolution consumers.
- `internal/ahm/tasks_test.go` — ID parsing edge cases.

## Fix Direction

Propagate conversion failure through the existing `ok` result and ensure all callers reject or ignore overflowing numeric IDs consistently.

## Acceptance Notes

- [x] Overflowing numeric prefixes, with or without suffixes, return
  `ok=false` and preserve the original ID as the nonnumeric value.
- [x] Regression coverage proves overflowing parsed IDs and filenames cannot
  influence next task ID allocation.
- [x] Existing valid-ID sorting and resolution tests remain green.

Verification: `go test ./internal/ahm -run 'Test(SplitTaskID|NextTaskID|ResolveTask|SortTaskListFields)'`; `just ci`.
