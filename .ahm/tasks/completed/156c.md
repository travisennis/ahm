# Add ahm task groom delegation command

## Summary

Add `ahm task groom [<id>]`: delegate backlog grooming to an external coding
agent using the `ahm task work` orchestration, require a schema-constrained
JSON verdict per task, and apply the verdicts mechanically through ahm's own
task machinery.

## Problem

The grooming procedure lives in
`internal/templates/workflow/grooming-backlog-SKILL.md` (~124 lines),
installed to `.agents/skills/grooming-backlog/SKILL.md`. Per ADR 014 (revised
2026-07-07), skills stop being installed and grooming becomes a delegation
command, not a scoped context: the delegated agent supplies judgment, and
`ahm` performs every resulting record write, so label vocabulary, timestamps,
status transitions, and index regeneration hold no matter which agent ran.

## Design

Scope: with no argument, groom all `Open` tasks plus a staleness review of
`Blocked` tasks (with their `depends_on` values); with `<id>`, groom that task
only. Only `Open` tasks are eligible for acceptance.

Prompt: procedure text adapted from the SKILL.md body (drop skill frontmatter
and "when to use" framing) composed with live state тАФ Open task lines, Blocked
task lines with dependencies, label vocabulary (`ahm task labels` data), and
validation findings. Compose in Go from the same internal listing functions
the task commands use (`internal/ahm/task_list.go`); do not shell out to
`ahm`. The prompt should tell the agent to inspect task files and code as
needed before judging readiness (it runs in the repository).

Delegation: reuse the task-work orchestration (`internal/ahm/task_work.go`,
`task_session.go`, `task_agents.go`) тАФ agent selection via
`default_work_agent`/`--agent`, execution from the repository root, session
capture per task 163. The global `--dry-run` flag prints the composed prompt
and performs no delegation and no writes.

Result contract: a schema-constrained JSON final answer. Supply the schema
per agent (`--json-schema` for claude, `--output-schema` for codex exec;
cake is expected to gain equivalent support; embed the schema in the prompt
for agents without one). Draft verdict shape (exact schema is
implementation-owned):

```json
{"verdicts": [{"task": "157", "action": "accept",
  "comment": "тАж", "add_deps": [], "remove_deps": [], "labels": []}]}
```

`action` is `accept` or `comment`. `ahm` validates the result against the
schema before any write; on invalid or missing output it writes nothing,
preserves the raw output for inspection, and exits nonzero.

Appliers (write authority per ADR 014):

- `accept` тЖТ `Open` to `Pending` via the existing accept path
  (`internal/ahm/task_status.go`), only when the verdict states nothing
  prevents the task from being worked.
- `comment` тЖТ timestamped comment via the existing comment path
  (`internal/ahm/task_comment.go`) stating what the task still needs.
- Dependency corrections via the existing dep logic
  (`internal/ahm/task_deps.go`); label normalization against the existing
  vocabulary.
- No autonomous cancellation; cancel recommendations arrive as comments.

The apply summary (what was accepted, commented, changed) supports text,
`--plain`, and `--json` from the same struct.
