# Warn when completing tasks linked to active ExecPlans

## Summary

`ahm context plan` already tells agents to move a completed ExecPlan to the completed bucket and update the task `exec_plan` field before running `ahm task complete`, but `task complete` itself does not recognize the linked ExecPlan lifecycle. A user or agent can complete a task first, which creates `task_completed_exec_plan_active` and `exec_plan_active_with_outcomes` warnings that only show up later in `ahm status`. Add a targeted completion-time guard or warning for this specific workflow.

Written against commit `5cbd271`.

## Evidence

- `internal/templates/workflow/PLANS.md:15` - the documented order is to fill acceptance notes, update the ExecPlan outcomes, move the ExecPlan to completed, update the task `exec_plan`, and then run `ahm task complete`.
- `internal/ahm/task_status.go:113` - completion validates acceptance notes, but not linked ExecPlan lifecycle state.
- `internal/ahm/validation.go:264` - status validation later warns when a completed task references an active ExecPlan.
- Recent cleanup had to move `.agents/exec-plans/active/138-ref-backed-workflow-records.md` to `completed/` and update tasks 138, 139, and 141-145 after `ahm status` exposed the drift.

## Fix Direction

Teach `task complete` to inspect `task.ExecPlan` before writing the completed task. If the reference resolves to the active ExecPlan bucket, emit a clear warning that the ExecPlan should be moved to the completed bucket and the task `exec_plan` path updated before completion; if the active plan already has a filled Outcomes & Retrospective section, make the message especially direct because the plan is ready to complete. Keep the first implementation warning-only unless project docs are updated to introduce a stricter flag or setting. Do not implement a broad `ahm plan complete` command here; task 154 already covers the broader plan/research command design.
