# Decide structured task revisions during grooming

## Problem

`ahm task groom` can correctly identify fixable gaps in an Open task, but ADR 014 grants its result applier authority only to accept, comment, correct dependencies, and normalize labels. The delegated agent cannot propose or apply revisions to the task body, acceptance notes, effort, or readiness status. As task 176 demonstrated, the only schema-valid outcome for a fixable task is therefore a comment that leaves the task ungroomed.

Expanding autonomous write authority changes a durable orchestration and workflow-record contract. The safety boundary, revision representation, validation rules, provenance behavior, and relationship between revision and acceptance must be decided before implementation.

## Relevant Files

- `docs/adr/014-replace-managed-skills-with-command-based-procedures.md` — current grooming authority and autonomy decision.
- `internal/ahm/task_groom.go` — current prompt, result schema, validation, and applier.
- `internal/ahm/tasks.go` — task parsing and rendering behavior that revisions must preserve.
- `internal/ahm/task_acceptance.go` — existing acceptance-note checks.
- `docs/references/cli/task-commands.md` — documented grooming behavior.

## Fix Direction

Revise ADR 014 or add a superseding ADR that decides how delegated grooming may return structured task revisions. Prefer an ahm-applied representation over allowing the external agent to edit files directly. Specify which body sections and metadata may change, how comments and unknown front-matter fields are preserved, which changes remain human-owned, how revisions are validated, whether a valid revision may be accepted atomically, and how dry-run/JSON output exposes proposed and applied changes.

The decision should explicitly evaluate full-body replacement, section-level replacement, and patch-like representations. It should preserve the guarantee that ahm writes only workflow records and that invalid delegated output causes no writes.
