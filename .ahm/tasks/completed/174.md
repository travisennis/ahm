# Support multiple task IDs in ahm task show

## Summary

Allow ahm task show to accept multiple task IDs (e.g. ahm task show 001 002 003) and print each task's contents, in all output modes (text, plain, json).

## Acceptance Notes

- [x] `ahm task show 001` remains backward compatible: prints raw file in text mode; emits single JSON object in `--json` mode
- [x] `ahm task show 001 002 003` prints each task file's raw content separated by `---\n` in text mode
- [x] `ahm --json task show 001 002` emits a JSON array of task objects
- [x] `ahm --json task show 001` emits a single JSON object (not an array) for backward compatibility
- [x] `ahm task show 999` with a nonexistent ID prints an error and exits non-zero
- [x] `ahm task show 001 999 002` prints the two found tasks, prints an error for 999, and exits non-zero
- [x] `ahm task show` with no arguments prints a usage error message
- [x] `ahm task show --help` shows updated usage `ahm task show <id> [<id>...]`
- [x] Docs updated in `docs/references/cli/task-commands.md` with new usage, examples, and behavior description
- [x] Tests added for: single-ID backward compat, multi-ID text output, multi-ID JSON array output, partial failure, no-arg error, and separator in text mode

## Relevant Files

- `internal/ahm/task_commands.go` — Change arg validation from exactArgs(1, ...) to accept 1+ args; update help text usage from `<id>` to `<id> [<id>...]`
- `internal/ahm/task_list.go` — Update `taskShow` to iterate over multiple args, handle partial resolution errors, format multi-task text output with separator, and emit JSON array when 2+ args
- `docs/references/cli/task-commands.md` — Update `task show` section for plural usage, text separator, and JSON array output
- `internal/ahm/context.go` (if applicable) — Update `ahm task show <id>` example references to show plural form if used in generated agent instructions
- `internal/ahm/prime.go` (if applicable) — Same as context.go

## Fix Direction

1. Replace `exactArgs(1, ...)` in `task_commands.go` with a `minimumArgs(1, ...)` validator that accepts 1+ positional args and prints a usage error like `"task show requires at least one id\n  ahm task show <id> [<id>...]"`.

2. Update `taskShow` in `task_list.go`:
   - Iterate over all args, calling `a.resolveTask(argv[i])` for each.
   - Collect successful tasks and error messages separately.
   - When `a.opts.json` is true: if exactly 1 arg resolved, emit a single object (backward compatible). If 2+ args, emit a JSON array of objects. If no tasks found, emit null (or empty array) and return the combined error.
   - When not JSON: for each resolved task, read and write the raw file content. When printing 2+ tasks, separate with `\n---\n` for visual clarity. Collect warnings for unresolved IDs on stderr.
   - Return a combined error (via `errors.Join`) if any IDs could not be resolved.

3. Update usage string from `<id>` to `<id> [<id>...]` and the `Long` help text to describe plural behavior.

4. Update `docs/references/cli/task-commands.md` section `### \`task show <id>\`` → `### \`task show <id> [<id>...]\`` with:
   - Description of multi-ID support
   - Text mode separator behavior
   - JSON mode array output for 2+ args, single object for 1 arg
   - Examples: single ID, multiple IDs, JSON with multiple IDs

5. Update `context.go` and `prime.go` only if their generated agent instructions embed `<id>` in a way that would mislead; the canonical `<id>` placeholder is still valid as a minimal example even after plural support.
