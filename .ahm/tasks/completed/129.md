# Document root command exit code behavior in help

## Summary

Running `ahm` with no args runs `status`, which exits 1 when validation has errors. Some agent frameworks treat any non-zero exit as failure and may discard the output. The `context` command (the recommended briefing command) doesn't have this problem — it always exits 0.

This is documented and intentional, not a bug. But the root `--help` doesn't mention it. An agent that runs `ahm` as its first command gets a failure exit code and may not read the output. `ahm context` is the safer briefing entry point.

## Evidence

```
$ ahm
installed: true
installed_version: 0.4.3
...
---EXIT: 1
```

The exit code 1 comes from `errValidationFailed` in `status()` when validation errors exist.

- `internal/ahm/cli.go` — root command `RunE` calls `a.status()`.
- `internal/ahm/status.go` — `status()` returns `errValidationFailed` when `len(validation.Errors) > 0`.
- `internal/ahm/cli.go` — `Main()` maps `errValidationFailed` to exit code 1.
- `internal/ahm/context.go` — `context()` always returns nil (exit 0) regardless of validation state.

The docs (`docs/references/cli/global-contract.md`) document the exit code behavior but the root `--help` output doesn't mention it.

## Fix Direction

Add a note to the root command's `Long` field (or `Short`) that mentions `ahm context` as the non-failing briefing alternative:

```
Manage repo-local .agents workflows.

When run with no command, ahm runs 'status' which exits with code 1
when validation errors are found. For a briefing that always succeeds,
run 'ahm context'.
```

No behavioral change — this is a documentation/help-text improvement only.
