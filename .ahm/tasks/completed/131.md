# contextReport silently discards getTasks error when metadata is missing

## Summary

In `internal/ahm/context.go`, when workflow metadata is missing (`metaErr != nil`), the fallback path calls `tasks, _ = a.getTasks()` and discards the error. If `getTasks()` fails (e.g., corrupt task file, unreadable directory), the context report shows zero tasks with no indication of the problem. The user sees `tasks: open=0 ready=0 blocked=0 in_progress=0` when tasks actually exist but couldn't be parsed.

## Evidence

- `internal/ahm/context.go:119` — `tasks, _ = a.getTasks()` in the `metaErr != nil` branch.
- `internal/ahm/context.go:139` — `contextTaskSummary` uses the (possibly nil/empty) tasks to build the report, producing misleading zero counts.
- The `metaErr == nil` branch (line 117) gets tasks from `validateWorkflow`, which also parses tasks — but the error path bypasses this entirely.

## Fix Direction

Change `tasks, _ = a.getTasks()` to assign the error to a variable and call `a.addWarning("some task files could not be parsed and were skipped")` when `err != nil`, matching the pattern used in `taskList`, `taskSearch`, and other task-collecting commands.

This is a two-line change: capture the error, add a warning. No structural changes needed.

Do NOT change the `metaErr == nil` path (where tasks come from validation). Do NOT change the `contextTaskSummary` signature or behavior.
