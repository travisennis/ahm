# Add stale research-inbox disposition: doctor check and prime surfacing

## Context

Research inbox notes accumulate without a lifecycle owner. Observed in the cake repository (an ahm 0.4.x consumer): inbox stubs (`cli-approaches.md`, `code-mode.md`, and formerly `documentation-verification.md`) sat indefinitely — `ahm prime` lists them under Recent Research, but nothing ever forces a disposition, so the inbox rots silently. The same smell shows up as `exec_plan_orphan` infos for completed ExecPlans: records that exit their lifecycle without a disposition.

Downstream repos should not each invent a local grooming rule for this; the workflow owner (ahm) should ship it, so every managed repo gets it via `ahm upgrade`.

## Proposed shape

1. **Doctor check**: a new validation code (e.g. `research_inbox_stale`) — warn (or info) when a note under `.ahm/research/inbox/` is older than a threshold. Age basis: front matter date if present, else file mtime/git. Threshold configurable in `.ahm/config.json` (e.g. `research.inbox_stale_days`, default ~21) with a way to disable. The message should name the three dispositions so the fix is legible at the warning: promote to `research/topics/`, convert to a task, or delete.
2. **Prime surfacing**: show note age in the Recent Research section of `ahm prime` output, so staleness is visible at session start where it can change behavior, and flag notes past the threshold.
3. Consider whether the disposition guidance also belongs in `ahm context research` output.

## Constraints

- New validation code is additive to the compatibility surface (validation codes are a documented compatibility surface); do not change the meaning of existing codes.
- Applies to `.ahm/` layout; decide explicitly whether legacy `.agents/` layouts get the check or are out of scope.
- No behavior that mutates records automatically — disposition stays a human/agent decision; ahm only reports.
