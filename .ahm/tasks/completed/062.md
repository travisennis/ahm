# Surface task parse failures during index generation

## Problem

`indexes.go:64` (`indexWrites`) discards the task-collection error:
`tasks, _ := a.getTasks()`. Consequences:

- `ahm index` silently omits malformed task files from all generated indexes.
  Other commands (`task list`, `task next`, `resolveTask`) at least print
  "warning: some task files could not be parsed and were skipped"; the index
  path prints nothing. Since the indexes are the agent-facing task queue, a
  silently dropped task means work silently stops being scheduled.
- Worst case: if `taskFilePaths` fails outright (directory read error),
  `collectTasks` returns nil tasks and all task indexes are regenerated as
  empty — destroying index content on a transient I/O error.

## Fix Direction

- In `writeIndexes`/`indexWrites`, when `getTasks` reports errors, print the
  standard warning (and ideally the per-file errors, which
  `collectTasks` already joins) to stderr.
- Distinguish partial parse failures (proceed with warning) from total
  collection failure (`taskFilePaths` error): abort index regeneration rather
  than writing empty indexes.
- Note `install` also calls `writeIndexes`; behavior should be consistent.
