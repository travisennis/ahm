# Reject unsatisfiable dependencies in task dep add

## Problem

`task_deps.go:70` (`taskDepUpdate`) accepts dependency edges that can never be
satisfied:

- Self-dependency: `ahm task dep add 5 5` succeeds, leaving the task
  permanently blocked.
- Cycles: `dep add` performs no cycle check at write time; cycles are only
  discovered later via `dep cycles` or validation errors.
- Dependencies on Cancelled tasks: `depsComplete`
  (`task_commands.go:532`) only counts `Completed`, so a task depending on a
  Cancelled task is never ready and `task complete` refuses with "incomplete
  dependencies". `validateTaskDependencies` (`validation.go:175`) does not
  flag this because the cancelled task still exists in `byID` — the task is
  silently stuck forever.

## Fix Direction

- Error on self-dependency in `taskDepUpdate`.
- After computing the new dependency set, run `taskDependencyCycles` on the
  would-be result and reject (or at minimum warn) when the add introduces a
  cycle.
- Reject or warn when the dependency's status is Cancelled.
- Add a validation finding (e.g. `task_dependency_cancelled`) for
  non-completed tasks depending on Cancelled tasks, so existing repositories
  surface the condition.

## Decision

All three cases — self-dependency, cycles, and depending on a Cancelled task —
will be **rejected** (not warned). Rejecting is safer: silent warnings in
dependency semantics create permanent roadblocks that are hard to debug.
Validation will additionally report existing cancelled-task dependencies as a
finding so existing repositories can surface the condition.
