# Improve task command scalability for large queues

## Summary

Review issue 20. task create and dependency commands scan all task files and rewrite all indexes, which can become slow with thousands of tasks. Acceptance: measure current behavior, document acceptable scale, and introduce caching or incremental updates if warranted.

## Acceptance Notes

- [x] Added in-memory task caching to `app` struct (`getTasks`/`invalidateTasks`)
- [x] Replaced redundant `collectTasks` calls in all app methods with `getTasks`
- [x] Cache invalidates in `writeIndexes` so mutations see fresh data
- [x] Before: `task create` read all task files twice, `task complete` read 3×, `task dep add` read 3×
- [x] After: each command reads all task files once (first call populates cache, subsequent calls reuse it)
- [x] `indexWrites` uses `getTasks` so it benefits from caching in app methods; standalone validation path (which creates a temporary `app`) still reads from disk directly
- [x] All tests pass, `go vet` and `gofmt` clean, full CI passes
