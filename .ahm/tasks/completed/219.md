# Reject grooming of terminal task records

## Summary

At survey baseline commit `5ead0ae`, an explicit `ahm task groom <id>` accepts a task in any lifecycle state. A `comment` or `revise` verdict for a Completed or Cancelled task passes semantic validation, after which the applier writes the record into the active bucket and removes its terminal-bucket file while retaining the terminal status. This violates the task bucket/status invariant and ADR 017's explicit exclusion of terminal records from grooming authority.

## Evidence

- `internal/ahm/task_groom.go:285-300` — `groomTargets` returns an explicitly matched task without checking its status; only the no-ID path restricts targets to Open and Blocked.
- `internal/ahm/task_groom.go:484-537` — validation rejects `accept` for non-Open tasks but permits `comment` and `revise` verdicts for Completed or Cancelled targets.
- `internal/ahm/task_groom.go:655-667` — `applyGroomVerdicts` writes every verdict to `taskFile("active", task.ID)` and removes the original path when it differs.
- `docs/adr/017-apply-structured-task-revisions-during-grooming.md:100-106` — Completed and Cancelled records are outside the permitted revision surface.
- `internal/ahm/task_groom_test.go` contains no explicit-ID coverage for terminal task buckets.

## Fix Direction

Restrict explicit grooming targets to the lifecycle states the command is authorized to groom: Open and Blocked. Reject unsupported states before external delegation with an actionable error, and retain a defensive status check in semantic validation/application so direct helper use cannot relocate a terminal record. Keep `task comment` behavior and ordinary lifecycle commands out of scope.

## Decision

Explicit and batch grooming are limited to Open and Blocked tasks. Pending, In Progress, Tracking, Completed, and Cancelled records must be rejected before delegation; expanding grooming authority to another lifecycle state requires a separate documented decision.
