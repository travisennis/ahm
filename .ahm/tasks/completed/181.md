# Keep Comments last when groom adds new task sections

## Summary

When `ahm task groom` applies a revision that introduces a section the task
body does not already have (for example Relevant Files or Fix Direction), the
new section is appended to the end of the body. If the task already has a
`## Comments` section — which `ahm task comment` keeps at the end of the
file — the appended sections land after Comments, stranding the comment log
in the middle of the task description. The groom flow then appends its own
comment into that mid-file Comments block, which can also drop the blank line
before the following heading. Observed on task 175 during grooming on
2026-07-12.

## Relevant Files

- internal/ahm/task_groom.go (`replaceGroomSection` append fallback when `start < 0` concatenates the new section onto the end of the body)
- internal/ahm/task_groom_test.go
- internal/ahm/task_comment.go (comment appending assumes Comments stays last)

## Fix Direction

Treat `## Comments` as an always-last section. In the `replaceGroomSection`
fallback that handles a missing section, insert the new section before an
existing `## Comments` heading instead of appending to the end of the body,
preserving a blank line before the heading. Add a regression test that grooms
a task that already has comments and asserts the revised body keeps Comments
as the final section with correct spacing.

## Acceptance Notes

- [x] Groom revisions that add new sections insert them before an existing `## Comments` section
- [x] Comments remains the final section after groom apply, with a blank line separating sections
- [x] Regression test covers grooming a task that already has comments
