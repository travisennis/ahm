# Add --priority and --effort filters to task list

## Summary

`task list` supports filtering by `--status` and `--label` but not by `--priority` or `--effort`. An agent wanting only P0 tasks has to list everything and filter client-side. This breaks the filter pattern established by the existing flags.

## Evidence

- `internal/ahm/task_commands.go` — `taskListCommand()` wires `--status` (only for `all` mode) and `--label` flags. No `--priority` or `--effort` flags exist.
- `internal/ahm/task_list.go` — `taskList()` filters by status and labels. No priority or effort filtering.
- `internal/ahm/task_enum.go` — `validTaskPriority()` and `validTaskEffort()` already exist for enum validation, and `priorityOrder()` and `effortOrder()` return the valid values.

## Fix Direction

1. Add `--priority strings` and `--effort strings` flags to `task list` (the `all` mode variant), matching the `--status` and `--label` pattern: comma-separated or repeatable.

2. Add `filterTasksByPriority(tasks []Task, allowed map[string]bool)` and `filterTasksByEffort(tasks []Task, allowed map[string]bool)` helpers in `task_list.go`, following the same pattern as `filterTasksByStatus`.

3. Validate priority and effort values using the existing enum validators. Reuse `normalizeTaskStatus` as a pattern for a `normalizeTaskPriority` / `normalizeTaskEffort` if case-insensitive matching is desired (recommended for consistency with `--status`).

4. Apply filters in `taskList()` after the existing status and label filters.

## Acceptance Notes

- [x] `task list --priority P0` filters tasks to only P0 priority
- [x] `task list --priority P0,P1` or `--priority P0 --priority P1` filters to multiple priorities
- [x] `task list --effort S,M` filters tasks to only S and M effort
- [x] Invalid priority/effort values produce a usage error with supported values listed
- [x] Filters compose with `--status` and `--label` (AND semantics)
- [x] `task list --priority P0 --json` works correctly
- [x] `just ci` passes
