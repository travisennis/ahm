# Harden groom verdict application against partial failure

## Problem

`applyGroomVerdicts` writes verdicts one file at a time; an error mid-loop aborts before `writeIndexes`, leaving some verdicts applied, others not, indexes stale, and the summary lost. Its `os.Remove(task.Path)` also fails on ErrNotExist, unlike every other bucket-move site, so a benign missing file aborts the batch. The groom help text ("ahm validates the complete result before applying") implies more atomicity than exists.

## Relevant Files

- `internal/ahm/task_groom.go` (applyGroomVerdicts)
- `internal/ahm/task_status.go` (tolerant os.Remove pattern for comparison)

## Fix Direction

Tolerate ErrNotExist on the post-move remove; on mid-batch failure, still attempt index regeneration and report which verdicts were applied vs skipped instead of discarding the summary. Align the help text with the per-file failure model if full atomicity is not wanted.

## Acceptance Notes

- [ ] Missing source file during a groom move no longer aborts the batch
- [ ] Mid-batch failure reports applied vs unapplied verdicts and leaves indexes regenerated
- [ ] Help text matches actual atomicity guarantees
