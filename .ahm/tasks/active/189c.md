# Design durable resumable task work runs

## Blocker

Blocked on task 163. Durable run-state design assumes every supported
session-capable provider reliably creates and exposes a resumable session;
task 163 has implemented that invariant but still awaits required live smoke
evidence for externally limited providers.

## Problem

Task-work session IDs and phase progress live only in memory for one process invocation. An interruption after implementation, review, or finalization loses ahm's orchestration state even when the external provider session remains resumable. Users cannot inspect which phase completed, safely continue the run, distinguish an abandoned run from active work, or recover after a timeout without manually reconstructing provider commands.

## Relevant Context

- Task 163 — establishes session creation and capture as an invariant but does not define durable run records.
- Completed task 056 — deliberately limited earlier work to in-process capture and rejected session metadata at that time.
- `internal/ahm/task_session.go` — current in-memory phase sequencing.
- `.ahm/` generated/local-only ownership rules — candidate location and compatibility constraints for run state.

## Fix Direction

Produce an ADR for a local-only task-work run record containing task ID, run ID, phase state, provider session IDs, selected agents/models, CLI versions when available, timestamps, timeout/budget data, review result references, verification evidence, and terminal outcome. Define atomic writes, crash recovery, stale-run detection, redaction, cleanup, concurrent worktrees, and commands such as status/resume/abort. The record must not contain provider credentials or become committed workflow state by default.

## Acceptance Notes

- [ ] An accepted ADR reconciles the new persistence requirement with completed task 056 and the committed/local-only `.ahm` ownership model.
- [ ] The run state machine and legal transitions for implementation, review, finalization, verification, awaiting commit, commit, failure, interruption, and abort are specified.
- [ ] Atomic persistence, redaction, retention, cleanup, concurrency, and crash-recovery behavior are defined.
- [ ] `ahm task work status`, resume, and abort UX plus JSON output and exit behavior are specified.
- [ ] Resume behavior validates the task/worktree baseline before invoking an existing provider session.
- [ ] Concrete implementation tasks and an ExecPlan are created after the ADR is accepted.
