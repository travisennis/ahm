# Handle front matter closing delimiters at EOF

## Problem

`splitFrontMatter` recognizes a closing delimiter only when followed by a newline. A Markdown file ending exactly at `---` is treated as though it has no front matter, producing misleading downstream behavior instead of a precise malformed/unsupported boundary result.

## Relevant Files

- `internal/ahm/tasks.go` — shared front-matter splitter.
- `internal/ahm/adrs.go`, `task_migrate.go` — splitter consumers.
- `internal/ahm/tasks_test.go`, `task_migrate_test.go` — parsing tests.

## Fix Direction

Define EOF behavior for a closing delimiter and make the splitter distinguish absent front matter from malformed front matter where callers need an error. Cover LF and CRLF inputs.

## Acceptance Notes

- [ ] A closing delimiter at EOF is handled according to the documented grammar.
- [ ] Unterminated front matter produces an explicit parse/validation error rather than appearing absent.
- [ ] Task, ADR, and migration consumers have consistent tests.
