# Refuse delegated task work when dependencies cannot be parsed

## Problem

When a task has `depends_on` entries, `ensureTaskDependenciesComplete` calls
`a.getTasks()` to certify that every dependency is Completed. If any task file
fails to parse, `getTasks` returns a partial list and an error, but the helper
continues with that incomplete list. A completed dependency whose record cannot
be parsed is then reported as merely incomplete, hiding the actual workflow
corruption. More importantly, dependency readiness cannot be certified from a
partial task set, so external work should not start until parsing succeeds.

## Relevant Files

- `internal/ahm/task_work.go`
- `internal/ahm/task_commands_test.go`

## Fix Direction

When `a.getTasks()` returns an error during dependency validation, stop before
agent resolution, task status mutation, or subprocess lookup. Return a clear
error that dependency readiness cannot be checked because task files are
unparseable and retain the existing warning behavior without printing duplicate
warnings. Do not report the parse failure as an `incomplete dependencies` result.
Tasks with a fully parsed task set keep their current dependency behavior.

## Acceptance Notes

- [ ] A task with dependencies is not started or delegated when any task record fails to parse
- [ ] The error distinguishes an unparseable task set from genuinely incomplete dependency IDs and directs the user to repair the records
- [ ] No agent executable lookup or task status mutation occurs on the parse-error path
- [ ] The existing parse warning is emitted at most once
- [ ] Completed and incomplete dependency behavior is unchanged when all task files parse
- [ ] Focused `task work` tests and the existing package tests pass
