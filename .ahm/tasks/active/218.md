# Enforce read-only permissions for advisory agent delegations

## Summary

At survey baseline commit `5ead0ae`, `ahm audit` and `ahm task groom` are documented as read-only advisory delegations whose only writes flow through ahm's validated workflow-record appliers. Codex advisory runs instead receive the full bypass-sandbox flag, and Cursor advisory runs inherit `--trust`, so the external process retains authority to modify project source or Git state before ahm validates its structured response. Enforce the read-only boundary at the provider invocation layer rather than relying only on prompt compliance.

## Evidence

- `internal/ahm/task_groom.go:319-349` — `delegatedResultArgs` adds `codexBypassApprovalsAndSandboxFlag` for Codex and falls back to the normal agent argument builder for Cursor.
- `internal/ahm/task_agents.go:137-180` — the normal Codex builder uses `--dangerously-bypass-approvals-and-sandbox`; the normal Cursor builder uses `--trust`.
- `internal/ahm/audit.go:54-117` — the audit procedure promises read-only behavior but executes through `delegatedResultArgs`.
- `docs/adr/014-replace-managed-skills-with-command-based-procedures.md:183-186` — the accepted design states that delegated writes flow through ahm's appliers into workflow records only, never project source, branches, or HEAD.
- `docs/references/cli/commands.md:9-25` — the public CLI contract calls `audit` strictly read-only.

## Fix Direction

Introduce advisory-specific provider invocation arguments or execution profiles for both audit and grooming, preserving native structured-output/schema flags while removing write-capable permission modes wherever the provider supports a read-only mode. Treat repository content as untrusted data in the embedded advisory prompts and add a pre/post repository-state check as defense in depth, with a clear error if an advisory run mutates source or Git state. Keep mutating `task work` argument behavior unchanged and do not weaken its required non-interactive permissions.

## Decision

Read-only execution is a required capability for a provider to support audit or grooming. If a provider cannot perform an advisory workflow without write authority, fail before delegation with an actionable unsupported-capability error; do not silently fall back to its mutating task-work profile. Re-enabling that provider requires a separate compatibility decision.
