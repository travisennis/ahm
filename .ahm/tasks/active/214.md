# Add configurable sorting to task list commands

## Problem

`ahm task list`, `ahm task ready`, and `ahm task blocked` always order results by priority rank and then task ID. That default is useful for selecting work, but backlog review also needs chronological, recently updated, effort-based, status-based, and alphabetical views. Users currently have to post-process text or JSON output, and the CLI has no documented way to select a different deterministic ordering.

## Relevant Files

- `internal/ahm/task_commands.go` — shared list-command wiring and flag definitions.
- `internal/ahm/task_list.go` — task filtering, ordering, and output behavior.
- `internal/ahm/task_list_test.go` and related CLI tests — list, ready, blocked, and output-mode coverage.
- `docs/references/cli/task-commands.md` and `docs/cli.md` — public CLI contract.
- `ARCHITECTURE.md` — deterministic output and compatibility invariants.

## Fix Direction

Add a shared `--sort <field>` flag to `task list`, `task ready`, and `task blocked`, supporting `priority`, `id`, `created`, `updated`, `effort`, `status`, and `title`. Add `--reverse` to reverse the selected ordering. Preserve the existing priority-then-ID ordering when neither flag is supplied, and use task ID as a deterministic tie-breaker for selected fields.

Use domain ordering rather than lexical ordering for priority and effort. Define and document a stable status ordering. Apply the selected order consistently to text, plain, and JSON output. Reject unsupported sort fields as usage errors with actionable help. Keep `task next` unchanged: it must continue to select the highest-priority ready task using the existing priority-then-ID semantics.

## Acceptance Notes

- [ ] `task list`, `task ready`, and `task blocked` accept the same documented `--sort` values.
- [ ] `--reverse` reverses the selected ordering, with deterministic behavior for ties.
- [ ] Omitting sorting flags preserves the current priority-rank-then-task-ID output exactly.
- [ ] Priority and effort use their defined domain ranks; status has an explicitly documented stable rank; ID, timestamps, and title sort predictably.
- [ ] Sorting is consistent across text, plain, and JSON output.
- [ ] Invalid sort values return a usage error and identify the valid values.
- [ ] `task next` behavior remains unchanged regardless of the new presentation options.
- [ ] Focused tests cover every sort field, reverse ordering, tie-breaking, default compatibility, all three list commands, output modes, and invalid input.
- [ ] CLI help and `docs/references/cli/task-commands.md` document defaults, fields, direction, tie-breaking, and examples.
- [ ] Repository verification required by `CONTRIBUTING.md` passes.
