# Prevent stale-lock reclamation from stealing live workflow locks

## Problem

`removeStaleWorkflowLockAfterObservation` declares a lock stale purely from the lock directory ModTime, which the holder never refreshes. Any operation legitimately holding the record lock longer than workflowLockStaleAfter (30 minutes) gets it silently reclaimed, allowing two concurrent mutators. Today all locked sections are short (delegations run unlocked), so this is latent, but nothing enforces or documents that invariant at call sites.

## Relevant Files

- `internal/ahm/lock.go` (workflowLockStaleAfter, removeStaleWorkflowLockAfterObservation, withWorkflowRecordLock)

## Fix Direction

Refresh the lock directory ModTime periodically while the lock is held. Stop
the heartbeat before releasing the lock and make stale reclamation require an
unchanged observation older than the threshold.

## Decision

Use a heartbeat rather than relying on a documented short-section convention.
The lock protects a correctness boundary, and elapsed time alone must never let
a second writer steal a lock from a live owner. The implementation should keep
the existing bounded stale-lock recovery behavior for abandoned locks.

## Acceptance Notes

- [ ] A lock held longer than the stale threshold is not reclaimable while its owner is alive
- [ ] Test covers the chosen mechanism
