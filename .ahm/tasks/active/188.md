# Design Environment Capability Gates for Delegated Task Work

## Summary

At commit `b6af9f7`, `ahm task work` validates task state, dependencies, executables, and session support, but it has no way for a task to declare that acceptance requires an environment capability such as enforceable macOS Seatbelt. A delegated agent can therefore run in an environment where the decisive check is impossible, observe a skip, and still complete the task. Design a general, project-owned capability/preflight contract so unavailable required environments are surfaced before delegation or final completion instead of being left to model interpretation.

## Evidence

- `internal/ahm/task_work.go:70-122` — task-work intake validates role executables and constructs the delegation but has no task- or project-defined environment capability gate.
- `internal/ahm/task_work.go:158-164` — the implementation prompt generically requires verification and completion without conveying a machine-checkable capability requirement.
- `docs/VISION.md:37-40` — ahm already distinguishes workflow health from environment health through `doctor`, providing an architectural precedent for explicit environment checks.
- Cake task 260 required real Seatbelt enforcement, but the delegated cake process reported that `sandbox-exec` could not apply profiles in its process context; the workflow continued and the agent marked the requirement satisfied.

## Fix Direction

Produce an ADR and implementation-ready follow-up scope for project-defined capability gates on delegated task work. Evaluate at least: task metadata naming capabilities, project configuration mapping capability names to safe preflight checks, reuse/extension of `ahm doctor`, and an explicit manual-evidence escape hatch. Define where checks run so the result reflects the delegated agent's actual process context, how unknown/unavailable capabilities affect `task work`, review, `--no-review`, and completion, and how existing task/config formats remain compatible. Do not hard-code Seatbelt or cake-specific probes into ahm, do not infer requirements from free-form Acceptance Notes, and do not execute arbitrary task-authored shell text without a separate safety decision.

## Acceptance Notes

- [ ] An accepted ADR defines the ownership boundary between ahm, project configuration, task metadata, and the delegated agent for environment capability checks.
- [ ] The decision compares task-declared names plus project-owned probes, `ahm doctor` integration, prompt-only guidance, and manual evidence, with security and portability trade-offs.
- [ ] The design specifies whether capability checks run in the ahm parent, the delegated process context, or both, and explains how it prevents the nested-sandbox failure observed in cake task 260.
- [ ] Backward-compatible schema/config changes, unknown capability behavior, unavailable capability exit status/message, dry-run reporting, and machine-readable output are specified.
- [ ] Behavior is defined for normal review, `--no-review`, commit handoff, unsupported platforms, and an explicitly approved manual override; unavailable required capabilities cannot silently produce a Completed task.
- [ ] Safe execution constraints prohibit arbitrary task-authored commands unless separately designed and approved.
- [ ] Concrete implementation tasks are created with dependencies and focused test/live-smoke requirements after the ADR is accepted.
- [ ] Documentation maps and ADR indexes are regenerated through ahm commands, and `just ci` passes for the decision/documentation change.
