# Re-resolve task state inside status mutation lock

## Problem

`taskStatusWithArgs` resolves the target task before acquiring `task-mutate`. The locked helper invalidates the task cache and reloads the task tree for dependency checks, but it continues mutating and rendering the pre-lock `Task` value. A concurrent comment, groom revision, or dependency update that lands before lock acquisition can therefore be overwritten.

## Relevant Files

- `internal/ahm/task_status.go` — status transition locking and target rendering.
- `internal/ahm/task_commands_test.go` — status-transition concurrency coverage.

## Fix Direction

Re-resolve the target task from fresh on-disk state after acquiring the mutation lock, then perform no-op checks and the full read-compute-write transition against that fresh value. Keep dry-run semantics explicit.

## Acceptance Notes

- [ ] Status transitions re-resolve the target under the lock before mutation.
- [ ] A regression test proves a change made between initial resolution and lock acquisition is preserved.
- [ ] Existing dependency, unblocking, no-op, and dry-run behavior remains stable.
