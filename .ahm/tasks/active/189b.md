# Decide explicit commit authorization for task work

## Problem

`ahm task work` currently treats invoking the command as authorization for a default commit handoff unless `--no-commit` is supplied. This can conflict with repository guidance that prohibits commits unless the user explicitly asks for one: the delegated agent receives a new commit instruction generated by ahm even when the initiating request only asked to work or review the task. The behavior was deliberately established by completed task 118, so changing it requires an explicit compatibility and migration decision rather than an incidental flag inversion.

## Relevant Context

- Completed task 118 — made review and commit default-on with `--no-*` opt-outs.
- `internal/ahm/task_commands.go` — task-work flag surface.
- `internal/ahm/task_session.go` — commit handoff prompt and phase invocation.
- `docs/adr/008-delegated-task-work-commit-handoff.md` — commit-handoff decision.
- `docs/references/cli/task-commands.md` — documented default behavior.

## Fix Direction

Compare at least three models: explicit `--commit`, a separate `ahm task commit <id>` command, and configurable project policy with a safe default. Define how non-interactive callers express authorization, how dry-run and JSON output expose the decision, and how existing scripts migrate. Preserve the boundary that ahm itself does not stage or commit project files; the decision concerns whether and when it may instruct a delegated agent to do so.
