# Reuse parsed workflow state after record mutations

## Problem

Task mutations again parse and render workflow state several times. Status changes resolve tasks, invalidate and reparse under the lock, invalidate and reparse for indexes, then post-mutation validation reparses tasks and regenerates index content. This regresses the single-parse goals of tasks 020 and 068 after post-mutation validation was added by task 169.

## Relevant Files

- `internal/ahm/task_status.go` — mutation state flow.
- `internal/ahm/indexes.go` — cache invalidation and index generation.
- `internal/ahm/validation.go` — post-mutation validation and index checks.
- `internal/ahm/cli.go` — task cache lifecycle.

## Fix Direction

Thread the fresh parsed task set and generated index content through mutation, index writing, and post-mutation validation where safe. Preserve validation independence for standalone status/doctor calls.

## Acceptance Notes

- [ ] A representative task mutation parses each task file no more than once after the lock is acquired.
- [ ] Index content is not regenerated redundantly during post-mutation validation.
- [ ] Benchmarks or instrumented tests demonstrate the reduced reads at hundreds of tasks.
- [ ] Validation findings and generated output remain unchanged.
