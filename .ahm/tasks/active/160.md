# Add ahm docs check with expanded project-docs validation

## Summary

Track the ADR, CLI, config, and guidance work to add an `ahm docs check`
command as the ergonomic front door for project documentation health checks,
expand the checks beyond what the `project-docs` validation scope covers
today, deprecate `--check project-docs`, and wire the checks into hooks so
doc drift is caught mechanically at commit time.

## Context

Design synthesized in `.agents/.research/topics/ahm-docs-check.md` (2026-07-02),
motivated by documentation drift found in this repo and cake despite both
following the progressive-disclosure docs structure: broken index links
(task 157), enforcement recorded but never wired into CI (task 158),
entry-point creep (task 159), and machine-specific `file://` links in cake's
ARCHITECTURE.md. Division of labor: docs skills judge, `ahm` enforces
mechanics, hooks make the trigger unconditional.

Key decisions (details in the research note and the ADR from the first
child task):

- `ahm docs check` is not a new subsystem — it runs an expanded
  `project-docs` validation scope through the existing validation engine.
- `--check project-docs` on `status`/`doctor` is deprecated with a warning;
  removal is a separate later decision.
- Config lives in a `projectDocs` block in `.agents/ahm.json`
  (`entryPointBudget`, `exclude`, `docMap`).
- Diff mode (`--staged` / `--diff <range>`) is advisory-only: it suggests
  docs to review from the docMap; it never fails the run.
- Coordinate with the task 156 tree: 156f's doctor check (AGENTS.md
  references `ahm prime`) is adjacent, and 156g rewrites the same guidance
  docs phase 2 here touches.
