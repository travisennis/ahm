# Require Falsifiable Evidence in Delegated Task Reviews

## Summary

At commit `b6af9f7`, the embedded `ahm task work` reviewer is told to compare checked Acceptance Notes with implemented behavior, but it is not told to challenge whether the cited verification actually exercised the claimed behavior. In a security-sensitive sandbox task, both implementation and review treated a skipped test as enforcement evidence and accepted a fixture that remained writable through a broader path grant. Strengthen the review contract so contradictory, skipped, filtered, or non-falsifiable evidence becomes a blocking finding rather than confirmation of a checked box.

## Evidence

- `internal/ahm/task_session.go:74-92` — the review procedure covers correctness and boundaries generally but does not require the reviewer to distinguish executed tests from skips or filtered-out cases.
- `internal/ahm/task_session.go:108-113` — the completion checklist says Acceptance Notes must match behavior, but supplies no evidence-validation questions or negative-control requirement.
- `internal/ahm/task_session.go:152-161` — an empty parsed review is accepted as no feedback and the workflow proceeds, so the prompt is the primary quality boundary for semantic review.
- Cake task 260's worker session `3954579a-afc0-4c5c-a497-4cbbd33047a1` explicitly observed that its test fixture was already under a broad writable grant and later checked off sandbox enforcement after the test output said enforcement was unavailable; review session `b09b6303-8ba2-45c3-b543-9a8757e33d78` repeated both conclusions without raising a finding.

## Fix Direction

Extend the task-work review prompt and its tests with a compact evidence-challenge checklist. Require reviewers to verify that relevant tests ran rather than skipped or filtered, identify contradictory tool output, and use a negative-control/falsifiability check for boundary or security claims (for example: would the test still pass if the target implementation were removed, or is the fixture covered by another permission?). Keep this as general review guidance rather than cake-specific sandbox logic, and avoid turning the prompt into an exhaustive domain checklist. Role-specific agent/model selection already exists from completed task 167 and is out of scope.

## Acceptance Notes

- [ ] The embedded review prompt explicitly requires distinguishing passed assertions from skipped, ignored, filtered-out, or unavailable-capability test cases.
- [ ] The prompt requires reviewers to treat contradictions between Acceptance Notes and tool/test output as findings even when the task checkbox is checked.
- [ ] Security and boundary claims receive a negative-control/falsifiability check, including whether a fixture is reachable through a broader permission or whether removing the target change would leave the test green.
- [ ] The reviewer is instructed to report inability to verify a required environment-dependent claim, not infer success from an overall zero exit code.
- [ ] Focused prompt tests in `internal/ahm/task_commands_test.go` assert the durable review requirements without coupling to the cake task or session UUIDs.
- [ ] Relevant CLI/workflow documentation explains the evidence standard and the existing option to configure an independent review agent/model; no new model-selection mechanism is added.
- [ ] `just ci` passes, and `just smoke-agents` evidence is recorded if the project guardrail classifies the prompt change as requiring a live orchestration check.
