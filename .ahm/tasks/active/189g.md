# Decide acceptance enforcement for delegated task work

## Problem

`ahm task complete` intentionally warns rather than blocks on incomplete Acceptance Notes unless the repository enables `strict_acceptance`. `ahm task work` inherits that policy, so a delegated workflow can report success and proceed to commit handoff while required acceptance items remain unchecked. Strengthening delegated completion may improve trust in the command, but silently overriding repository policy would conflict with ADR 005 and completed task 170 and could break automation that deliberately uses warn-only completion.

## Relevant Context

- ADR 005 — defines warning and strict-acceptance completion behavior.
- Completed task 170 — surfaces acceptance warnings when completing tasks linked to active plans.
- Task 171 — adds workflow-status validation to delegated completion and commit handoff.
- `internal/ahm/task_acceptance.go` — canonical acceptance finding rules.
- `internal/ahm/task_session.go` — post-finalization acceptance validation.

## Fix Direction

Decide whether delegated task work should preserve repository-wide strictness, require satisfied Acceptance Notes by default, or introduce an explicit task-work policy/override such as `--allow-incomplete` with a recorded reason. Define behavior for `--no-review`, `--no-commit`, commit handoff, missing acceptance sections, unavailable environment evidence, and machine-readable output. Keep canonical parsing and finding codes shared with `ahm task complete` rather than inventing a second acceptance model.

## Acceptance Notes

- [ ] An accepted decision retains, amends, or supersedes the relevant parts of ADR 005 and completed task 170.
- [ ] The default delegated-work acceptance policy and any CLI/config override are explicit.
- [ ] Missing sections, TODO placeholders, unchecked items, unavailable evidence, and manually approved exceptions have defined outcomes.
- [ ] Behavior is consistent across reviewed, `--no-review`, `--no-commit`, resumed, and commit-handoff paths.
- [ ] JSON output, warnings, exit codes, audit evidence, and compatibility/deprecation requirements are specified.
- [ ] The design reuses canonical acceptance parsing and integrates task 171 workflow validation without duplicating it.
- [ ] Concrete implementation and documentation follow-ups are created after the decision is accepted.
