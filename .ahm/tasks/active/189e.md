# Protect pre-existing changes during delegated task work

## Blocker

Blocked until task 185 establishes the worktree-isolation decision and task
189c establishes the durable task-work run-record format.

## Problem

Review and commit handoff operate against the current uncommitted worktree without distinguishing changes that existed before `ahm task work` from changes created during the run. A reviewer may modify unrelated user work, and a delegated commit prompt may stage or commit files outside the task's actual delta. This conflicts with the repository rule to preserve uncommitted user changes and weakens the guarantee that task work remains scoped.

## Relevant Context

- `internal/ahm/task_session.go` — review prompt and broad commit handoff.
- `internal/ahm/task_work.go` — worktree intake before external mutation.
- Task 185 — designs stronger worktree isolation and may change the preferred implementation.
- Durable run records — can own baseline identity and changed-file manifests once designed.

## Fix Direction

Design a read-only worktree baseline captured before delegation and a task-delta manifest refreshed between phases. The baseline must handle tracked, staged, untracked, renamed, and overlapping files without modifying the Git index or project files. Review prompts should identify pre-existing changes as protected context. Commit handoff should receive an explicit candidate manifest and instruct the agent not to include protected changes. Define what happens when task work intentionally overlaps a pre-existing modified file and how baseline validity is checked after interruption or resume.

## Acceptance Notes

- [ ] The baseline model covers tracked, staged, untracked, renamed, deleted, and overlapping-file cases without mutating Git state.
- [ ] Review receives a clear task delta and treats pre-existing changes as protected unless overlap is explicitly required.
- [ ] Commit handoff names candidate task files and prohibits staging unrelated baseline changes.
- [ ] Concurrent edits, stale baselines, interrupted runs, and resumed sessions produce explicit conflicts rather than silent inclusion.
- [ ] The design is reconciled with task 185's worktree-isolation decision and the durable-run record format.
- [ ] Tests use dirty-worktree fixtures and verify ahm itself never stages, commits, moves HEAD, or mutates branches.
- [ ] CLI documentation explains remaining limitations when a task legitimately overlaps user changes.
