# Design a structured single-writer review contract

## Blocker

Awaiting task 187 to define falsifiable-evidence requirements before designing
the structured review schema.

## Problem

The embedded reviewer is currently told both to identify findings and to apply worthwhile fixes, while its entire free-form result is then sent to the implementation session to address. This creates ambiguous ownership, duplicate edits, noisy feedback, and no reliable distinction between "no findings," "findings fixed by reviewer," "findings requiring implementation," and unparseable prose. Provider-specific text parsers cannot enforce a stable lifecycle contract on top of unconstrained reviewer output.

## Relevant Context

- `internal/ahm/task_session.go` — embedded review procedure, feedback parsing, and finalization handoff.
- `internal/ahm/task_parsers.go` — provider-specific review result parsers.
- Task 187 — strengthens evidence quality inside reviews and should inform the structured finding schema.
- Task 186 — established review followed by implementation-session finalization.

## Fix Direction

Choose a single writer for project changes. Prefer a read-only independent reviewer that returns a bounded structured result and leaves all fixes to the implementation session, unless an ADR demonstrates that reviewer-owned edits have better recovery and ownership semantics. Define a provider-neutral schema covering result state, actionable findings, severity, path/location, evidence, verification limits, and disposition. Treat schema failure as a review failure rather than no findings.

## Acceptance Notes

- [ ] An accepted decision identifies the sole writer during review/finalization and explains how it preserves independent review.
- [ ] A provider-neutral structured review schema distinguishes no findings, actionable findings, unverifiable requirements, and malformed output.
- [ ] Findings are bounded and contain sufficient location, evidence, and remediation context without forwarding an entire narrative transcript.
- [ ] Provider adapters, schema-version compatibility, parse failure, retries, and fallback behavior are specified.
- [ ] Task 187's falsifiable-evidence requirements are represented without duplicating that task's prompt work.
- [ ] Finalization consumes only the structured review result and remains idempotent across retry or resume.
- [ ] Implementation tasks include golden transcripts, focused parser/orchestration tests, and required live-agent smoke evidence.
