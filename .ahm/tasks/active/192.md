# Serialize all workflow record mutations with one lock

## Problem

Workflow mutations use unrelated lock names or no lock. `task create`, task lifecycle changes, ADR creation, and records migration do not exclude each other; task comments, dependency updates, ADR lifecycle/supersede operations, and `task work` status marking are unlocked. Concurrent moves and writes can recreate a task in its old bucket or lose record changes.

## Relevant Files

- `internal/ahm/lock.go` — workflow lock implementation and naming.
- `internal/ahm/task_create.go`, `task_status.go`, `task_comment.go`, `task_deps.go`, `task_groom.go`, `task_work.go` — task mutations.
- `internal/ahm/adr_commands.go`, `records_migrate.go` — ADR and migration mutations.

## Fix Direction

Define one repository-scoped record-mutation lock and hold it across each non-dry-run read-compute-write sequence. Avoid nested acquisition deadlocks by separating locked helpers from public entry points. Add cross-command concurrency tests, including comment versus complete.

## Acceptance Notes

- [ ] All workflow record mutations participate in one exclusion domain.
- [ ] Concurrent comment/complete cannot leave duplicate active and completed task files.
- [ ] Task create, dependency, ADR lifecycle, groom, task-work marking, and migration paths have focused lock coverage.
- [ ] Dry-run remains lock-free and write-free.
