# Recover invalid batch grooming verdicts without discarding audit work

## Problem

`ahm task groom` validates the delegated verdict batch atomically and rejects
the entire result when any single verdict is invalid. Atomic application is an
intentional safety property from task 179 and ADR 017, but the current failure
path discards the practical value of every valid verdict after an expensive
multi-task audit.

During a live backlog grooming run, the delegate returned `action=accept` for
Blocked task 155 while producing usable verdicts for the rest of the backlog.
`validateGroomResult` stopped at that semantic error and ahm applied nothing,
forcing the user to recover the valid work manually from a very large raw
transcript. The command should preserve all-or-nothing writes while giving the
delegate one bounded opportunity to correct validation errors.

- `internal/ahm/task_groom.go` — delegated prompt construction, result parsing,

- `internal/ahm/task_groom_test.go` — groom validation, rollback, output, and

- `internal/ahm/output.go` — stable text and JSON output conventions.

- `docs/references/cli/task-commands.md` and `docs/cli.md` — public grooming
  behavior and failure reporting.
- Task 179 — implementation history for structured revisions and invalid-batch
  rollback.

## Fix Direction

- Keep batch application atomic: no task or index writes occur until the whole
  corrected result passes parsing and semantic validation.
- Collect and report all independently detectable verdict validation errors
  instead of returning only the first error.
- When a delegated result parses successfully but fails semantic validation,
  perform at most one correction retry with the original target scope, the
  original structured result, and precise machine-generated validation errors.
- Do not retry provider execution failures, timeouts, malformed transport
  output that cannot yield a structured result, or task-state changes detected
  under the mutation lock.
- Validate the corrected batch from scratch and apply it only when every
  verdict is valid. Never silently coerce an invalid action or partially apply
  the original batch.
- Make the retry visible in human and JSON output without dumping the complete
  raw provider transcript by default. On final failure, preserve concise,
  actionable validation diagnostics and enough structured result context for
  manual recovery.
- Record the bounded correction-retry behavior in ADR 017 and the CLI
  documentation because it changes external-agent invocation and failure
  semantics.

## Acceptance Notes

- [ ] A parsed groom result with multiple semantic errors reports every error
  that can be determined without mutation.
- [ ] A semantic validation failure triggers no more than one correction retry,
  and the retry prompt includes the original scope, original structured result,
  and exact validation failures.
- [ ] A fully valid corrected result is applied atomically through the existing
  task mutation lock and index-regeneration path.
- [ ] If the correction retry fails, no task or generated index changes are
  applied and the command returns concise recovery-oriented diagnostics.
- [ ] Provider failures, timeouts, unparseable results, and task-state races do
  not trigger an inappropriate correction retry.
- [ ] Text, JSON, and dry-run behavior clearly describe whether correction is
  available, attempted, succeeded, or failed.
- [ ] Regression coverage includes the observed case: one Blocked task receives
  `action=accept` while the remaining verdicts are valid.
- [ ] Tests prove invalid original and corrected batches remain all-or-nothing
  and never duplicate comments, dependency changes, or revisions.
- [ ] ADR 017, CLI references, and relevant architecture documentation describe
  the bounded retry and retained atomicity contract.
