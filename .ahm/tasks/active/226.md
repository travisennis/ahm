# Fix atomic write directory sync and errno checks on Windows

## Problem

GoReleaser ships windows/amd64 and windows/arm64 binaries, but CI runs ubuntu-latest only, and two portability problems mean Windows is likely broken: (1) `fsyncDir` opens the directory O_RDONLY and calls Sync; on Windows FlushFileBuffers on a read-only directory handle fails (ERROR_ACCESS_DENIED), so every `writeFileAtomic` reports an error after the rename already succeeded — every mutating command "fails" while half-working. (2) `errors.Is(err, syscall.ENOTEMPTY)` tolerance checks will not match Windows' ERROR_DIR_NOT_EMPTY, turning tolerated non-empty-dir cases into hard errors.

## Relevant Files

- `internal/ahm/write.go` (fsyncDir)
- `internal/ahm/install.go` (removeObsoleteManagedFiles ENOTEMPTY check)
- `internal/ahm/records_migrate.go` (removeEmptyRecordDirs ENOTEMPTY check)
- `.goreleaser.yaml` (windows targets)
- `.github/workflows/ci.yml` (ubuntu-only matrix)

## Fix Direction

Skip directory fsync on Windows, replace raw `ENOTEMPTY` comparisons with a
cross-platform helper that also matches Windows' directory-not-empty error,
and add a Windows leg to CI while retaining the existing Windows release
targets.

## Decision

Preserve the published Windows compatibility surface. Skipping directory fsync
on Windows is the established portability compromise; swallowing arbitrary
sync errors would hide unrelated failures. CI must exercise the targets that
GoReleaser continues to publish.

## Acceptance Notes

- [ ] `writeFileAtomic` succeeds on Windows in CI
- [ ] Non-empty-directory tolerance works on Windows
- [ ] CI matrix and release targets are consistent
