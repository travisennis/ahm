# Separate current upgrade guidance from release history

## Summary

`docs/guides/workflow-upgrades.md` combines current upgrade instructions with dated release-history entries that describe superseded behavior. Contradictory statements about scaffold files and record tracking now appear in the same live guide, making it difficult for users and agents to identify the current migration contract. This task separates operational guidance from history while preserving useful compatibility context as of commit `5d231a4`.

## Evidence

- `docs/guides/workflow-upgrades.md:81-92` — correctly says fresh installs no longer create `.ahm/tasks/README.md`, `.ahm/research/README.md`, or `docs/adr/README.md`.
- `docs/guides/workflow-upgrades.md:114-122` — later says fresh `.ahm` initialization creates those scaffold READMEs.
- `internal/ahm/install.go:200-205,263-264` — classifies those files as preserved scaffolds that are no longer created or managed.
- `docs/guides/workflow-upgrades.md:164-171` — describes source records as normal committed files and then says migrated records leave normal branch history.
- `docs/adr/015-use-committed-ahm-workflow-record-storage.md:75-109` — establishes committed `.ahm/` source records and gitignored generated indexes as the accepted model.
- The guide is 382 lines with many date-stamped changes, duplicating ADR and changelog history inside an operational guide.

## Fix Direction

Make `docs/guides/workflow-upgrades.md` a concise current guide covering supported fresh-init, legacy-layout, migration, upgrade, rollback, and compatibility behavior. Move release-specific chronology that still has value into an explicitly historical document or rely on ADRs and `CHANGELOG.md`, preserving stable redirect links where necessary. Ensure historical text is visibly non-normative and cannot be mistaken for current instructions. Do not edit ADR decisions or change implementation behavior. If an old entry is the only surviving explanation for a still-supported compatibility rule, retain that explanation in the current guide without retaining obsolete version narration.

## Acceptance Notes

- [ ] The primary upgrade guide contains one unambiguous current description of fresh init, upgrade, records migration, generated indexes, and committed source records.
- [ ] Superseded scaffold creation, removed agent-suggestion behavior, and old unscoped-context behavior are absent from the normative current instructions.
- [ ] Useful release chronology is either moved to a clearly historical location with appropriate links or removed when already captured by ADRs/`CHANGELOG.md`.
- [ ] Existing stable entry point `docs/upgrades.md` continues to resolve to the current guide.
- [ ] `ahm docs check --strict` and `just docs-md-lint` pass.
- [ ] Documentation claims are spot-checked against `internal/ahm/install.go`, `internal/ahm/records_migrate.go`, `internal/ahm/context.go`, and ADR 015.
