# Design an ahm config command for workflow settings

## Summary

Repository-scoped settings are configurable only by hand-editing
`.ahm/config.json` (or legacy `.agents/ahm.json`). The surface now includes
`strict_acceptance`, `default_work_agent`, role-specific `taskWork` settings,
`projectDocs`, and `research`, plus preserved unknown fields. Corrupt metadata
deliberately fails commands loudly, so a validating `ahm config` surface could
reduce editing risk, but its key grammar and ownership boundary require a
durable decision first. The original two-key proposal was written against
commit `ff15056` and is retained only as historical context.

## Evidence

- `internal/ahm/install.go:19-62` — workflow metadata includes task-work roles,
  project-doc settings, research settings, strict acceptance, and the legacy
  default agent.
- `internal/ahm/install.go:430-461` — metadata reads prefer the active layout
  and writes use the atomic workflow write primitive.
- `docs/references/workflow-spec.md` — documents current `.ahm/config.json`
  ownership and supported settings.
- `internal/ahm/task_agents.go` — consumes legacy and role-specific agent/model
  configuration.

## Fix Direction

Produce an ADR that selects a bounded configuration command model. Compare a
typed fixed-key surface, dotted paths for supported nested settings, and a
whole-document edit/validate workflow. The decision must preserve unknown
fields, reject writes to ahm-owned `version` and `files` metadata, validate
values before atomic replacement, support the active and legacy metadata
layouts, and define text/JSON/plain plus dry-run behavior. After acceptance,
create implementation tasks rather than silently reviving the obsolete
two-key scope.

## Decision

This task produces the prerequisite ADR; it does not implement the command.
The ADR must choose the mutable-key surface and addressing model, after which
separate implementation tasks will carry the selected behavior.

## Acceptance Notes

- [ ] An accepted ADR defines the supported configuration command model and
      reconciles current `.ahm/config.json` with legacy `.agents/ahm.json`.
- [ ] Supported mutable keys, nested-key syntax, value validation, unknown-field
      preservation, and prohibited ahm-owned keys are explicit.
- [ ] Missing/corrupt metadata, dry-run, output modes, exit codes, and atomic
      write behavior are specified.
- [ ] Security and safety constraints prohibit arbitrary command execution or
      implicit metadata creation/reset.
- [ ] Concrete implementation and documentation tasks are created after the
      decision is accepted.
- [ ] `just ci` passes for the ADR/documentation change.
