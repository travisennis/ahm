# generatedIndexTargetsList hardcodes both storage layout paths

## Problem

The generatedIndexTargetsList in indexes.go contains 20 hardcoded relative paths covering both .agents/ and .ahm/ layouts, including legacy .ahm/.tasks/ dot-prefixed paths. Adding a new generated index target (e.g., a new bucket or directory) requires manually updating multiple parallel entries. This manual synchronization is fragile and has already diverged from the dynamic paths computed by workflowPaths methods.

## Relevant Files

- `internal/ahm/indexes.go`

## Fix Direction

Replace generatedIndexTargetsList with a function that derives targets dynamically from workflowPaths methods (tasksRel, researchRel, execPlansRel) and a known bucket list. Collect all active/config combinations from legacyRecordsDirName and toolRecordsDirName, plus any legacy dot-prefixed paths that are still in use. This ensures new targets only need to be added in one place.

## Acceptance Notes

- [ ] generatedIndexTargets() returns the same set of paths as the current hardcoded list for both legacy and migrated repositories
- [ ] generatedIndexTargets() does not include stale or out-of-scope paths
- [ ] Generated index checks (validateGeneratedIndexes) continue to pass
- [ ] Callers that depend on the exact list (install.go metadata cleanup, validation) work unchanged

## Cancellation Reason

The static list is an intentional compatibility cleanup inventory spanning current and historical generated-index paths; current-layout validation already derives targets from indexWritesFor, and workflowPaths cannot safely reconstruct obsolete paths.
