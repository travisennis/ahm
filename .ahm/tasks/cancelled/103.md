# Align binary version with template version

## Summary

`internal/version/version.go` has `var Binary = "0.2.0"` while `internal/templates/templates.go` has `const Version = "0.3.1"`. They've drifted 55% apart. The comments say they're "semantically independent," but nothing enforces that, and the drift means a binary release built without goreleaser ldflags ships as 0.2.0 even though the workflow templates it embeds are at 0.3.1.

## Files in scope

- `internal/version/version.go` — the binary version string
- `docs/guardrails/dependencies-build-ci-release.md` — version-bumping policy

## Files out of scope

- `.goreleaser.yaml` or release workflow files (those pass ldflags)
- `internal/templates/templates.go` (leave `Version` alone at 0.3.1)
- Any source code or test files

## Current state

```go
// internal/version/version.go
var Binary = "0.2.0"
```

```go
// internal/templates/templates.go
const Version = "0.3.1"
```

## Steps

### Step 1: Bump `Binary` to match current template version

Change line 11 of `internal/version/version.go`:

```go
var Binary = "0.3.1"
```

### Step 2: Document the bump policy

Add a section to `docs/guardrails/dependencies-build-ci-release.md`. Add after the existing content:

```markdown
### Version alignment

Two version constants are maintained separately:

| Constant | File | Advances when |
|----------|------|---------------|
| `version.Binary` | `internal/version/version.go` | A new binary release is cut (via goreleaser ldflags override). |
| `templates.Version` | `internal/templates/templates.go` | Embedded workflow templates change. |

These are semantically independent but should not drift across a release boundary.
Before tagging a release, verify that `version.Binary`'s default value (used when
goreleaser ldflags are absent) is not materially lower than `templates.Version`.
As a rule of thumb, bump `version.Binary` to match `templates.Version` when the
template version has advanced since the last binary bump.
```

### Step 3: Verify

```bash
go build -o /dev/null ./cmd/ahm
go vet ./internal/ahm/
go test -count=1 ./internal/ahm/
go run ./cmd/ahm version
# Expected output: 0.3.1
```

## Cancellation Reason

Superseded by release work. The premise was that Binary and templates.Version needed alignment, but they are semantically independent (now documented in guardrails). The real problem — Binary defaulting to a misleading stale version — was already fixed by making the default "dev" and letting goreleaser ldflags set releases.
