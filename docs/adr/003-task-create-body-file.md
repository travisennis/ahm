# Task Create Body Input From File or Stdin

## Context

`ahm task create` allocates the next task ID, writes generated front matter,
places the file in `.agents/.tasks/active/`, and regenerates indexes. The body
was limited to a single `--description` string that filled a fixed
`## Summary` / `## Acceptance Notes` scaffold.

Agents and scripts that need complete task records with sections such as
Problem, Relevant Files, Fix Direction, and Acceptance Notes had to patch the
created Markdown file afterward. That two-step workflow increases the chance of
stale indexes and inconsistent task shape, and it splits ownership of a single
created task across two operations.

## Decision

Add a `--body-file <path>` flag to `ahm task create`.

- The flag value is read as the full Markdown body that appears after the
  generated `# <title>` H1 heading.
- `ahm` still owns ID allocation, front matter, the H1 heading, the active task
  location, and index regeneration. Only the content below the H1 comes from the
  caller.
- `--body-file -` reads the body from stdin so callers can pipe content.
- File content is whitespace-trimmed and CRLF line endings are normalized to LF,
  matching how `ahm` reads other workflow Markdown.
- `--body-file` and `--description` are mutually exclusive.
- Explicit errors are returned when the file cannot be read, when stdin is
  requested but unavailable, or when the resolved body is empty.

The contract is intentionally "body after the H1" rather than "whole document
excluding front matter" because `ahm` must remain the sole writer of front
matter and the title heading, and the simpler contract is easier to explain and
test.

## Rationale

- Keeps task creation a single command that produces a complete, valid record.
- Preserves `ahm` ownership of ID allocation, front matter, title, placement,
  and indexes, avoiding drift and stale indexes.
- The "body after H1" contract keeps front matter and the title generated by
  `ahm` and avoids parsing user-supplied front matter.
- Stdin support fits agent and script pipelines that generate body content
  dynamically.

## Consequences

### Positive

- Agents and scripts can create full task bodies in one step.
- The default `--description` scaffold behavior is unchanged for existing
  callers.
- Validation errors for unreadable files, missing stdin, and empty bodies are
  explicit.

### Negative

- Callers cannot set front matter fields through the body file; they must use the
  dedicated flags. This is intentional to preserve front matter ownership.
- Structured per-section flags are out of scope and remain a possible future
  follow-up.

## Alternatives Considered

- **Accept the whole Markdown document including front matter**: Rejected
  because it would let callers override `ahm`-owned front matter and the title,
  weakening the ownership boundary and complicating validation.
- **Add many structured section flags**: Rejected for this change as
  over-engineered; a single body input covers the need and keeps the surface
  small.
- **Keep post-creation file patching**: Rejected because it is a two-step
  workflow that risks stale indexes and inconsistent task shape.
