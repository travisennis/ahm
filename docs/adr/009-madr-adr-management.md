# MADR ADR Management

## Context and Problem Statement

`ahm` currently installs and documents Architecture Decision Records under
`docs/adr/`, but the format is a project-local Nygard-style Markdown convention:
`# ADR NNN: Title`, bold `Status` and `Date` lines, and body sections such as
Context, Decision, Rationale, and Consequences. The files are durable human
records, but they are not structured enough for reliable creation, listing,
status changes, generated indexes, validation, or migration.

The next ADR feature adds an `ahm adr` command family. That command family needs
a single supported ADR data model, deterministic numbering, clear ownership
boundaries, and migration behavior for the existing legacy ADRs.

## Decision Drivers

- ADRs should be manageable by `ahm` without making free-form decision bodies
  brittle.
- Existing repositories and this repository already use `docs/adr/`; moving the
  directory would create churn without improving the feature.
- `ahm` already has a scalar front matter parser for tasks, and ADR metadata
  should reuse the same simple constraints instead of introducing a second YAML
  dialect.
- Existing ADRs are decision history. Migration must preserve the body content
  that explains past choices.
- Generated files and user-owned writing need an explicit boundary so commands
  do not silently rewrite human-authored prose.

## Considered Options

- Adopt MADR 4.x in `docs/adr/` with a constrained scalar front matter profile.
- Keep the current project-local Nygard-style ADR format and add custom
  parsing around bold metadata lines.
- Adopt upstream MADR defaults exactly, including `docs/decisions/` and
  four-digit numbering.
- Support both legacy and MADR records indefinitely.

## Decision Outcome

Chosen option: "Adopt MADR 4.x in `docs/adr/` with a constrained scalar front
matter profile", because it gives `ahm adr` structured metadata while preserving
the repository's existing ADR location and keeping body prose user-owned.

`ahm adr` supports only this constrained MADR profile for new records:

- ADR files live in `docs/adr/`.
- Filenames use stable three-digit IDs and kebab-case slugs:
  `NNN-short-title.md`. Numbers are never reused.
- The ID is derived from the filename prefix. If an `id`-like metadata field is
  present later, it must agree with the filename.
- ADR 009 is the first record written in the constrained MADR profile.
- Front matter fields are scalar only. The core fields are `status`, `date`,
  `decision-makers`, `consulted`, and `informed`.
- List-like MADR fields use comma-separated scalar values. YAML block lists,
  block scalars, and other multi-line front matter values are invalid.
- Unknown front matter fields are preserved when `ahm` rewrites an ADR.
- Valid statuses are `proposed`, `accepted`, `rejected`, `deprecated`, and the
  pattern `superseded by ADR-NNN`.
- MADR has no first-class "superseded in part" status. Partial supersession is
  represented by keeping the old ADR's status as `accepted` and recording the
  partial replacement in the body, usually under `## More Information`; the new
  ADR states which part of the old decision it replaces.
- ADR bodies are user-owned. `ahm` may mutate front matter through `ahm adr`
  commands, generate `docs/adr/index.md`, and write the defined supersession
  note or reciprocal reference used by `ahm adr supersede`. Other body content
  is not rewritten by lifecycle commands.
- `docs/adr/README.md` remains a managed workflow template installed by `ahm`.
- `docs/adr/index.md` is generated by `ahm index` and must not be edited by
  hand.
- Supersession is explicit and bidirectional through
  `ahm adr supersede <old-id> --by <new-id>`.
- Legacy ADR migration is metadata-only. `ahm adr migrate` converts the legacy
  heading and bold metadata lines into constrained MADR front matter, strips the
  consumed legacy metadata, and leaves body sections intact.
- ADR validation belongs to the existing `workflow` check scope. A separate
  `adr` validation scope is not added for v1 because ADRs are part of the
  managed workflow state alongside task, research, and ExecPlan indexes.
- Before migration exists, legacy ADR files produce non-fatal workflow
  validation findings that point at `ahm adr migrate`; they must not make
  `ahm status`, `ahm doctor`, `ahm index`, or `ahm adr list` unusable.

### Consequences

Good, because new ADR commands have a single file format and status model.

Good, because existing repositories can keep `docs/adr/` and three-digit
numbering.

Good, because migration preserves historical decision prose instead of
pretending old records were originally authored in MADR.

Good, because validation and generated indexes fit the existing `workflow`
scope and `ahm index` command.

Bad, because this is not fully general YAML. Users who expect MADR block lists
must use comma-separated scalar fields in `ahm`-managed ADRs.

Bad, because legacy ADRs remain visible as validation findings until
`ahm adr migrate` is implemented and run.

### Confirmation

Implementation tasks 075-081 must cite this ADR and the active feature
ExecPlan. The feature is complete when `ahm adr create`, `list`, `show`,
`accept`, `reject`, `deprecate`, `supersede`, and `migrate` operate on the
constrained MADR profile; `ahm index` generates `docs/adr/index.md`; and
`ahm status` / `ahm doctor` report ADR workflow findings without breaking on
legacy files.

## Pros and Cons of the Options

### Adopt MADR 4.x in `docs/adr/` with constrained scalar front matter

Good, because it uses a known ADR template shape while fitting `ahm`'s existing
front matter parser and workflow directory.

Good, because it gives command implementations stable fields for status, date,
participants, generated indexes, and validation.

Neutral, because the constrained profile is a subset of MADR rather than every
YAML shape MADR users might write by hand.

Bad, because existing legacy ADRs need migration before this repository is fully
clean under ADR validation.

### Keep the current project-local Nygard-style format

Good, because existing ADRs would not need metadata migration.

Bad, because bold-line metadata is weak command input and would force custom
parsing behavior that does not compose with `ahm`'s front matter machinery.

Bad, because the project would keep maintaining a custom format when MADR
already covers the intended record structure.

### Adopt upstream MADR defaults exactly

Good, because upstream examples and tools would match paths and numbering more
closely.

Bad, because renaming `docs/adr/` to `docs/decisions/` and switching to
four-digit IDs would create avoidable churn in this repository and consumers
that already follow the installed `ahm` workflow.

### Support both legacy and MADR records indefinitely

Good, because it would reduce migration pressure.

Bad, because every command, validator, renderer, and generated index would need
two permanent code paths. That makes the durable workflow less predictable and
keeps old ambiguity alive.
